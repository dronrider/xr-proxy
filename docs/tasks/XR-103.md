# XR-103. Реализация LLD-23: доступ к шаре за NAT через xr-relay

Дизайн закрыт в [LLD-23](../lld/23-share-relay-nat.md), развилки решены там же
(§3). Здесь ход реализации, разбивка по коммитам и сценарий проверки.

## Что делаем

Недостижимый снаружи агент (за NAT/CGNAT) остаётся доступен через посредника
`xr-relay`: агент держит исходящий реверс-туннель к relay, потребитель ходит к
relay тем же обфусцированным mux, relay слепо сплайсит байты. Внутри сплайса TLS
1.3 потребитель <-> агент с пиннингом на ключ агента, relay видит только
шифртекст. Хаб остаётся чистым сигналингом: раздаёт relay-дескриптор и минтит
relay-токены в грантах.

## Разбивка по коммитам (общие крейты раньше зависящих)

1. `feat(proto)` XR-103: `RelayToken`/`RelayDescriptor` + подпись/проверка
   (домен `xr-relay-token`), поле `relay` в `ShareGrant`,
   `CLOSE_REASON_AGENT_OFFLINE = 3`, адаптер `MuxStream -> AsyncRead+AsyncWrite`,
   чётность id стримов (тест).
2. `feat(proto)` XR-103: `relay_client` (mux-клиент потребителя, loopback-forwarder).
3. `feat(relay)` XR-103: новый крейт `xr-relay` (листенер, реестр агентов,
   challenge-response регистрации, проверка relay-токенов, сплайс, счётчики,
   лимиты).
4. `feat(share)` XR-103: `[relay]` в конфиге, uplink-петля с бэкофом,
   регистрационный стрим, обслуживание реверс-стримов тем же роутером,
   identity-TLS (self-signed из identity-ключа).
5. `feat(hub)` XR-103: `[relay]` в конфиге, `via_relay` в `ShareRecord`,
   relay-дескриптор агенту и потребителю, минт `RelayToken`, отметка в админке.
6. `feat(core)` XR-103: pinned-TLS verifier (SPKI == `agent_pubkey`), relay
   последним адресом перебора; тот же fallback в `xr-share pull`.
7. `docs` XR-103: факты LLD-23 в `ARCHITECTURE.md`.

## Статус (что сделано, что осталось)

Сделано и покрыто тестами (нативно зелёное, 0 warnings в затронутых крейтах):

- **xr-proto**: `RelayToken`/`RelayDescriptor`/`RelayGrant`/`RelayObf` + подпись
  и проверка (домен `xr-relay-token`), `RelayRegister` challenge-response +
  `sign/verify_relay_register`, `CLOSE_REASON_AGENT_OFFLINE`, признак `via_relay`
  в `ShareRecord`, чётность id стримов, адаптер `MuxStream::into_io()`
  (AsyncRead+AsyncWrite), `relay_client` (mux к relay, hello с токеном,
  `LoopbackForwarder`).
- **xr-relay**: весь сервис. Реестр агентов (вытеснение дубля, generation-guard),
  регистрация по challenge-response, проверка relay-токенов офлайн, слепой сплайс
  (`copy_bidirectional`), счётчики байт per share, лимиты (семафоры соединений и
  стримов, кап регистраций с IP). Интеграционный тест: агент регистрируется,
  потребитель с валидным токеном гоняет байты через сплайс, счётчики считают;
  плюс агент-офлайн, подделанный токен, слепой сплайс произвольных байт.
- **xr-hub**: блок `[relay]` в конфиге, `via_relay` в `add`, минт `RelayToken`
  и relay-плечо в грантах, дескриптор relay агенту в ответах `exchange`/`add`.

Оконечный data-path (второй заход, сделано и покрыто тестами):

- **xr-proto** (фича `relay-tls`): `relay_tls` с пиннинг-verifier
  (`SPKI == agent_pubkey`, имя хоста игнорируется, подпись хендшейка проверяется
  через провайдер) и билдерами rustls-конфигов на ring. rustls уже в дереве
  через reqwest, x509-cert чистый Rust, кросс-сборка musl-клиента проверена.
  Тесты: извлечение SPKI, отклонение чужого ключа, полный TLS-хендшейк с
  отклонением MITM.
- **xr-share** (фича `relay`, default off): `[relay]` в конфиге (захват
  дескриптора при `install`/`share`, флаг `--relay` у `share`), uplink-петля с
  экспоненциальным бэкофом, регистрация challenge-response, обслуживание
  реверс-стримов тем же axum-роутером поверх identity-TLS (self-signed из
  ed25519-ключа, rcgen на ring), hyper `serve_connection`. Тесты: `SPKI == ключ
  агента`, роундтрип блоба мандата, полный e2e (агент + реальный relay +
  пиннинг-потребитель: манифест и файл через сплайс, MITM ломается). Фича off по
  умолчанию, обычная сборка агента без rcgen/rustls/hyper.
- **xr-core**: `sync_share_grant` пробует прямой адрес первым, relay
  (pinned-TLS поверх loopback-forwarder) последним и только при недостижимости
  прямого. Внутренние `*_with`-варианты принимают готовый клиент+базу, публичное
  API не изменилось. Тесты: декод токена, сборка пиннинг-клиента, классификация
  «недостижимо», грант без relay пробрасывает прямую сетевую ошибку.

Осталось за пределами XR-103 (заведено/деплой):

- JNI/Kotlin проброс relay-плеча гранта в `sync_share_grant` на Android
  (слой без автотестов, правила проекта).
- identity-TLS на прямом листенере агента (сейчас прямой путь plain-HTTP,
  целостность закрыта подписью манифеста XR-046).
- relay-fallback в десктопном `xr-share pull` (ureq, синхронный).
- Отметка «через relay» в Admin SPA (данные `via_relay` уже отдаются
  `admin_list_shares`, нужен пересбор встроенного SPA).

## DoD

- Автотесты §4 LLD-23 зелёные, `cargo test --workspace` без warnings. **Готово.**
- Relay слепой: сплайс структурно не парсит содержимое (`test_relay_splice_opaque`). **Готово.**
- Пиннинг реально ломает MITM со стороны relay
  (`relay_tls::pinned_handshake_accepts_agent_rejects_mitm`,
  `relay::agent_serves_over_relay_with_pinned_tls`). **Готово.**
- Прямой путь не задет: грант без relay идёт прямым путём, relay только на
  недостижимости (`grant_without_relay_reports_direct_failure`, порядок
  `sync_share_grant`). **Готово.**
- Кросс-сборка: musl-клиент (`xr-client`) и агент с фичей `relay`
  (`xr-share --features relay`) собираются под `aarch64-unknown-linux-musl`.
  **Готово.**

## Сценарий проверки

Живой сценарий из [LLD-23 §6](../lld/23-share-relay-nat.md#6-план-проверки):

1. Агент за NAT (без пробросов), `[relay]` включён -> в админке хаба шара
   помечена «через relay», агент онлайн.
2. Android вне LAN агента: список шар приходит, манифест открывается, файл
   качается, SHA-256 сходится. На агенте в логах реверс-стримы, на relay только
   счётчики байтов.
3. `xr-share pull --invite ...` с третьей машины: тот же флоу без устройства.
4. Прямой путь жив: шара с белым IP качается напрямую, relay не участвует.
5. Остановить агента -> потребитель получает «источник недоступен» быстро;
   запустить -> доступ восстанавливается без действий на relay/хабе.
6. Рестарт relay -> агент переподключается в пределах минуты.
7. E2E: tcpdump на VPS relay показывает только обфусцированный поток; подмена
   сертификата на relay (тестовый MITM) роняет скачивание ошибкой пиннинга.
8. Большой файл (сотни МБ) через relay на медленном потребителе (LTE): скачка
   доезжает до конца, в логе relay нет `mux stream channel full, closing`.
   Известное ограничение: в mux нет пер-стримового flow control, при быстром
   аплоаде агента и медленном потребителе стрим может обрезаться (хвост XR-115).
9. `cargo test --workspace` зелёный, 0 warnings.

## Ревью

Ревью транзита и сигналинга (первый прогон, до data-path):

1. Нет пер-стримового flow control в сплайсе: быстрый агент + медленный
   потребитель переполняет rx-канал стрима (1024 кадра), reader режет стрим.
   Зафиксировано как ограничение MVP, хвост XR-115; в сценарий проверки добавлен
   п. 8.
2. Счётчики байт учитывали только чистый выход `copy_bidirectional`, обрыв и
   лайфтайм-кап теряли всё перегнанное. Исправлено: обёртка `CountedIo` считает
   по ходу, итог пишется при любом исходе; регрессионный тест
   `test_counters_survive_splice_lifetime_cut`.
3. `serve()` крутил горячий цикл на устойчивой ошибке accept (EMFILE).
   Исправлено: пауза 100мс перед повтором.
4. Исчерпание `max_streams` было неотличимо от агент-офлайн (голый Close).
   Исправлено: `CLOSE_REASON_RELAY_BUSY = 4`, тест `test_relay_stream_cap_refuses`.
5. Длинные тире в добавленных строках ARCHITECTURE.md и комментарии
   `xr-proto/src/lib.rs`. Исправлено попутно.
6. DoS-поза: `max_connections` как backpressure позволяет неаутентифицированным
   соединениям остановить приём вообще, per-read таймаут хендшейка пропускает
   slow-loris. Отклонено как правка ревью (backpressure это выбор LLD-23 §5.2),
   заведён хвост XR-116.
7. `open_relay_stream` молча выбрасывал хвост ответа после байта `OK`.
   Исправлено: строгая проверка `reply == [RELAY_HELLO_OK]`.
8. Пробелы тестов: непокрыты tail-путь `poll_read` (кадр больше буфера читателя)
   и медленный путь записи `write_fut` в `MuxStreamIo`. Осознанно оставлено:
   оба пути косвенно прогоняются e2e-сплайсом, отдельные юниты добавим вместе с
   data-path.

Ревью оконечного data-path (второй прогон, оси пиннинг/relay-last/гейтинг
чисты, три low в `xr-share/src/relay.rs`, все исправлены):

1. Закрытие только регистрационного стрима (дерегистрация без обрыва mux) не
   поднимало reconnect: цикл крутился на `new_stream_rx`, агент обслуживал в
   мёртвый реестр. Исправлено: `reg_stream` в `select!`, его конец завершает
   соединение и запускает перерегистрацию.
2. Чистое завершение (`Ok`) реконнектило без задержки: патологический relay
   (accept -> register OK -> мгновенный обрыв) давал горячий цикл. Исправлено:
   пол задержки `BACKOFF_MIN` и на ветке `Ok`.
3. Нет таймаута на identity-TLS хендшейк реверс-стрима: потребитель с валидным
   токеном, открывший стрим и не досылающий ClientHello (slow-loris), держал
   таску и слот реестра стримов до смерти mux. Исправлено: `REVERSE_HANDSHAKE_
   TIMEOUT` на `accept` (только хендшейк, не сам трансфер файла).
