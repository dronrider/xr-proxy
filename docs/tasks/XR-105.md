# XR-105. CI release-xr-share падает на musl-сборке

## Разбор

Обе linux-джобы валились на шаге Build при зелёном windows, publish
пропускался, а лог рана без админ-токена не открывался, поэтому релиз
2026-07-10 выкладывали вручную. Корень нашёлся на прогоне XR-133 по таймингам
шагов через публичный API: Build умирает через 2 секунды, то есть до всякой
компиляции. Workflow ставил cargo-zigbuild, но не сам zig. На ubuntu-раннере
zig не предустановлен, и zigbuild падал, не найдя кросс-линкер. Локально те же
цели собирались, потому что на машине zig был.

## Что сделано

В build-linux добавлен шаг `pip3 install ziglang==0.13.0.post1` перед сборкой
(cargo-zigbuild умеет брать zig из pip-пакета). Версия 0.13 та же, что
собирала musl-бинари флота вручную.

Фикс подтверждён раном по перевыпущенному тегу `xr-share-v0.3.0`: обе
linux-матрицы собрались за полторы-две минуты каждая, publish дошёл до заливки
на хаб. Сама заливка упёрлась в отсутствие секретов репозитория, это уже хвост
XR-133, не musl-сборка.

## Сценарий проверки

Открыть свежий ран release-xr-share по тегу `xr-share-v0.3.0`: обе джобы
build-linux зелёные, артефакты xr-share-linux-* собраны. После заведения
секретов ран прошёл и publish до конца, share-dist обновлён с CI (проверка
выложенного в tasks/XR-133.md).
