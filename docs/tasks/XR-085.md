# XR-085: анализ и оптимизация CPU/RAM на слабых целях

OpenWRT-роутер + Android. Сейчас профиля нет: RSS видно только суммарно
(`VmRSS` в мониторе), CPU и аллокации по местам неизвестны, оптимизировать
вслепую нельзя.

## Что сделать

1. Инструментация в движке: байт-счётчик через `#[global_allocator]`-обёртку
   (точное живое потребление кучи) плюс доменные счётчики (размер `mux.streams`,
   глубина writer-очереди, число живых mux, число tokio-задач), вывод в
   лог/монитор.
2. Офлайн-профилирование на x86-сборке под синтетической нагрузкой: `dhat`
   (память по местам аллокации), flamegraph (`cargo flamegraph`/`perf`,
   CPU-хотспоты), `tokio-console` (учёт задач и время поллинга, ловит
   накопление футур как в XR-079).
3. По находкам оптимизировать.

Рантайм на роутере single-thread, поэтому накопление тасков/футур не только
жрёт память, но и топит исполнитель (корень зависаний XR-079/084 жил там), учёт
задач приоритетен.

## Место в программе

Зонтичная измерительная задача: даёт данные для точечных XR-065 (zero-copy
фрейминг), XR-071 (flow control mux), XR-072 (zero-copy JNI на Android), XR-068
(энергия мобильного), меряется на стенде XR-063. Серьёзность 0 (работает),
ценность в жизнеспособности на слабом железе и мобильном и в позиционировании
«легче xray/amnezia».

## Проверка

Есть воспроизводимый профиль CPU/RAM для router и mobile под нагрузкой с
атрибуцией по местам, из него список конкретных оптимизаций с числами.
