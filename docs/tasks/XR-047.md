# XR-047: редактор правил проксирования в приложении (LLD-05)

Дизайн в [docs/lld/05-android-rules-editor.md](../lld/05-android-rules-editor.md).
LLD писался под трёхвкладочное приложение, реализация адаптирована под
текущее (четыре вкладки, профили серверов); развилки согласованы с владельцем:

- редактор живёт не пятой вкладкой, а секцией «Маршрутизация» на вкладке
  «Серверы» (тап открывает полноэкранный редактор);
- «мои правила» глобальные: один список поверх пресета любого активного
  сервера, файл `filesDir/user_rules.json`;
- чип на главном экране показывает «имя сервера · пресет хаба» (метки
  Russia / Proxy all / Custom умерли вместе с хардкодом).

## Что сделано

- `xr-proto/src/user_rule.rs`: `UserRule`, `classify_pattern` (домен /
  `*.wildcard` / CIDR v4/v6, одиночный IP нормализуется в /32-/128, одиночная
  `*` это «любой домен»), `normalize_pattern`, `to_routing_config` (битые
  паттерны отбрасываются с WARN, не валя старт). Юнит-тесты, включая сквозной
  «моё правило перекрывает пресет» через `Router::from_merged`.
- `xr-core/presets.rs`: `PresetCache::refresh` с внятным исходом
  (обновился / актуален / ошибка), `fetch_if_stale` стал обёрткой над ним.
- `xr-android-jni`: `parse_config` понимает `user_rules` + `default_action`
  (главнее `routing_toml`, легаси-ветка осталась); новые экспорты
  `nativeClassifyPattern` (общая валидация для UI) и `nativeRefreshPreset`
  («Обновить сейчас», пишет в тот же кэш, что движок).
- Kotlin: `UserRulesStore` (атомарная запись, лимит 100, разовая миграция
  legacy-правил профилей: кастомные домены/CIDR -> proxy-правила, `proxy_all`
  -> правило `*`), `PresetCacheReader`, экран `RulesScreen` (карточка пресета
  read-only с «Обновить сейчас»/«Подробнее», список моих правил с пилюлей
  действия и меню перестановок, диалог с живой классификацией, TOML-превью
  с копированием), точка входа `RulesEntryCard` на «Серверах».
- Удалён хардкод: `PRESET_RUSSIA`, `buildRoutingToml`/`buildCustomRoutingToml`/
  `parseTomlDomains`, FilterChip'ы Russia/Proxy all/Custom и textarea'и в
  редакторе сервера, поля `routingPreset`/`customDomains`/`customIpRanges`
  профиля. `buildConfigJson` шлёт `user_rules` вместо `routing_toml`.

Сознательное отличие от LLD §3.10: снекбар «Правила сохранены. Применятся
при следующем подключении» показывается только при живом туннеле; в Idle
сохранение молчаливое (список сам себе подтверждение, меньше шума).

## Сценарий проверки (владельцем, на устройстве)

1. На вкладке «Серверы» появилась секция «Маршрутизация» со сводкой
   («Мои правила: N · пресет russia»), тап открывает редактор.
2. Карточка пресета показывает имя, версию, число правил, дату; «Обновить
   сейчас» отвечает «Пресет vN актуален» (или «обновлён до vN+1» после правки
   пресета на хабе); «Подробнее» открывает read-only список правил пресета.
3. «Добавить правило»: ввод `10.0.0.0/8` подписывается «IP-диапазон (IPv4)»,
   `github.com` -> «Домен», `*.github.com` -> «Домен с подстановкой», `ffff` ->
   красное «Некорректный формат» и заблокированный Save.
4. Тап по пилюле правила переключает proxy <-> direct; меню `⋮` двигает и
   удаляет; порядок переживает перезапуск приложения (user_rules.json).
5. Кнопка `{ }` в шапке редактора показывает собранный TOML (мои правила
   сверху, пресет ниже), «Скопировать» кладёт полный текст в буфер.
6. Правило `youtube.com -> direct` при активном пресете с youtube: после
   переподключения exit-IP для youtube реальный (проверка по
   cloudflare/cdn-cgi/trace, не по HTTP-200), остальной пресет проксируется.
7. Добавление правила при подключённом туннеле показывает снекбар про
   «применятся при следующем подключении»; после реконнекта правило работает.
8. Обновление со старой версии: кастомные домены из профиля видны в «моих
   правилах» как proxy-правила; чипов Russia/Proxy all/Custom в редакторе
   сервера больше нет; пресет Russia без хаба не применяется.
