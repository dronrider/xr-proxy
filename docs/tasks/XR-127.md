# XR-127. Раздача папки одной командой на любой ОС

## Зачем

Сейчас расшарить папку с чистой машины это одна команда только на Linux и
Windows: `install.sh`/`install.ps1` с `--token --relay --invite --dir` качают
бинарь, ставят службу с мандатом и сразу регистрируют шару. На маке та же
команда разваливается в простыню из пяти шагов (`-c ~/... --no-service`, ручной
`share`, запуск в foreground), потому что в агенте нет launchd-службы и `install`
спотыкается на `service_install`. Плюс даже на Linux/Windows пользователь держит
в голове два секрета (reg-токен и инвайт) и флаг `--relay`, хотя за него можно
решить самому.

Цель: одинаковая однострочная команда на всех трёх ОС, минимум того, что надо
вписать руками.

```
curl -fsSL https://xr-hub.zoobr.top/share/install.sh | sudo sh -s -- --setup <ONE> --dir ~/folder
```

## Что делаем

1. **launchd на маке.** `service_install`/`uninstall`/`status` получают ветку
   `target_os = "macos"`: LaunchDaemon `/Library/LaunchDaemons/top.zoobr.xr-share.plist`
   (запуск от root, `RunAtLoad` + `KeepAlive`, лог в `/var/log/xr-share.log`),
   поднимается через `launchctl`. Тогда `install --token` на маке отрабатывает
   как на Linux и `install.sh` без ветвлений идёт по общему пути.

2. **relay по умолчанию.** У `xr-share share` relay-плечо включается само, когда
   мандат принёс relay-дескриптор (`cfg.relay.is_some()`), и гасится флагом
   `--no-relay`. Прямой путь всё равно предпочтительнее у консьюмера, relay несёт
   трафик только когда direct недоступен, поэтому вопрос «я за NAT или нет» с
   пользователя снимается. Старый `--relay` остаётся (форсит on, обратная
   совместимость `install.sh`/`ps1`).

3. **setup-токен вместо двух.** Админка выпускает один blob
   `base64url("<reg-токен>.<инвайт>")` через `POST /api/v1/admin/shares/setup-token`
   (внутри переиспользует выпуск reg-токена и инвайта). `xr-share install --setup
   <blob>` разбирает его обратно: reg-токен идёт в обмен на мандат, инвайт
   сохраняется в конфиг как `default_invite`. Каждый последующий `share` без явного
   `--invite` привязывается к `default_invite`. `install.sh`/`ps1` получают
   `--setup`/`XR_SETUP`; при нём `--relay`/`--invite` в команде уже не нужны.

## Границы

- APK не трогаем: консьюмер и так ходит direct-then-relay, привязка к инвайту
  живёт на стороне агента.
- Пересекается с LLD-25/XR-122 (развязка срока жизни и срока действия инвайта):
  setup-токен привязывает шару к инвайту на стороне агента, это ортогонально
  переработке жизненного цикла инвайта у консьюмера. Когда LLD-25 введёт мандат
  на клейме, привязка агента к инвайту остаётся как есть.
- Autostart-бинарь для мака в релизном пайплайне и x86_64-сборка это XR-126,
  едет вместе (launchd бесполезен без выложенного бинаря).

## DoD

- [ ] launchd-ветка в `setup.rs`, `install`/`uninstall`/`status`/`service` на маке
      работают, конфиг по дефолтному пути (`/etc/xr-share`, root).
- [ ] `share` включает relay по умолчанию при наличии relay-дескриптора, `--no-relay`
      гасит; `--relay` сохраняет прежний смысл.
- [ ] `default_invite` в `AgentConfig`, авто-привязка в `share`.
- [ ] эндпоинт `setup-token` в хабе + `install --setup` в CLI, разбор blob, тесты
      на выпуск/разбор и на срок годности.
- [ ] `install.sh` и `install.ps1` понимают `--setup`/`XR_SETUP`.
- [ ] `cargo test --workspace` зелёный, без warnings.

## Сценарий проверки

1. В админке (или API) выпустить setup-токен, привязанный к инвайту, ttl запас.
2. На чистом маке за NAT (без проброса 8443) выполнить одну команду:
   `curl -fsSL https://xr-hub.zoobr.top/share/install.sh | sudo sh -s -- --setup <ONE> --dir ~/TestShare`.
   Ожидаемо: бинарь скачан и проверен, launchd-служба поднята, шара
   зарегистрирована и привязана к инвайту, агент работает после закрытия
   терминала (перезапуск мака переживает).
3. В приложении добавить сервер по тому же инвайту, открыть Files, скачать файл
   из шары. Мак за NAT => direct недоступен => скачивание идёт через relay
   (подтверждается транзитом на relay 8444).
4. Повторный запуск той же команды с другой папкой (`--dir ~/Other`) не плодит
   новую личность и не рушит существующую шару, только добавляет путь.
