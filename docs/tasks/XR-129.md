# XR-129. Режим доступа к шаре: auto/direct/relay + кеш вердикта + health-ручка

Расширение LLD-23. Реализуется через LLD-26 (пишется отдельным чатом). Ниже
дизайн-бриф с разбором, чтобы контекст не потерялся.

## Мотивация

XR-128 сделал устойчивый дефолт: консьюмер идёт direct-first, коротко прощупывает
прямой путь и уходит на relay, если direct молчит. Претензия по дизайну: зачем
прощупывать постоянно, не проще ли зафиксировать тип шары при добавлении и потом
переключать вручную.

Фиксировать тип один раз при добавлении как **единственный** механизм нельзя:
достижимость это не свойство шары, а свойство четвёрки (потребитель, агент,
сетевой путь, момент времени).

- IP агента плавает (DHCP, переезд ноута между сетями), тип протухает.
- Достижимость разная у разных потребителей: телефон в той же LAN дотянется
  напрямую, друг за CGNAT нет. Один тип на шару не покрывает обоих.
- Проверка при добавлении смотрит с точки хаба/агента, а важна точка потребителя
  в момент доступа: это разные вопросы.
- Транзиентные сбои (ребут роутера, моргнул провайдер) прощуп переживает, а
  фиксированный тип нет.

При этом претензия «зачем постоянно» бьёт в две реальные вещи: прощуп повторяется
на каждую операцию (и на каждый файл при пофайловом скачивании, хвост из ревью
XR-128), и нет ручного оверрайда, когда владелец точно знает ситуацию.

## Предлагаемый дизайн

1. **Поле `mode` у шары (`auto` | `direct` | `relay`)**, оверрайд на стороне
   хаба, переключается мгновенно без перепрощупа.
   - `auto` (дефолт): текущее поведение (прощуп direct + fallback на relay).
   - `relay`: агент заведомо за NAT, прощуп direct пропускаем, сразу relay
     (экономит дедлайн прощупа).
   - `direct`: у агента стабильный белый IP, relay не трогаем.
   Живёт в `ShareRecord` и в гранте (`ShareGrant`/`RelayGrant`), едет к консьюмеру.
2. **Кеш auto-вердикта на агента с коротким TTL** (порядка 30-60с) в xr-core:
   серия открытий и скачиваний переиспользует одно решение вместо перепрощупа
   на каждую операцию/файл. Прямой ответ на «постоянно проверяем».
3. **`/health`-ручка у агента xr-share**: без auth, крошечный ответ, для прощупа
   чище, чем неавторизованный GET `/manifest` (сейчас отвечает 401). Может заодно
   отдавать версию агента и статус relay-аплинка для диагностики. Важно: health
   проверяет **консьюмер по прямому адресу агента**, а не хаб (relay доступен
   всегда, вопрос только в достижимости direct с точки консьюмера).
4. **Умный дефолт `mode` при регистрации**: агент/хаб при добавлении тестит
   публичный адрес и ставит начальный `auto`/`relay`. Именно подсказка для дефолта,
   а не замена прощупа.
5. **UI-переключатель** режима на карточке шары в приложении (Android без
   автотестов, логика и кеш тестируются в xr-core).

## Границы и связи

- Поверх задеплоенного XR-128: тот остаётся правильным дефолтом (`auto`), эта
  задача добавляет контроль и кеш, ничего не ломая.
- Затрагивает xr-proto (поле в ShareRecord/грантах), xr-hub (API смены mode,
  умный дефолт), xr-core (кеш вердикта, учёт mode), xr-share (health-ручка),
  xr-android (UI). Отсюда LLD, а не просто задача.
- Не срочно: общий случай уже закрыт устойчивым дефолтом.
