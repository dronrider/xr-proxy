# XR-087: нагрузочный стенд для воспроизведения зависаний прокси

Полноценный стенд для воспроизведения и регресс-проверки зависаний (служит
XR-086, где синтетический флуд из одного источника дедлок не берёт, а реальная
многопоточная нагрузка берёт).

## Идея владельца

На резервном VPS (aeza) поднять маленький тест-сервер, отдающий РАЗНЫЕ виды
нагрузки, а нагрузку подавать с ноутбука через роутер, который по дефолту
проксирует на timeweb. Так проксирующий сервер (timeweb) и цель нагрузки (aeza)
разведены, одно не искажает другое. Путь: ноут (LAN) -> роутер (TPROXY) -> mux
к timeweb (xr-server) -> timeweb коннектится к тест-серверу aeza -> ответ
обратно.

## Состав

- Тест-сервер (эндпоинты): `/download?size=N` (крупная сплошная выгрузка,
  забивает mux writer сервер->клиент), `/slow?rate=R` (медленная капля,
  backpressure), `/small` (крошечный ответ, высокочастотный churn открытий),
  `/burst`.
- Нагрузчик на ноуте (развить flood.py): параметры
  concurrency/pattern/duration. КЛЮЧЕВОЙ сценарий под гипотезу XR-086 =
  несколько крупных загрузок, читаемых МЕДЛЕННО (забить серверный writer_tx),
  ОДНОВРЕМЕННО со шквалом новых мелких open'ов (нужен ConnectAck через
  переполненный writer) -> должно застопорить mux_handler на ConnectAck.
- Роутинг: тест-домен резолвить на aeza-IP и добавить правило `=> Proxy` (или
  проксируемый порт).
- Наблюдение: hangwatch + WARN `send_frame ... blocked >2s` на клиенте и
  сервере + метрики тест-сервера.

## DoD

Воспроизводимый прогон, надёжно вызывающий зависание на баговом бинаре и НЕ
вызывающий на фикс-бинаре (системный регресс-тест поверх юнитов).

## Прогресс (начато 2026-07-06)

`tools/loadtest/`: testserver.py, loadgen.py, README. Тест-сервер поднят на
aeza (systemd `xr-loadtest`, TLS:9443), путь через прокси ПОДТВЕРЖДЁН
(тест-сервер видит источником exit-IP timeweb). Грабли зафиксированы в README:
роутер исключает IP серверов из редиректа, поэтому для проксирования
`aeza:9443` нужен точечный `nft insert ... redirect to :1080` перед возвратом
по IP сервера (ставится на роутер, за которым ноут; временное правило на прод
убрано после прогона); ноут кочует между роутерами; пер-девайс байпасы.

Первый прогон (slow=50 churn=500) затык НЕ вызвал: высокая параллельность с
ОДНОГО ноута упирается в установку TLS-соединений (масса таймаутов хендшейка,
до mux не доходит, тест-сервер получил единицы коннектов). Осталось тюнить
генерацию (несколько источников/ближе к роутеру/иной микс).
