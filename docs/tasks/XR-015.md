# XR-015: Автоустановка (ZTP), этап 1а: каркас идемпотентных шагов `xr-setup` и server-профиль (доставка бинаря, конфиг, systemd, sysctl, инвайт по завершении). Ядро LLD-13, гейтит XR-177 и XR-171

Дизайн: [LLD-13](../lld/13-zero-touch-provisioning.md), разделы 2.1, 3.1-3.3, 4.
Этот файл держит решения, принятые при реализации этапа 1а, ход работы и
сценарий проверки; дизайн не пересказывает.

## Решения по ходу реализации

- **Источник бинарей.** Установщик не несёт бинари в себе (раздуло бы его до
  десятков мегабайт), а берёт их из `--from-dir <dir>` (локальная директория,
  накиданная scp) или качает с `--dist-url <base>` (тот же формат, что
  share-dist: файлы `xr-server-linux-<арка>` плюс `SHA256SUMS`, скачанное
  сверяется по хешу). Если бинарь уже стоит, а источника нет, шаг считается
  пройденным: доустановка без сети возможна.
- **Инвайт на повторных запусках.** Пароль админа хаба генерируется при первой
  установке; argon2-хеш уезжает в конфиг хаба, а сам пароль в
  `/etc/xr-hub/admin.pass` (0600, только root). Повторный запуск читает его
  оттуда и может сминтить свежий инвайт без участия владельца, что понадобится
  и SSH-обёртке этапа 2. Плейнтекст на диске рядом с конфигом, где уже лежит
  и signing-ключ, ничего нового к угрозе root-доступа не добавляет; сменить
  пароль можно штатным `xr-hub reset-password`.
- **Хаб слушает 127.0.0.1:8080.** Как на живом флоте: наружу хаб выходит через
  nginx с TLS, и это ручной follow-up (LLD-13 п. 5.7). Инвайт печатается сразу
  (минтится через localhost), ссылка начнёт открываться у получателя после
  настройки nginx; установщик про это прямо говорит в итоговой сводке.
- **Адрес сервера в инвайте.** Флаг `--server-addr`; без него берётся первый
  публичный адрес из `hostname -I` (на типовом VPS это и есть внешний IP),
  если не нашёлся, ранний отказ с подсказкой.
- **Ключ и salt.** Ключ 64 случайных байта base64 (как generate-key.sh), salt
  случайный u32. Существующий `server.toml` не перетирается без `--force`,
  ключ и salt для дефолтов инвайта читаются из него.
- **Router-профиль** заложен в CLI и enum профилей с первого дня (LLD-13
  п. 3.1), но `xr-setup router` пока завершается понятной ошибкой: реализация
  в XR-177.

## Ход работы

- Каркас шагов `Step { check, apply, verify }` и раннер с отчётом: крейт
  `xr-setup`, чистые функции рендера конфигов и плана покрыты юнитами.
- Server-профиль: доставка `xr-server`/`xr-hub`, `server.toml`, sysctl
  (bbr/fq и буферы 8М), systemd-юниты, signing-ключ и конфиг хаба, по
  завершении логин в хаб и печать одноразового инвайта.
- Хаб раздаёт `setup-dist` (`GET /api/v1/setup/{file}`) по образцу share-dist;
  тонкая обёртка `scripts/install.sh` качает `xr-setup` и запускает его.

## Сценарий проверки (агентский)

Прод не трогается: чистый VPS эмулируется контейнером с systemd (Docker на
маке, aarch64). Шаги:

1. Собрать musl-бинари под арку контейнера: `cargo zigbuild --release --target
   aarch64-unknown-linux-musl -p xr-setup -p xr-server -p xr-hub`, сложить в
   директорию вместе с `SHA256SUMS`.
2. Поднять контейнер ubuntu с systemd (privileged, `/sbin/init`), скопировать
   туда директорию.
3. В контейнере: `xr-setup server --with-hub --hub-domain hub.test
   --server-addr 203.0.113.1 --from-dir /opt/dist` завершается успешно; вывод
   содержит строку `https://hub.test/invite/` (инвайт выдан).
4. `systemctl is-active xr-proxy-server xr-hub` в контейнере печатает две
   строки `active`; `ss -tlnp` показывает 8443 и 127.0.0.1:8080.
5. Повторный запуск той же команды завершается успешно, все шаги в статусе
   «уже настроено», ключ в `/etc/xr-proxy/server.toml` не изменился (сравнить
   до и после), и напечатан свежий инвайт.

Ожидаемый итог: п. 3-5 выполняются без ошибок, каждый критерий проверяется
кодом возврата или конкретной строкой вывода.
