# XR-168. Структурирование шары с устройства: директории, move/copy

Дизайн-бриф для будущего LLD (пишется отдельным чатом). Здесь разбор текущей
модели и контур решения, чтобы контекст не потерялся.

## Мотивация

Write-контур уже есть (LLD-28: PUT/DELETE файла, LLD-29: импорт по URL), но из
приложения шару можно только пополнять: файл ложится в открытую папку, и на этом
всё. Хочется структурировать шару прямо с устройства: создавать директории любой
вложенности, перемещать и копировать между ними файлы, перемещать директории.

## Что мешает в текущей модели

1. Манифест перечисляет только regular files (`build_manifest` и `build_listing`
   в [manifest.rs](../../xr-share/src/manifest.rs) пропускают всё остальное,
   тест `manifest_skips_symlinks_and_dirs` это фиксирует). Директория существует
   лишь как префикс пути файла, пустая директория потребителю невидима вовсе.
   «Создать папку» без расширения модели бессмысленно: папка пропадает из вида,
   пока в неё не положат файл. Sentinel-файл мусорит шару, а служебный неймспейс
   `.xr-` вырезается из манифеста целиком, так что честный путь один: директории
   как сущности манифеста (отдельный список в `ShareManifest` либо признак у
   entry). Формат ломаем без слоя совместимости, парк тестовый.
2. У агента нет серверных операций структуры: PUT создаёт родителей через
   `create_dir_all`, DELETE берёт только файл (на директорию 409), ни move, ни
   copy, ни mkdir в [server.rs](../../xr-share/src/server.rs) не существует.
   Гонять move через скачивание и перезаливку с телефона не вариант: двойной
   трафик через устройство и потеря атомарности.
3. Синк дифает строго по path (`plan_with_selection` в
   [sync.rs](../../xr-core/src/sync.rs)): серверное перемещение обернётся
   delete+fetch, то есть полным перескачиванием зеркала. Вынесено в XR-169,
   эту задачу не блокирует.

## Контур решения (детали в LLD)

1. **Директории в манифесте.** Walk агента отдаёт и директории (пустые в том
   числе), потребители (дерево в xr-core и в приложении) начинают их видеть,
   синк зеркалит создание и удаление пустых директорий локально. Механизм
   подписи манифеста не меняется, меняется только тело.
2. **Эндпоинты агента под `share:write`**, новый scope не нужен, хаб и минтинг
   не трогаем: `POST /{id}/mkdir` c `{path}`, `POST /{id}/move` и
   `POST /{id}/copy` c `{from, to}`. mkdir создаёт вложенность разом, как
   `create_dir_all`. move работает для файла и директории (`fs::rename`),
   copy только для файла: рекурсивная копия директории дорогая, прогресс
   стримить некуда, и запроса на неё нет. Оба конца пути идут через
   `resolve_within` (safepath уже готов к несуществующим путям, `.xr-`
   ловится там же). Занятый `to` даёт 409 без перезаписи, перезапись это
   явный delete перед move. move директории внутрь самой себя запрещён.
3. **Хеш-кеш агента.** Ключи там по абсолютному пути, при move файла запись
   пересеивается на новый путь, при move директории перевешивается префикс.
   Иначе листинг после перемещения отдаёт пустые sha до ленивого прогрева.
4. **Клиент.** Логика в xr-core (общая для Android и харнесса), JNI-обёртки
   `nativeMkdir`/`nativeMove`/`nativeCopy`. В приложении действие «создать
   папку» в текущей директории и контекстные «переместить»/«копировать» с
   пикером целевой папки по дереву манифеста; rename файла и папки выходит
   бесплатно (move в ту же директорию под новым именем). Контролы видны только
   при `share:write` в гранте, как импорт при `share:import`. Android без
   автотестов, логика тестируется в Rust. Тексты UI согласовать до реализации
   (урок XR-049).
5. **Харнесс.** Подкоманды `xr-share mkdir`/`mv`/`cp` рядом с `push`/`rm`,
   ими же гоняются интеграционные тесты агента.

## Известные ограничения

- Конкуренция с импорт-джобой: джоба публикует результат в путь, взятый на
  старте, и move директории под активной джобой воскресит старый путь через
  `create_dir_all`. Фиксируем как известное поведение, гейт не городим: джобы
  короткие, окно узкое.
- Windows-агент: rename файла, который в этот момент раздаётся ридеру, падает
  с sharing violation. Отдаём честную ошибку, клиент повторит.

## Границы и связи

- Затрагивает xr-proto (манифест), xr-share (эндпоинты, кеш, харнесс), xr-core
  (клиент, синк пустых директорий), xr-android-jni и xr-android (UI). Отсюда
  LLD, а не просто задача.
- Оптимизация синка под серверный move это XR-169.
- Заливка локального файла с устройства в шару сюда не входит: её в приложении
  нет вообще, и это отдельный разговор.
- С XR-163 (переименование шары в приложении) стык только терминологический:
  там алиас самой шары, тут файлы и папки внутри неё.
