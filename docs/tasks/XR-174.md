# XR-174. Мгновенный подхват нового правила хаба

Цель: добавил правило на хаб, и ресурс открывается сразу, без ожидания. И под
роутером с xr-client, и в Android. Дизайн-бриф ниже, LLD пишется отдельным чатом.

## Как устроено сейчас

Горячая подмена правил уже работает: и xr-client, и движок Android держат
`Router` за `RwLock<Arc<Router>>`, фоновая задача при новой версии пресета
собирает новый роутер и подменяет его без рестарта туннеля
([xr-client/src/main.rs:199](../../xr-client/src/main.rs#L199),
[xr-core/src/engine.rs:272](../../xr-core/src/engine.rs#L272)). Вся задержка
сидит в опросе: `fetch_if_stale` гоняется раз в `hub_refresh_interval_secs`,
дефолт 300 секунд ([xr-proto/src/config.rs:353](../../xr-proto/src/config.rs#L353)),
Android жёстко подставляет те же 300
([VpnViewModel.kt:1200](../../xr-android/app/src/main/java/com/xrproxy/app/ui/VpnViewModel.kt#L1200)).
В худшем случае правило доезжает через пять минут. Мгновенный путь один: кнопка
«Обновить сейчас» на экране правил Android (`nativeRefreshPreset`); на роутере
только рестарт сервиса.

Хаб к быстрой доставке уже наполовину готов: `GET /api/v1/presets` отдаёт ETag
из суммы версий, `GET /presets/{name}` отвечает 304 по `If-None-Match`
([xr-hub/src/api/presets.rs:17](../../xr-hub/src/api/presets.rs#L17)), а `PUT`
из админки инкрементит `version`. Клиентский `PresetCache::refresh` условными
запросами не пользуется, сравнивает `version` из списка числом
([xr-core/src/presets.rs:84](../../xr-core/src/presets.rs#L84)). Пуша от хаба к
клиентам нет.

## Развилки дизайна

1. Long-poll или SSE: клиент висит на запросе со своей версией, хаб отвечает
   сразу при bump версии пресета либо по таймауту. Дёшево для хаба, ложится на
   существующий цикл refresh, входящей достижимости клиентов не требует.
   Выглядит основным кандидатом.
2. Частый условный опрос (интервал в секунды, ответ 304): проще всего, но на
   Android будит радио и ест батарею, плюс вопрос поведения в Doze.
3. Push через туннель: постоянного соединения с хабом у клиентов нет, туннель
   идёт к xr-server. Новая связность хаб-сервер-клиент для этой задачи перебор;
   подписанная команда `apply_preset` из LLD-20 живёт в той же плоскости, но
   поверх реестра XR-025.

Учесть в дизайне: механизм класть в общий код (`PresetCache` в xr-core), чтобы
роутер и движок получили его одним куском; батарею Android (long-poll против
Doze, поведение в фоне); лимиты соединений на хабе при выросшем парке; обрыв
канала (backoff и откат на текущий 300-секундный опрос как заглушка). Уже
открытые соединения доигрывают со старым действием, новые видят правило сразу;
для «открыть сайт» этого достаточно.

## Границы и связи

- XR-118: resume на Android поднимает туннель из устаревшего lastConfigJson,
  свежие user_rules мимо. Смежная дыра той же боли, но про локальные правила и
  жизненный цикл сервиса, чинится отдельно.
- XR-020: живые правила хаба для серверов, добавленных вручную. Эта задача
  ускоряет доставку там, где подписка на пресет уже есть; XR-020 расширяет круг
  подписчиков.

## Черновик проверки

Добавить домен в пресет на хабе. В пределах нескольких секунд, без рестартов и
без кнопки «Обновить сейчас»: новое соединение с LAN-устройства под роутером
уходит через прокси, ресурс открывается на Android. Проверять по exit-IP, не по
HTTP 200.
