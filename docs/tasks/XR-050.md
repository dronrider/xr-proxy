# XR-050: шара с несколькими адресами (LAN + публичный)

Потребитель выбирает достижимый адрес.

## Проблема

Сейчас у шары один addr: на одной LAN с агентом публичный IP недоступен из-за
NAT hairpin (роутер не заворачивает LAN -> свой WAN-IP обратно на 192.168.x), а
LAN-адрес агента недоступен снаружи.

## Что сделать

Хранить в записи шары список адресов (LAN-адреса агента + публичный/DDNS),
потребитель пробует их по очереди и берёт первый ответивший: внутри LAN по
LAN-IP без hairpin, снаружи по публичному. Роутеро-независимо, в отличие от
включения NAT reflection на роутере.

Код: [xr-share/src/cli.rs](../../xr-share/src/cli.rs),
[xr-core/src/sync.rs](../../xr-core/src/sync.rs)

## Что сделано

Модель адресов стала списком, обратносовместимо (аддитивное поле `addrs` рядом
с `addr`, старые записи и гранты грузятся как раньше).

- Прото ([share.rs](../../xr-proto/src/share.rs)): `addrs: Vec<String>` в
  `ShareRecord`/`ShareInfo`/`ShareGrant`; `ShareGrant::candidate_addrs()` отдаёт
  LAN-адреса вперёд, публичный `addr` последним, с дедупом и триммингом.
- Хаб ([share_v2.rs](../../xr-hub/src/api/share_v2.rs)): `share/add` принимает
  `addrs`, чистит (тримминг, дедуп, выкидывает дубль публичного) и хранит;
  `invite_shares` отдаёт список в гранте.
- Агент ([cli.rs](../../xr-share/src/cli.rs)): определяет свой LAN-адрес
  UDP-connect-трюком без отправки пакета и анонсирует его как `addrs`, публичный
  вход остаётся за `--addr` (или source-IP хаба).
- Потребитель ([sync.rs](../../xr-core/src/sync.rs)): `direct_then_relay`
  перебирает адреса по очереди, пробуя каждый коротким liveness-probe при наличии
  запасного пути; первый ответивший обслуживает операцию, чужой хост по адресу
  (подпись манифеста не сошлась) и недоступность уводят на следующий, relay
  поднимается последним. CLI `pull`/`push`/`import` перебирают так же
  ([pull.rs](../../xr-share/src/pull.rs) `resolve_base`).
- Android: адреса едут в модели ([Share.kt](../../xr-android/app/src/main/java/com/xrproxy/app/model/Share.kt))
  и в натив через тот же строковый параметр (кандидаты через перевод строки), без
  новых JNI-сигнатур.

## Проверка

Быстрая проверка агентом после выката хаба (объективная, прод не меняет сверх
одной временной шары, которую снимаем):

1. С машины за NAT, где стоит агент: `xr-share share /tmp/xr050-check --addr <публичный/DDNS>`.
   Ожидаемо: в выводе строка `LAN: <lan-ip>:<порт> ...` рядом с публичным адресом.
2. `curl -s https://<хаб>/api/v1/invite/<инвайт>/shares | jq '.[] | {addr, addrs}'`
   для инвайта с этой шарой. Ожидаемо: поле `addrs` содержит LAN-адрес агента,
   `addr` это публичный.
3. Снять временную шару: `xr-share unshare /tmp/xr050-check`.

Полная проверка пользователем (ради hairpin нужен реальный телефон в той же сети):

1. Расшарить папку с домашней машины за NAT (агент новый): `xr-share share <папка>`,
   БЕЗ проброса порта на роутере.
2. На телефоне в той же Wi-Fi открыть шару (приложение с новым APK): файлы
   открываются и качаются, хотя порт наружу не проброшен. Это доказывает заход по
   LAN-IP без hairpin.
3. Переключить телефон на мобильный интернет: та же шара недоступна (порт не
   проброшен) либо, если проброшен и задан `--addr`, открывается по публичному
   адресу. Это доказывает, что снаружи берётся публичный кандидат.
4. Регресс: старые уже добавленные шары (без `addrs`) продолжают открываться по
   единственному адресу.
