# XR-093. Авиарежим: раздел Files встречает ошибкой про не найденные шары, интерфейс подвисает

## Симптом

В авиарежиме вход в раздел файлов показывает ошибку, что шары не найдены,
хотя список шар уже известен и лежит в сторе. Несколько раз при этом
подвисал интерфейс приложения.

## Корень

Дефекта два, у каждого свой корень.

**Ошибка вместо списка.** При каждом входе на вкладку стреляет `refreshHub`
(обновление списка шар с хаба по инвайту). Без сети запрос падает, и ошибка
уходила тостом «Шары по инвайту: network: ...» поверх вполне рабочего
сохранённого списка. Сам список из стора при этом никуда не девается, но
пользователь видит ошибку и читает её как «шары не найдены».

**Подвисание интерфейса.** На пути входа в Files нашлись две блокировки
главного потока:

- `ShareStore.create()` в конструкторе `FilesViewModel`: создание
  EncryptedSharedPreferences это IPC к Keystore плюс расшифровка файла
  настроек, на холодном процессе занимает до секунд. Конструктор выполняется
  на главном потоке в момент первого входа на вкладку, ровно когда владелец
  видел фриз.
- Первый `WorkManager.getInstance()` в процессе (из `syncAllNow` при входе
  на вкладку) открывает Room-базу, тоже на главном потоке.

## Что сделано

[FilesViewModel.kt](../../xr-android/app/src/main/java/com/xrproxy/app/ui/files/FilesViewModel.kt),
[FilesScreen.kt](../../xr-android/app/src/main/java/com/xrproxy/app/ui/files/FilesScreen.kt):

- Сетевые ошибки `refreshHub` (префикс `network:` из native-слоя) больше не
  тостятся: ставится флаг `hubOffline`, под заголовком «Файлы» появляется
  мелкая серая строка «Хаб недоступен, показан сохранённый список» (в стиле
  пометки «Офлайн» из проводника). Не-сетевые ошибки (протухший инвайт,
  http_4xx) остаются видимым тостом. Успешное обновление флаг снимает.
- Пустой стейт стал честным: без сети и без сохранённых шар показывается
  «Сети нет, а сохранённых шар пока нет» вместо «Пока нет шар. Обнови
  список...». Тексты согласованы с владельцем.
- Стор создаётся на `Dispatchers.IO` (`storeDeferred`), все обращения идут
  через suspend `store()`. Список шар для UI отдаёт зеркальный `_configs`,
  наполняемый коллектором, так что композиция стор не трогает. Флаг
  `storeReady` не даёт мигнуть пустому стейту, пока стор грузится.
- Вызовы `ShareSyncScheduler` (WorkManager) уведены на IO.
- Заодно с асинхронностью закрыта гонка в `downloadAndOpen`: selection
  персистится строго до старта скачивания (иначе пробегающий mirror-синк мог
  удалить файл, ещё не попавший в selection).

Минутные крутилки внутри самой шары (манифест с таймаутом 60с при поднятом
VPN) это отдельная XR-059 (cache-first), здесь не трогались.

Подтверждение, что фризы ушли, возможно только на устройстве; блокировок
главного потока на этом пути в коде не осталось.

## Сценарий проверки

1. Включить авиарежим (VPN не трогать, как обычно).
2. Открыть приложение, перейти в Files. Ожидаемо: сохранённый список шар
   виден сразу, тоста с ошибкой нет; под заголовком «Файлы» серая строка
   «Хаб недоступен, показан сохранённый список» (при поднятом VPN она может
   появиться с задержкой до 15с, пока запрос не протухнет); интерфейс не
   подвисает ни при первом входе, ни при повторных.
3. Выключить авиарежим, дождаться сети, нажать «Обновить» на вкладке.
   Ожидаемо: строка офлайна исчезает, список обновляется.
4. Опционально, на устройстве без сохранённых шар: в авиарежиме вместо
   «Пока нет шар...» показывается «Сети нет, а сохранённых шар пока нет».
