# XR-139: реализация LLD-28 (запись в шару)

Дизайн и решения в [LLD-28](../lld/28-share-write-scope.md); тут только что
сделано и как проверить. Ветка `xr-139-share-write`, серия коммитов с ID в
subject (proto -> hub -> core -> share -> docs).

Реализовано по слоям:

- `xr-proto`: `scope` в `ShareToken` (подпись v2, v1 отвергается), `SCOPE_READ`/
  `SCOPE_WRITE`, `verify_share_token(required_scope)`, `writable` в `ShareRecord`,
  `write_share_ids` в `Invite`.
- `xr-hub`: `share:write` минтится только в грантах инвайта при write-привязке И
  writable-записи; `add`/`mint`/ссылка/админ дают только `share:read`;
  attach/detach ведут обе оси списков.
- `xr-share`: роуты `PUT`/`DELETE /{id}/file/{*rel}` (safepath, temp+rename, посев
  хеша, `If-Match`/`If-None-Match`, `X-Xr-Sha256`, `max_file_mb`, резерв префикса
  `.xr-part-`), флаг `share --writable`, харнесс `push`/`rm`.
- `xr-core`: `upload_file`/`delete_file` поверх direct/relay со scope-проверкой до
  сети и трансляцией ожидаемого хеша в `If-Match`.

## Проверка

### 1. Автотесты (логика, без стенда)

```sh
cargo test --workspace --exclude xr-client         # на маке xr-client не собирается
cargo test -p xr-share --features relay            # relay-путь агента e2e
```

Ключевое, что должно быть зелёным:

- скоуп токена: `xr-proto` `test_token_scope_sign_verify`, `scope_contains_matches_whole_names`
  (read+write проходит обе проверки, read-only не проходит write, подделка scope
  ломает подпись, v1-блоб отвергается);
- приём агента: `xr-share` `server::tests::test_put_creates_and_overwrites`,
  `test_put_requires_write_scope` (нет токена 401, read-only 403),
  `test_put_readonly_share_rejected`, `test_put_path_traversal_blocked`,
  `test_put_sha256_mismatch`, `test_conditional_requests`, `test_put_cap_exceeded`,
  `test_delete_file`, `test_file_share_not_writable`,
  `manifest::tests::test_manifest_skips_upload_temp`;
- хаб: `share_v2::tests::test_grant_write_scope_only_for_write_binding`,
  `test_attach_write_subset`;
- потребитель: `xr-core` `sync::tests::test_upload_roundtrip`, `test_delete_via_grant`,
  `write_refused_without_scope_before_network`.

### 2. Ручной end-to-end на стенде (агент + хаб)

Позитивный путь. На машине агента (папка уже привязана к инвайту `T`):

```sh
xr-share share /srv/dropbox --writable --invite <T>   # в выводе: "запись: разрешена"
```

С ноутбука:

```sh
xr-share push --invite <T> --share dropbox report.pdf   # "готово, создан файл ..."
# файл появился в /srv/dropbox на агенте; pull/приложение его видят, sha сходится:
xr-share pull --invite <T> --share dropbox --all
xr-share rm   --invite <T> --share dropbox report.pdf   # "готово, файл удалён"
```

Оптимистический контроль: изменить файл на агенте после того, как `push` прочитал
манифест, повторный `push` того же пути отказывает по 412, с `--force` проходит.

Негативные проверки (значения токена и адреса взять из ответа
`GET /api/v1/invite/<T>/shares`: поля `token`, `addr:port`, `share_id`):

```sh
AG=<addr>:<port>/<share_id>; TOK=<token из гранта>
# read-only токен по PUT -> 403
curl -s -o /dev/null -w "%{http_code}\n" -X PUT -H "Authorization: Bearer <read-токен>" --data x http://$AG/file/a.txt
# traversal -> 403
curl -s -o /dev/null -w "%{http_code}\n" -X PUT -H "Authorization: Bearer $TOK" --data x http://$AG/file/../evil
# чужой If-Match -> 412, файл не изменился
curl -s -o /dev/null -w "%{http_code}\n" -X PUT -H "Authorization: Bearer $TOK" -H "If-Match: 00" --data x http://$AG/file/a.txt
# несходящийся X-Xr-Sha256 -> 422
curl -s -o /dev/null -w "%{http_code}\n" -X PUT -H "Authorization: Bearer $TOK" -H "X-Xr-Sha256: dead" --data hi http://$AG/file/a.txt
```

Ожидаемо: `403`, `403`, `412`, `422`. Дополнительно:

- инвайт без write-привязки: `push` падает локально текстом «нет права записи»,
  до сети;
- `writable = false` в конфиге агента при валидном write-гранте -> PUT даёт 403
  (второй рубеж на самой машине данных);
- тело больше `max_file_mb` -> 413, мусорного `.xr-part-*` в папке не остаётся;
- шара за NAT: `push` проходит через relay, файл доезжает.

Полный прод-чеклист (10 шагов) в [LLD-28 п. 6](../lld/28-share-write-scope.md).

## Стыки

Гейтит [XR-052](XR-052.md) (импорт по URL поверх этого safepath-контура) и Android
UI правок шары (отдельная задача поверх `upload_file`/`delete_file`). Колонка
`writable` в SPA админки не собрана (поле есть в JSON `admin_list_shares`, нужен
пересбор встроенного бандла).
