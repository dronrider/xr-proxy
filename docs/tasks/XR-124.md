# XR-124. Продление TLS: серт на Aeza и сторож срока

## Разбор

Инцидент 2026-07-12: сертификат xr-hub.zoobr.top истёк, авто-продление месяц
молча падало (certbot standalone не мог занять :80 под nginx). Горячая часть
устранена тогда же: перевыпуск, pre/post-hook стоп/старт nginx на Timeweb,
серт скопирован на Aeza руками. Хвост из двух частей: Aeza сам продлить не
может (DNS смотрит только на Timeweb, HTTP-челлендж не проходит), его копия
серта протухнет при следующем продлении; и никто не заметит, если продление
снова сломается молча.

## Что делаем

1. Deploy-hook на Timeweb: после успешного продления серт уезжает на Aeza сам.
   Отдельный ssh-ключ, на Aeza зажат forced command на скрипт-приёмник:
   принять tar с fullchain/privkey, свалидировать, забэкапить старые, положить
   в `/etc/ssl/xr-hub/` и перезапустить nginx (reload новый серт не
   подхватывает). Ничего другого этим ключом сделать нельзя.
2. Сторож срока на обоих хабах: ежедневный cron сверяет, сколько дней до
   истечения серта; меньше 14 (продление должно было пройти за 30) - алерт в
   Telegram-бота. Токен и chat_id живут в `/etc/xr-proxy/alert.env` (chmod
   600), в git не попадают.

Скрипты в репозитории: `deploy/cert-sync-aeza.sh` (hook),
`deploy/cert-receive.sh` (приёмник на Aeza), `deploy/cert-alert.sh` (сторож).

## Сценарий проверки

1. На Timeweb прогнать hook руками:
   `/etc/letsencrypt/renewal-hooks/deploy/cert-sync-aeza.sh`. Ожидаемо: на
   Aeza свежие mtime у `/etc/ssl/xr-hub/*.pem`, тот же notAfter, что на
   Timeweb, nginx перезапущен и отвечает.
2. Сторож: на обоих хабах запуск `cert-alert.sh` с заниженным порогом
   (`WARN_DAYS=9999`) приносит тестовое сообщение в Telegram; со штатным
   порогом молчит (до истечения далеко).
3. Настоящее продление в октябре: серт на Aeza обновится сам, алертов нет.
   Пункт долгий, закрывать задачу можно по 1-2, отметив календарную проверку.
