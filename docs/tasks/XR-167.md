# XR-167: старые шары с v1-токеном падают криптичной ошибкой манифеста

Фидбек с прода после выката 0.75.0: на старых шарах вылезает
«Манифест: bad token json: missing field `scope` at line 1 column 57».

## Корень

Шары, добавленные до XR-139, хранят в `ShareStore` v1-токен без поля `scope`.
С XR-139 `ShareToken` в `xr-proto` требует `scope` (v1-совместимости нет
сознательно: подпись v1 всё равно не пройдёт у агента, парк тестовый), поэтому
любой нативный вызов со старым токеном падает уже на
`parse_token` ([xr-android-jni/src/lib.rs:1087](../../xr-android-jni/src/lib.rs#L1087)),
и serde-простыня уходит тостом в UI.

Самолечение есть, но дырявое: `refreshHub -> reconcileShares` перезаписывает
токен свежим грантом инвайта. Не покрыты три случая:

- гонка: вкладка «Файлы» запускает рефреш, но открытие шары стреляет
  `fetchManifest` со старым токеном раньше, чем грант доехал;
- офлайн: parse-ошибка не начинается с `network`, поэтому вместо тихого
  офлайн-состояния показывается криптичный тост;
- шара больше не привязана к инвайту: токен не обновится никогда, ошибка
  постоянная.

## Направление правки

При parse-ошибке токена не показывать serde-текст: человеческое «токен шары
устарел, обновите список по инвайту» и попытка повторить fetch после удачного
рефреша грантов; для шары, пропавшей из грантов, честное «доступ истёк»
(удалить или перевыпустить инвайт). Толерантный parse со скоупом по умолчанию
не выход: агент такой токен всё равно отвергнет по подписи.

## Проверка

На устройстве с шарой, добавленной до XR-139 (или с подложенным в стор
v1-токеном): открыть шару при живом инвайте -> листинг открывается после
подтяжки гранта, без криптичного тоста; в самолёте -> обычное офлайн-состояние;
для шары, отвязанной от инвайта -> сообщение про истёкший доступ, не serde.
