# XR-070: Happy Eyeballs при коннекте к цели вместо первого адреса из DNS

`resolve_target` в [xr-server/src/handler.rs](../../xr-server/src/handler.rs#L107)
собирает все адреса из `lookup_host`, но возвращает только первый
(`addrs.next()`), и `handle_client` коннектится к нему одному с общим
таймаутом 5с.

## Чем это плохо

1. Мёртвый первый адрес (протухшая A-запись, упавший бэкенд за DNS
   round-robin) валит Connect целиком, хотя у домена есть живые адреса.
   Браузер напрямую такой сайт открывает, потому что сам перебирает адреса,
   а через прокси сайт выглядит лежащим.
2. Если первым приходит AAAA, а у VPS нет IPv6-связности, соединение к цели
   съедает весь 5-секундный таймаут при живом IPv4 рядом. Порядок адресов
   отдаёт getaddrinfo, влиять на него мы не можем.

## Что сделать

Happy Eyeballs (RFC 8305) в упрощённом серверном варианте:

- `resolve_target` возвращает весь список адресов, а не первый;
- список пересортировывается чередованием семейств (v6, v4, v6, ...), как
  описано в разделе 4 RFC 8305;
- попытки коннекта стартуют лесенкой: следующая через ~250мс (Connection
  Attempt Delay), если предыдущая ещё не установилась; первый успешный сокет
  побеждает, остальные попытки отменяются;
- общий бюджет на установление остаётся 5с, как сейчас.

Последовательный перебор адресов (его умеет сам `TcpStream::connect` по
списку) как финальное решение не годится: мёртвый первый адрес всё равно
добавляет целый свой таймаут, лесенка из RFC эту задержку и убирает.

Стык с XR-069 (TTL-кэш DNS): кэш живёт в том же `resolve_target` и должен
хранить весь список адресов. Кто первым берётся за свою задачу, тот и
переводит функцию на `Vec<SocketAddr>`.

## Тесты

- Юнит на пересортировку списка (чередование семейств): чистая функция,
  детерминирована.
- Гонка лесенки на замоканном коннекте (попытка инжектится функцией) с
  `tokio::time::pause` вместо реального ожидания.
- Интеграционный регресс: первый адрес из TEST-NET (192.0.2.1, пакеты в
  никуда), второй это локальный листенер. На старом коде тест падает по
  таймауту, на новом соединение устанавливается за ~250-300мс.

## Проверка

1. `cargo test --workspace`: регрессионный тест из раздела выше зелёный
   (и краснеет, если временно вернуть `addrs.next()`).
2. С VPS без IPv6-связности открыть через прокси dual-stack домен, у
   которого AAAA отдаётся первым: страница открывается сразу, без
   5-секундной паузы.
