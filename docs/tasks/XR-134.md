# XR-134. Переонбординг плодит осиротевшие шары, потребитель тычется в мёртвую

## Инцидент

Коллега переонбордила агент (одной командой, setup-токеном), и синк на её
телефоне встал с `network: error sending request for url
(https://127.0.0.1:PORT/<share_id>/manifest)`. Перезапуск агента и приложения не
помогал. Диагностика заняла много шагов, потому что снаружи всё выглядело живым.

Причина: на хабе оказались **две шары «data» с одного адреса, но с разными
identity**:

- старая `hTpOzg1...` (ключ `LWcly...`) на инвайте потребителя `PEwmCKzg`;
- новая живая `BIQ6WRA...` (ключ `wcrh5F6q...`) на инвайте `c9QN7bB4`.

Переонбординг создал **новый identity** и опубликовал шару на **другом инвайте**,
а старая осталась висеть на инвайте телефона. Агента со старым ключом на relay
больше нет, поэтому reverse-стрим к нему не поднимается, и потребитель молча
тычется в мёртвую шару.

## Что чинить

1. **Переиспользовать identity при повторной установке.** `install` идемпотентен
   только когда конфиг по тому же дефолтному пути; установка по другому пути (или
   после ручной установки с `-c ~/...`) генерит новый ключ и осиротяет прежние
   шары. Нужен более надёжный поиск существующей личности (или её импорт), а если
   не найдено, предупредить, что прежние шары останутся без агента.
2. **Детект/прунинг мёртвых шар.** Шара, чей `agent_pubkey` давно не
   регистрировался на relay (агента нет), не должна выглядеть для потребителя
   живой. Варианты: хаб помечает шару stale по отсутствию агента, потребитель
   показывает «агент недоступен» вместо сырой сетевой ошибки.
3. **Понятная ошибка вместо `error sending request (127.0.0.1)`.** Адрес
   loopback это relay-плечо (норма), но текст пугает и не подсказывает, что агент
   недоступен. Мапить обрыв relay-плеча в «шара недоступна, агент не на связи».

## Границы

Не блокер XR-127 (сам онбординг отработал), но вскрылось на его приёмке. Дубль
возник в том числе потому, что для переонбординга использовали временный
тест-инвайт из проверки выката это операционная накладка, но код должен вести
себя предсказуемее (пп. 1-3). Пересекается с XR-125 (рефреш списка) и XR-050
(несколько адресов у шары).

## Что сделано

Ветка `xr-134-dead-shares`, три пункта закрыты так:

1. **Переиспользование identity.** `xr-share install` без `--force` теперь ищет
   конфиг существующего агента не только по запрошенному пути: сначала путь из
   `-c` установленной службы автозапуска (systemd unit / launchd plist / task-XML
   Windows, читается в UTF-16 напрямую, минуя OEM-кодировку консоли), затем
   дефолтный путь ОС. Найденный конфиг переезжает на запрошенный путь целиком
   (личность из `identity.key` инлайнится, старый файл паркуется в
   `*.toml.imported`, чтобы мандатный конфиг остался один), мандат добирается
   по токену, свежий `--setup` перенацеливает только дефолтный инвайт.
   Страховки: конфиг без читаемой личности не переносится; переонбординг на
   другой хаб не подхватывает старый конфиг, а требует явного `--force` (иначе
   смешались бы мандат старого хаба и инвайт нового); токен при уже живом
   мандате не тратится, о чём печатается прямо. Личность создаётся заново
   только если конфиг не найден нигде (при следах службы печатается
   предупреждение об осиротении прежних шар) либо по явному `--force`, который
   после успешной установки best-effort снимает прежние шары с индекса хаба
   (прунинг: мёртвые записи не остаются на инвайтах потребителей; оборванная
   установка хаб не трогает).
2. **Детект мёртвой шары у потребителя.** Relay и раньше отвечал
   `Close(AGENT_OFFLINE)`, но потребитель схлопывал это в безликий обрыв. Теперь
   mux отдаёт причину Close (`MuxStream::close_reason`), relay-клиент называет
   ошибку (`relay: agent offline`), loopback-forwarder держит вердикт последнего
   отказа (флаг, чистится успешным открытием), а sync подменяет сырую сетевую
   ошибку категорией `agent_offline: агент шары не на связи`. Сам relay при
   этом перестал слать AGENT_OFFLINE, когда агент жив, но реверс-стрим не
   открылся (затык mux): такой транзиент уходит как connect-fail и смертью шары
   не выглядит.
3. **Человеческий текст.** Приложение по префиксу `agent_offline` показывает
   «Агент шары не на связи» (открытие шары, ручной синк, скачивание по тапу;
   текст берётся из самой ошибки, источник один в Rust) и помечает шару офлайн
   вместо `network: error sending request (127.0.0.1)`.

Хаб-сторона (stale-пометка шары на самом хабе) не делалась: хаб не знает
liveness агентов на relay (нет канала между ними), это отдельная задача уровня
XR-129; выбранный в задаче вариант «потребитель показывает "агент недоступен"»
закрывает симптом инцидента.

Регрессионный тест инцидента: `test_relay_agent_offline` в xr-relay (настоящий
relay без агента, ассерт на именованный текст ошибки), прогнан `regcheck
--tests xr-relay/src/lib.rs` (краснеет на старом коде). Плюс юниты на причину
Close в mux, именование ошибки в relay_client, маппинг в sync, adopt/refresh
конфига и парсеры путей службы в xr-share.

## Сценарий проверки

Пункт 1 (переиспользование identity, нужен хост с уже настроенным агентом):

1. На машине с работающим агентом посмотреть текущий ключ:
   `grep identity_key /etc/xr-share/config.toml` (или путь из службы).
2. Переонбордить той же командой установки со свежим setup-токеном (как в
   инциденте). Ожидаемо: в выводе «личность, шары (N) и мандат сохранены»,
   identity_key в конфиге не изменился, на хабе НЕ появилась вторая шара
   «data», старая шара осталась на своём инвайте и живая.
3. (Опционально) `xr-share install --force --setup <...>`: в выводе строки
   «прежняя шара ... снята с хаба», на хабе старых записей не осталось.

Пункты 2-3 (мёртвая шара, воспроизведение инцидента):

1. На хосте агента остановить службу агента (systemctl stop xr-share или
   launchctl unload), шару с хаба не снимать.
2. На телефоне открыть Files, открыть эту шару и/или нажать ручной синк.
   Ожидаемо: тост «Агент шары не на связи» (при синке: «Синк «имя»: агент шары
   не на связи»), шара помечена офлайн; в журнале категория agent_offline
   вместо `network: error sending request for url (https://127.0.0.1:...)`.
3. Запустить агента обратно: открытие и синк снова работают.

## Ревью

- --force снимал шары с хаба до валидации ввода: оборванная установка (кривой токен, Ctrl-C) теряла записи живого агента: исправлено
- переонбординг на другой хаб молча смешивал мандат старого хаба с инвайтом нового: исправлено
- миграция конфига без читаемого identity.key переносила агент без личности с рапортом об успехе: исправлено
- после миграции оставались два мандатных конфига: share без -c регистрировал на хабе шару, которую служба не раздаёт: исправлено
- токен при уже живом мандате молча игнорировался, install рапортовал успех: исправлено
- скачивание по тапу показывало «Не удалось скачать» вместо вердикта про офлайн агента: исправлено
- relay слал AGENT_OFFLINE и при неоткрывшемся реверс-стриме к живому агенту: транзиент выглядел смертью шары: исправлено
- schtasks-вывод в OEM-кодировке ломал кириллические пути на Windows, конфиг существующего агента не находился: исправлено
- Mutex<Option<String>> и кросс-крейтовый substring-матч ради одного бита agent offline: исправлено
- дубли: lookup конфига в двух местах, скопированный unshare-блок, литералы пути systemd-юнита, три копии текста ошибки в Kotlin, инлайновый xml-unescape: исправлено
- вердикт agent_offline теряется при отвале агента посреди синка: per-file ошибки остаются network:: отклонено, op-уровневый вердикт есть, per-file причины требуют типизированной прошивки сквозь generic direct_then_relay; следующая попытка покажет agent_offline
- leg-scoped last-writer-wins вердикт форвардера протухнет при параллельных запросах внутри одного плеча: отклонено, внутри плеча запросы последовательные (контракт направлен в док-комментарий поля), флаг чистится успехом
- вместо парсинга трёх форматов служб надёжнее писать путь конфига в свой файл при install: отклонено, парсеры упрощены до токенов и покрыты тестами; отдельный pointer-файл это новая сущность на трёх ОС, вернёмся при реальных промахах
- усыновление legacy-конфига без hub_url молча роняло токен и рапортовало про сохранённый мандат: исправлено
- в ветке конфига без личности печатались противоречащие предупреждения (найден конфиг / конфиг не найден): исправлено
