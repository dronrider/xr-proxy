# XR-094. Сервер с мёртвым DNS/egress выглядит живым для пула: failover не срабатывает

## Инцидент 2026-07-10

После ночного ребута Timeweb VPS (видимо, maintenance провайдера) легли оба
провайдерских резолвера (85.193.93.x, таймаут на прямые запросы). IP-связность
с VPS была (ping наружу ходил), туннель и keepalive работали, поэтому пул
роутера считал сервер здоровым. При этом каждый Connect падал на сервере с
`relay error: failed to lookup address information`, и всё проксируемое
(YouTube и остальное) не открывалось у всех клиентов, пока DNS не починили
руками: `resolvectl dns eth0 1.1.1.1 8.8.8.8` плюс постоянный drop-in
`/etc/systemd/resolved.conf.d/public-dns-fallback.conf` (DNS=1.1.1.1 8.8.8.8).

## Корень

Health сервера в пуле меряется живостью туннеля (слот, keepalive, ConnectAck),
но не успехом самой работы (relay до апстрима). Отказ DNS или egress на VPS
это частичная смерть: транспорт жив, полезная работа нулевая. Класс тот же,
что в XR-082 (интермиттентная связность) и XR-086 (zombie-mux): «сервер
отвечает, но не работает». Кэш DNS (XR-069) смягчил бы удар только на время
TTL, дырку в health он не закрывает.

## Что сделать

- Учитывать успех relay в health сервера: доля Connect'ов, кончившихся relay
  error (особенно resolve-ошибками), как сигнал деградации наравне с
  dead-link детектом. Порог и окно продумать, чтобы фон одиночных
  connect timeout (мусорные IP, недоступные порты) не давал ложных
  срабатываний.
- По сигналу деградации пул переключается на следующий сервер с тем же
  гистерезисом failback, что в XR-082.
- Рассмотреть серверную самопроверку: xr-server сам периодически резолвит
  канарейку и репортит в Pong/статусе, тогда клиент узнаёт о беде без
  статистики по живому трафику.

## DoD

- Стенд: сервер с намеренно сломанным резолвером (или egress) при живом
  туннеле. Ожидаемо: роутер в пределах минуты уводит трафик на следующий
  сервер пула, после починки failback с гистерезисом.
- Фоновые одиночные connect timeout не приводят к переключению.

## Что сделано

Health пула теперь учитывает успех самой работы, а не только живость туннеля.

- Протокол: сбой установки relay уезжает клиенту причиной в payload `Close`
  (байт после stream_id): 1 = упал resolve, 2 = упал connect до апстрима.
  Обычное закрытие остаётся пустым, ошибки уже идущего обмена причиной не
  считаются. Старые клиенты лишний байт игнорируют, старый сервер просто не
  шлёт причин, так что смешанные версии живут вместе.
- Клиент: у каждого `MuxPool` появился `RelayHealth`, скользящее окно исходов
  по live-трафику (бакет 30с, смотрим текущий плюс предыдущий). Успех
  засчитывается первому Data-кадру стрима (ConnectAck не в счёт, сервер шлёт
  его до resolve), сбои по причинам из `Close`. Деградация объявляется, когда
  сбоев одного класса в окне не меньше пяти и строго больше, чем сопоставимых
  успехов; resolve-сбои сравниваются только с доменными успехами, чтобы живые
  IP-стримы (CIDR-роутинг Telegram) не маскировали мёртвый DNS. Одиночные
  connect timeout на мусорные IP порога не набирают.
- `ServerPool::health_tick` первым делом проверяет relay-здоровье активного:
  деградировавший помечается Down, активность уезжает на следующий живой
  сервер (событие `failover (relay degraded)` в журнале). Возврат идёт обычным
  failback с hold-down; если после failback деградация повторяется, слот
  получает тот же экспоненциальный анти-флаппинг-штраф, что в XR-082. Смена
  сети (recycle) сбрасывает окно счётчиков.

Серверная самопроверка канарейкой (репорт в Pong) не понадобилась: сигнала по
live-трафику хватает, а окно счётчиков после ухода с деградировавшего сервера
чистится, чтобы failback подтверждался свежим трафиком. Если захочется ловить
деградацию резерва без трафика через него, канарейка остаётся кандидатом.

## Сценарий проверки

Стенд: оба VPS живые, роутер на пуле из двух. На активном (primary для этого
роутера) ломаем резолвер при живом туннеле:

1. На активном VPS: `resolvectl dns eth0 127.0.0.99` (мёртвый резолвер вместо
   рабочих; вернуть потом `resolvectl dns eth0 1.1.1.1 8.8.8.8`). Drop-in
   `public-dns-fallback.conf` на время проверки убрать или переопределить тем
   же resolvectl.
2. С устройства за роутером открыть пару сайтов (нужен live-трафик, без него
   деградацию мерить не по чему). Первые секунды запросы падают.
3. Ожидаемо в течение ~30-60с: в логах роутера
   `server failover (relay degraded): <primary> -> <backup>`, сайты снова
   открываются, exit-IP (`cloudflare.com/cdn-cgi/trace`) показывает backup.
4. Починить DNS на primary (вернуть резолверы). Ожидаемо: failback на primary
   после hold-down (~60с непрерывного up), exit-IP возвращается. Если DNS не
   чинить, качели failback -> деградация замедляются экспоненциально (штраф
   XR-082), между ними трафик стабильно идёт через backup.
5. Фон: единичные connect timeout (недоступный порт/IP) переключений не
   вызывают - порог пять сбоев в окне 30-60с и только при большинстве сбоев
   над успехами.

Регрессионные тесты: `test_relay_degraded_active_fails_over_to_backup`
(падает на коде без учёта relay-исходов), пороги и окно `RelayHealth` в
`mux.rs`, причины в `Close` в `mux_handler.rs` сервера.
