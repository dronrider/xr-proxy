# XR-175. Очередь импортов по URL в приложении

## Мотивация

Пока идёт импорт, второй запустить нельзя: приложение отвечает «Импорт уже
идёт, дождись или отмени его». Диалог при этом открывается и URL принимает,
отказ приходит уже после отправки. Накидать несколько ссылок и уйти не
получается, а это основной сценарий импорта видео.

## Что сейчас

Очередь на стороне агента уже есть: в [import.rs](../../xr-share/src/import.rs)
один воркер, одна активная джоба, до четырёх в ожидании, при переполнении 429.
Импорты разных пользователей агент и сейчас складывает в одну очередь.
Ограничение чисто клиентское: [FilesViewModel](../../xr-android/app/src/main/java/com/xrproxy/app/ui/files/FilesViewModel.kt)
держит ровно одну `ImportJob` в состоянии экрана и второй запуск гейтит.
Комментарий у гейта честный: вторая джоба осиротела бы на агенте, её нельзя
ни отследить, ни отменить. Чинится трекингом списка, а не снятием гейта.

## Что сделать

1. **Список джоб вместо одиночной.** В `FilesViewModel` вместо `importJob`
   список (jobId, шара, прогресс, состояние queued/running), гейт и сообщение
   убрать. Запуск нового импорта просто добавляет строку.
2. **Опрос всех активных.** Один цикл раз в 2 секунды обходит все джобы
   списка: done убирает строку и обновляет листинг, failed показывает текст
   ошибки, потери сети терпятся как сейчас (3 промаха подряд на джобу).
3. **UI.** Строка на джобу с прогрессом и крестиком отмены, как нынешняя
   одиночная. Джоба в состоянии queued подписывается «в очереди», а не висит
   с пустым прогрессом.
4. **429 от агента** (очередь полна) переводить в человеческий текст, а не
   показывать сырую сетевую ошибку.
5. **Глубина очереди агента.** Четыре места на всех пользователей шары может
   оказаться мало, когда импортируют несколько человек сразу. Вынести глубину
   в блок `[import]` конфига агента, дефолт оставить как есть. Единственная
   Rust-часть задачи, едет с тестом; клиентская часть это Android, автотестов
   там нет по правилам проекта.

## Развилки

- **Чужие джобы.** Каждый видит только запущенное им самим; чтобы показать,
  что шару сейчас грузит кто-то ещё, агенту нужен листинг джоб
  (`GET /{id}/import`). В первый заход не берём, при живом спросе отдельная
  задача.
- **Жизнь за пределами экрана.** Джобы живут в состоянии экрана и забываются
  при закрытии шары, на агенте при этом докачиваются до конца. Минимум для
  этой задачи: не терять список при повороте и уходе в другую шару в рамках
  живой ViewModel. Переживание смерти процесса не требуется.

## Тексты UI (согласовать до реализации, урок XR-049)

- подпись очереди у строки джобы: «в очереди»;
- ответ на переполнение очереди агента: «Очередь импорта заполнена, попробуй
  позже».

## Сценарий проверки

Пользовательский, на проде:

1. Открыть шару с правом импорта, запустить импорт длинного видео.
2. Не дожидаясь конца, запустить ещё два импорта другими ссылками. Ожидаемо:
   отказа «Импорт уже идёт» нет, видны три строки, вторая и третья подписаны
   «в очереди».
3. Отменить крестиком одну из ожидающих: остальные продолжаются как ни в чём
   не бывало.
4. Дождаться конца: два файла появились в листинге без ручного обновления,
   отменённого нет.
