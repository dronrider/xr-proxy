# XR-132. Авто-резюм из доверенной сети при неудачном коннекте молча падает в Disconnected

## Симптом

Настроены доверенные сети: дома VPN на паузе, вне дома поднимается сам. Уходишь
из дома, приложение пытается подняться, но коннект к серверу не удался (новый
аплинк ещё не готов, сервер временно недоступен). Туннель не встаёт и остаётся так
висеть: авто-ретрая нет, надо руками открыть приложение и нажать «Подключить».
Пользователь при этом не в курсе, потому что телефон в кармане.

## Разбор

Уход из доверенной сети запускает `requestResume` -> `publish(Connecting)` ->
`bringTunnelUp`. Если `establish()` вернул null или `nativeStart` отдал ошибку,
дальше по коду:

- [`bringTunnelUp`](../../xr-android/app/src/main/java/com/xrproxy/app/service/XrVpnService.kt#L426-L450):
  `publish(Phase.Error)`, затем `stopInternal()`, `stopForeground(STOP_FOREGROUND_REMOVE)`,
  `stopSelf()`. Сервис умирает, постоянное уведомление снимается.
- То же при фатальной ошибке движка в рантайме:
  [`pollLoop`](../../xr-android/app/src/main/java/com/xrproxy/app/service/XrVpnService.kt#L510-L519).
- В UI [Error маппится в Idle](../../xr-android/app/src/main/java/com/xrproxy/app/ui/VpnViewModel.kt#L1141),
  текст ошибки уходит в снекбар (живёт пару секунд), затем `unbindAndClear()`.

Ретраев коннекта на уровне Kotlin нет. Failover есть только внутри живой сессии
движка (пул серверов, backup). Как только движок отдал `Error:` или `nativeStart`
не поднялся, всё сносится и заново никто не пробует.

Отдельно от неудобства тут есть приватность. Сценарий авто-резюма это ровно тот
случай, когда пользователь не смотрит на экран. При провале уведомление пропадает,
снекбар мелькает в пустоту, на экране `Idle`, визуально неотличимый от «я сам
выключил VPN». Человек настроил доверенные сети ради «вне дома защита сама», а по
факту трафик молча пошёл напрямую и приложение выглядит выключенным.

Асимметрия важная: при ручном «Подключить» падение в `Idle` + снекбар допустимо
(пользователь смотрит на кнопку, которую нажал). При авто-резюме решение
подключаться приняла система, значит и держать ретрай должна она.

## Что решили

Обсудили с владельцем, рамка принята:

1. Отдельная фаза (условно `Reconnecting`), которая держится, видна на экране VPN и
   в уведомлении и читается как «должно быть защищено, пока не вышло, работаем».
   Ключевое отличие от `Paused`: пауза это намеренно напрямую и всё ок, новое
   состояние это ненамеренно напрямую, обрати внимание.
2. Авто-ретрай не сдаётся никогда (пока «должны быть подключены»). Ретрай вешаем не
   на слепой таймер, а на приход коннективности (`defaultNetworkCallback` +
   `nativeOnNetworkChanged` уже есть) плюс backoff, чтобы не жечь батарею в мёртвой
   зоне. Сервис при этом не делает `stopSelf`, foreground-уведомление держим живым.

## Развилки для LLD

- Восьмое (девятое) значение `Phase`: правит оба enum
  ([`Phase`](../../xr-android/app/src/main/java/com/xrproxy/app/service/XrVpnService.kt#L94)
  и [`ConnectPhase`](../../xr-android/app/src/main/java/com/xrproxy/app/ui/VpnViewModel.kt#L48)),
  маппинг, иконку щита, тексты статуса, уведомление.
- Общая машина состояний с ручным коннектом, но политика разная: авто-резюм
  ретраит сам, ручной коннект при падении оставляем как есть.
- Стык с движком: kotlin-ретрай не должен драться с пуловым failover внутри
  `xr_core`. Ретраим только когда движок уже умер, не поверх живой сессии.
- Форма backoff и потолок интервала, поведение при полном отсутствии сети (ждём
  колбэк, а не крутим таймер).
- Взаимодействие с `restrictedNetwork` (сеть, где туннель всё равно не помогает):
  не зациклить ретрай там, где заведомо не поднимемся.

## Границы

Только `xr-android` (плюс, возможно, тонкая ручка в `xr-core` для явного сигнала
«переподключись»). Едет в APK. LLD пишется в отдельном чате по общему порядку.
