# XR-083: mux не детектил blackhole-линк (зомби-слоты до 4ч)

НАСТОЯЩИЙ корень «прокси зависает намертво и НЕ оживает сам даже за час, лечится
только рестартом процесса» (после мультисервер-обновления; до него один Timeweb
работал; роутер DE).

## Корень

mux не детектил blackhole-линк: `reader_task` слал Ping каждые 30с
([mux.rs:434](../../xr-proto/src/mux.rs#L434)), но НИКОГДА не проверял ответный
Pong (комментарий «detect dead links» не был реализован), поэтому при blackhole
(egress тихо дропается, RST нет, `reader.read()` висит без EOF/ошибки) `alive`
оставался `true` до `MUX_MAX_LIFETIME=4ч`. Зомби-mux `is_alive()` ->
`acquire_or_connect` его переиспользует -> `mux_open_stream` виснет. Копились
зомби на ОБОИХ серверах (каждый со своего blackhole-окна), не чистились, и когда
оба слоя пула зомби -> `open_stream` фейлит всё -> с fail-closed (XR-081) блок
навсегда до рестарта.

Второй слой: внешний `PER_SERVER_OPEN_TIMEOUT=5с` (XR-009 Fix A) отменял
`MuxPool::open_stream` раньше внутренней логики
`MAX_CONSECUTIVE_TIMEOUTS -> invalidate_slot`, поэтому зомби не выкидывался; до
мультисервера этого таймаута не было и MuxPool самолечился.

## Фикс (коммит 80e1a3d)

Детект мёртвого линка в `reader_task`: нет входящих/Pong дольше
`DEAD_LINK_TIMEOUT=75с` -> Err -> `alive=false` (тайминги на tokio-часах для
теста), плюс `acquire_or_connect` явно закрывает мёртвый mux перед переподнятием.
Тест `test_reader_detects_dead_link`. Развёрнуто на ОБА роутера (md5 b993e11d).

## Живая проверка

На DE: blackhole активного aeza -> failover на timeweb за 6с, зомби-mux'ы aeza
померли сами на ~+105с (не 4ч), после снятия blackhole слоты aeza переподнялись
за секунды и failback вернулся на aeza, всё БЕЗ рестарта.

## Проверка пользователем

Эпизод «намертво» больше не должен требовать ручного рестарта, пул
самоисцеляется.
