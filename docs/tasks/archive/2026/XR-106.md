# XR-106: pull --select режет значение по запятым

## Симптом

`xr-share pull --invite ... --select "<имя файла>"` для файла, в имени которого
есть запятая, молча скачивает ноль файлов («Готово: 0 файлов»). Поймано при
проверке XR-046 с VPS: торрент-файл «...[5 2026, RUS]...» выбрать флагом
нельзя.

## Корень

`--select` всегда сплитится по запятой как список путей, поэтому имя с запятой
разваливается на куски, ни один из которых не совпадает с путём манифеста.
Вторая половина беды: несовпавшая выборка не диагностируется, фильтр просто
находит ноль пересечений, и pull рапортует «Готово: 0 файлов» без намёка на
причину (ветка «ничего не выбрано» не срабатывает, выборка-то непустая).

## Что сделано

- `--select` стал повторяемым флагом. Каждое значение сначала сверяется с
  путями манифеста целиком; запятая работает разделителем только когда точного
  совпадения нет (старый формат `--select a,b` продолжает работать).
- Куски выборки, не совпавшие ни с одним файлом шары, теперь печатаются
  предупреждением, молчаливого нуля больше нет.

## Сценарий проверки

1. Обновить xr-share (`irm https://xr-hub.zoobr.top/share/install.ps1 | iex`,
   на Linux/macOS `curl -fsSL https://xr-hub.zoobr.top/share/install.sh | sh`).
2. `xr-share pull --invite <инвайт> --dest <папка> --select "<имя файла с запятой>"`
   (подойдёт торрент из шары «Осознанная Меркантильность»). Ожидаемо: файл
   скачивается с `✓`, «Готово: 1 файл(ов)».
3. Опечатка в `--select` даёт предупреждение «не найдено в шаре ...» вместо
   молчаливого «Готово: 0 файлов».
4. Старый списочный формат жив: `--select "a.txt,b.txt"` с двумя настоящими
   именами без запятых скачивает оба.
