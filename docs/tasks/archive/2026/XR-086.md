# XR-086: серверный zombie-mux по лайфтайм-капу

Настоящий корень «проксирование зависает намертво через ~1ч и лечит ТОЛЬКО рестарт»
найден вживую и устранён. Ранее закрытая ctrl/data-раздвойка writer'а (head-of-line
ConnectAck, коммиты 9bfe795/9d286cf) была реальным, но вторичным улучшением: хэнг
воспроизвёлся уже на ней.

## Как нашли

Живой захват на роутере DE (процесс не убивали) + декод mux-кадров по ключу +
серверные логи дали точный механизм. Независимо подтверждено multi-agent воркфлоу
(6 линз сошлись).

## Корень: рассинхрон лайфтаймов на сервере

`handle_mux_client` оборачивает accept-петлю в `timeout(MAX_LIFETIME=1ч)`, а
reader/writer-таски Multiplexer'а detached (держат клоны Arc, не сам Arc) и живут до
`MUX_MAX_LIFETIME=4ч`. Через ~1ч хендлер выходит по капу, дропает `new_stream_rx`,
но mux НЕ закрывался. Дальше 3ч сокет ЖИВ: reader читает (Recv-Q=0), отвечает Pong
на keepalive (dead-link молчит, клиент считает слот живым, `is_alive=true`), старые
стримы релеятся, НО каждый новый Connect молча теряется на
`new_stream_tx.try_send -> Closed` -> ConnectAck не шлётся (new_sids=0) -> клиент
вечно ловит `open timed out`. Пул не переподнимает: `PER_SERVER_OPEN_TIMEOUT=5с` в
ServerPool отменяет `MuxPool::open_stream` раньше внутреннего ConnectAck-таймаута
10с, из-за чего ветка invalidate/breaker недостижима. Когда все 4 слота обоих
серверов старше 1ч -> весь пул глух -> fallback=Block. Рестарт лечит (свежие слоты).

Прямое подтверждение: серверный лог показал те же 4 слота (45182/36630/36634/53534)
`mux session ended (175-180 streams)` в 07:03, а захват в 07:06 застал их
ESTABLISHED, с keepalive и Data старого стрима 349, но глотающими новые Connect.

## Фикс (коммиты 9f6e41e proto + 6230ab8 server)

- `handle_mux_client` при завершении accept-петли ПО ЛЮБОЙ причине зовёт
  `mux.shutdown()`;
- writer при выходе делает ЯВНЫЙ `writer.shutdown()` (FIN), потому что просто дроп
  half-сокета при `tokio::io::split` соединение не закрывал (read-половину держит
  reader), из-за чего shutdown() раньше не рвал зомби;
- `shutdown_notify` переведён на `notify_one` (персистентный пермит, без гонки);
- лайфтайм accept-петли стагерится по эфемерному порту (0..15мин), чтобы 4 слота
  не переподнимались лок-степом.

Детерминированный регресс `server_closes_mux_at_lifetime_cap_no_zombie` (короткий
кап < dead-link: без фикса клиент не получает EOF и тест падает по таймауту, с
фиксом сразу EOF).

## Деплой

Сервер md5 fdbc2327 на оба VPS, клиент md5 156efb27 на оба роутера; exit-IP
verified (LAN за роутером DE выходит через exit-IP timeweb).

## Проверка пользователем

Раньше залипало ИМЕННО спустя ~1ч работы под нагрузкой (слоты зомбировались по
капу); теперь под нагрузкой и через час+ проксирование не встаёт намертво и не
требует kill; в логе роутера не копятся вспышки `open timed out`.
