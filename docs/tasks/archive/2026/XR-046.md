# XR-046: подписанный манифест агента

## Проблема

Манифест шары ездит по data-path как обычный HTTP-JSON без подписи. SHA-256 в
манифесте защищает скачивание только от случайной порчи: активный MITM на пути
потребитель -> агент переписывает и файл, и его хеш в манифесте разом, после
чего проверка хеша радостно подтверждает подмену. Пиннинг `agent_pubkey` в
гранте существовал, но ни к чему не применялся. На relay-пути (LLD-23) без
подписи было бы ещё хуже: там между потребителем и агентом стоит целый VPS.

## Что сделано

Подпись ключом агента поверх точных байтов ответа, проверка по уже пиннутому
`agent_pubkey` из гранта. Fail-closed на потребителе, обратная совместимость по
формату ответа (тело не меняется, подпись в заголовках).

- `xr-proto/src/share.rs`: канонические байты
  `xr-share-manifest\nv1\n{share_id}\n{signed_at}\n` + JSON манифеста как есть
  (`manifest_signing_bytes`, единая точка для подписанта и проверяющего, правило
  `app_update`: подписываем сырые байты, канонизация не нужна). `share_id` в
  подписанной форме закрывает реплей манифеста чужой шары того же агента,
  `signed_at` фиксирует момент подписи. Пара `sign_share_manifest` /
  `verify_share_manifest` + `parse_agent_pubkey`, заголовки
  `x-xr-manifest-sig` / `x-xr-manifest-signed-at`.
- `xr-share`: агент сериализует манифест один раз, подписывает эти байты
  identity-ключом и шлёт их же телом ответа. Ключ берётся из `identity_key`
  конфига (флоу `install`) или файла `identity.key` рядом (флоу `init`);
  без ключа агент служит без подписи с громким warn на старте (легаси-конфиг),
  битый ключ это ошибка старта, а не тихий даунгрейд. `pull` проверяет подпись
  по `agent_pubkey` из гранта хаба.
- `xr-core/sync.rs`: `fetch_manifest`/`sync_share*` принимают `agent_pubkey`;
  при непустом ключе заголовки подписи обязательны и проверяются по точным
  байтам тела до парсинга (`manifest_unsigned` / `manifest_signature: ...`).
  Пустой ключ = легаси-путь без проверки (нечего пиннить).
- JNI (`nativeFetchManifest`, `nativeSyncShare`) и Kotlin
  (`ShareRepository`) прокидывают `ShareConfig.agentPubkey`, который и так
  хранится с гранта.

## Что осталось за рамками (осознанно)

- **Реплей устаревшего подписанного манифеста.** MITM может показать вчерашний
  манифест (скрыть новые файлы, воскресить удалённые) и отдать вчерашние байты
  под вчерашние хеши. Это откат, а не подмена содержимого. `signed_at` уже в
  подписанной форме, так что проверку свежести можно добавить без смены
  формата; полностью путь закроет pinned TLS из LLD-23 (XR-103).
- **Пустые хеши холодного кеша (XR-039).** Подписанный листинг честно говорит
  «хеш ещё не посчитан», и скачивание такого файла не верифицируется. Окно
  сужают XR-045 (ленивый хеш при скачивании) и XR-038 (персист кеша), транспорт
  закрывает тот же pinned TLS.
- **Опц. TLS** из исходной постановки закрывается identity-TLS в LLD-23 (§3.3),
  здесь не дублируем.

## Порядок раскатки

Потребитель fail-closed, поэтому сначала обновить агент(ы), потом APK:
новый APK против старого агента откажет с `manifest_unsigned` (Files перестанет
открывать шару до обновления агента). Старый APK против нового агента работает
как раньше (заголовки игнорирует).

## Сценарий проверки

1. Обновить бинарь `xr-share` на машине агента, перезапустить. В логе старта
   должно быть `manifest signing enabled (agent identity <pubkey>)`, и pubkey
   совпадает с `agent_pubkey` шары в хабе.
2. Поставить свежий APK. Files -> открыть шару: список приходит, скачивание и
   синк работают как раньше (позитивный путь).
3. Негатив на старом агенте (до обновления бинаря или с убранным
   `identity_key`+`identity.key`): открыть шару с нового APK. Ожидаемо: список
   не приходит, в журнале приложения `манифест шары ...: manifest_unsigned`.
4. Десктоп: `xr-share pull --invite <токен>` против обновлённого агента
   скачивает с проверкой; против неподписывающего агента падает с
   «агент не подписал манифест: обнови xr-share на стороне агента».
5. Автотесты: `cargo test --workspace` зелёный; регрессия
   `fetch_manifest_rejects_tampered_body` (подделка тела при валидной подписи
   отклоняется) проверена красной на коде без верификации.
