# XR-123. Relay-онбординг агента одной командой

## Что сделано

Реализовано и выкачено вживую 2026-07-12, доска отставала от факта:

- хаб раздаёт relay-сборки агента в share-dist (с XR-133 их собирает и
  выкладывает CI, relay-гвард в publish не пропустит сборку без фичи);
- `install.sh`/`install.ps1` понимают `XR_RELAY`/`--relay` и
  `XR_INVITE`/`--invite`, при обновлении бинаря без токена перезапускают
  службу;
- публичный `GET /api/v1/relay` на хабе, агент на старте сам тянет
  relay-дескриптор (`fetch_relay_descriptor`, приоритет над конфигом). Это и
  был хвост из строки задачи: обновлённый агент получает relay без повторного
  токена и без правки конфига.

Поверх этого XR-127 упростил сам онбординг до setup-токена (relay по
умолчанию), так что ручные `--relay`/`--invite` остались как обратная
совместимость.

## Сценарий проверки

Пройден вживую 2026-07-12: агент без `[relay]` в конфиге после рестарта сам
зарегистрировался на relay, телефон качал шару через relay при недоступном
прямом адресе. Приёмка XR-127 (одна команда с setup-токеном на маке за NAT,
скачивание через relay) прошла по этому же пути ещё раз. Свежая проверка по
желанию: на машине с агентом убрать `[relay]` из конфига, перезапустить службу
и увидеть в логе `relay reverse tunnel enabled`.
