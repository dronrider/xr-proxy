# XR-097: синк сразу после старта агента перекачивает все файлы заново

Фидбек 2026-07-10 (по ходу проверки XR-059): владелец запустил агент шары,
нажал синк; новых файлов не было (счётчик 0/8), но все 8 файлов начали
скачиваться заново.

## Корень

`/manifest` агента отдаёт листинг без хэширования
([xr-share/src/manifest.rs](../../../../xr-share/src/manifest.rs), `build_listing`,
XR-039): после старта агента кэш хэшей холодный (он в памяти и не
персистится), поэтому у ещё не прогретых файлов `sha256` пустой, его
досчитывает фоновый прогрев. Контракт задокументирован там же: «потребитель
трактует пустой хэш как ещё не посчитанный (не верифицировать / откатиться на
size+mtime)». Для верификации скачивания контракт выполнен
(`download_entry` пропускает проверку при пустом хэше), а вот план синка
([xr-core/src/sync.rs](../../../../xr-core/src/sync.rs), `plan_with_selection`)
сравнивает пустой хэш с локальным «в лоб»: пустая строка не равна настоящему
хэшу, файл попадает в fetch. Итог: синк в окне холодного кэша перекачивает
всю выборку, хотя локальные копии целы.

## Что сделать

В `plan_with_selection` выполнить документированный контракт: при пустом
`sha256` манифеста не считать файл изменённым, а откатываться на сравнение
размера (mtime не годится: скачивание не сохраняет серверный mtime локально).
Файл, реально изменившийся на сервере при том же размере, догонит следующий
синк после прогрева кэша. Для этого `LocalFile` несёт размер файла из
`scan_local_dir`.

## Проверка

1. Перезапустить агент шары (кэш хэшей холодный) и сразу нажать синк на
   телефоне при полностью скачанной выборке. Ожидаемо: перекачки нет,
   тост «+0 −0», трафика по файлам нет.
2. Изменить файл на сервере (размер поменялся), перезапустить агент, сразу
   синк. Ожидаемо: перекачивается только изменённый файл.
3. Дать агенту прогреться (хэши досчитаны) и синкнуть ещё раз. Ожидаемо:
   «+0 −0», содержимое совпадает.
