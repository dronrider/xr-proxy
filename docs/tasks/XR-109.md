# XR-109: скрипт релиза APK одной командой

Релиз APK сейчас это цепочка ручных шагов, которую агент повторяет в каждом
релизном чате: clean + `build.sh --release`, подпись `xr-hub sign-release` с
проверкой подписи, staging-заливка на первичный хаб с бэкапом манифеста и
проверкой `latest`, заливка на второй хаб (rsync --partial) с проверкой через
Host-хедер. Шаги детерминированные, но каждый прогон съедает десятки
tool-вызовов и токенов на логи gradle/scp, а пропуск шага (staging, бэкап
манифеста) выясняется только на устройстве. Прецедент упаковки такого флоу уже
есть: `scripts/release-xr-share.sh`.

## Что делает скрипт

`scripts/release-apk.sh`, запуск с мака из корня репозитория. Шаги:

1. Префлайт: ключ подписи на месте, env-файл прочитан, версия в
   `build.gradle.kts` совпадает с ожидаемой (versionName/versionCode берутся из
   gradle, не передаются руками).
2. Сборка: clean + `xr-android/build.sh --release`, versionCode/pubkey через
   `ORG_GRADLE_PROJECT_*`. Полный лог сборки в файл, в stdout только итог.
3. Подпись: `xr-hub sign-release` локально, следом проверка подписи манифеста.
4. Первичный хаб: бэкап текущего манифеста, файлы в staging-каталог, атомарный
   mv, проверка `latest` по HTTPS.
5. Второй хаб: rsync --partial, проверка `latest` локально через Host-хедер.
6. Итоговая сводка одним блоком: versionName/versionCode, sha256 APK, ответы
   `latest` обоих хабов. Агенту для верификации достаточно этого блока.

Падение на любом шаге останавливает прогон (`set -euo pipefail`). Для
перезапуска с середины (сборка долгая) флаги вида `--skip-build` /
`--only-upload`.

## Конфигурация

Адреса, порты, пути на хабах и путь к ключу живут в гитигнорнутом
`local-docs/release.env`, скрипт его source'ит. В коммитируемом скрипте только
роли (первичный хаб, второй хаб), без IP и доступов.

## Что остаётся за агентом и владельцем

Скрипт не решает, когда и что релизить: бамп versionName/versionCode в gradle,
запись в доску и device-verify остаются как были.

## DoD

- `scripts/release-apk.sh` проходит весь путь от сборки до проверенного
  `latest` на обоих хабах одной командой.
- Секреты и адреса только в `local-docs/release.env`, скрипт коммитится чистым.
- Частичный перезапуск после падения без повторной сборки.
- Строка про скрипт в разделе Scripts проектного `CLAUDE.md`, чтобы агент
  находил его вместо ручной цепочки.

## Сценарий проверки

Выпустить очередную версию APK этим скриптом: один запуск, по сводке видно
версию и sha256, оба хаба отдают новый `latest`, приложение на устройстве
видит обновление.
