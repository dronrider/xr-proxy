# XR-098: синк большой шары висит в «Подготовке» десятки секунд

Фидбек 2026-07-10 (по ходу проверки XR-097): нажатие синка на полностью
скачанной шаре держит прогресс-бар в «Подготовка…» около 40 секунд, потом
«+0 −0».

## Корень

`sync_share_selected` перед диффом зовёт `scan_local_dir`
([xr-core/src/sync.rs](../../xr-core/src/sync.rs)), а тот на каждый синк
заново считает SHA-256 каждого локального файла шары: однопоточно, чтение
куском 64 КБ. Трансфер в этот момент ещё не active, поэтому UI показывает
«Подготовку». На шаре в гигабайты (тем более в выбранной папке на общем
хранилище, где чтение идёт через FUSE-слой Android) пересчёт растягивается на
десятки секунд. Тот же скан гоняет и периодический фоновый синк
(WorkManager), то есть батарея тратится регулярно, не только по кнопке.

## Что сделать

Принести на клиент кэш хэшей по образцу агентского `HashCache`
([xr-share/src/manifest.rs](../../xr-share/src/manifest.rs)): локальный индекс
`(путь, размер, mtime) -> sha256`, персистентный между запусками (файл рядом с
директорией шары или в data_dir приложения). Файл перехэшируется только когда
(размер, mtime) не совпали с индексом; после скачивания хэш кладётся в индекс
сразу (он уже посчитан при верификации). Скан тогда сводится к stat-обходу.

Учесть:
- инвалидация при смене директории шары (миграция XR-043 переносит файлы,
  mtime сохраняется при rename, но путь корня меняется: ключи индекса должны
  быть относительными);
- ручная подмена файла с тем же размером и mtime остаётся за кадром, как и у
  агента (осознанный компромисс);
- индекс не доверяет чужим записям: битый/чужой файл индекса просто ведёт к
  полному пересчёту.

## Проверка

1. Шара в несколько ГБ, полностью скачанная, кэш прогрет (один синк после
   обновления): повторное нажатие синка показывает «Подготовку» меньше
   секунды-двух и «+0 −0».
2. Изменить один файл на сервере: синк перекачивает только его, без долгого
   «Подготовка» перед этим.
3. Первый синк после обновления приложения (индекса ещё нет) допустимо долгий
   (полный пересчёт), второй уже быстрый.
