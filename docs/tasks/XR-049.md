# XR-049: в доверенной сети «Включить здесь» не подключает

Кнопка «Включить здесь» (карточка паузы и действие в уведомлении) должна
поднимать туннель на текущей доверенной сети и держать его, пока телефон
эту сеть не покинет. По факту тап либо не давал ничего, либо туннель
поднимался и тут же откатывался обратно в паузу.

## Корней два

1. **Резюм работал только из фазы `Paused`.** `resumeOverride` звал
   `requestResume`, а тот выходит, если фаза не `Paused`. Тап, прилетевший в
   инстанс сервиса с другой фазой (стоп успел раньше, действие из уведомления
   после развала сессии), молча терялся. Плюс в `VpnViewModel` тап терялся,
   если binder ещё не подключён (`boundService == null`).

2. **Override был привязан к идентичности `Network` и не переживал смену
   netId.** `overrideNetwork` обнулялся в `onDefaultAvailable` при любой смене
   default-сети. Реконнект Wi-Fi (netId-блип, коалесцированные в Doze
   колбэки) выглядит как смена сети при том же SSID: override стирался,
   watcher (backstop после Connected, poll-loop, `maybeEvaluate`) видел
   доверенный SSID без override и честно откатывал туннель в паузу.

## Что сделано

- Override переведён с `Network` на **нормализованный SSID** (`overrideSsid`).
  Матчинг через тот же Rust `ssid_matches` (`nativeSsidMatches`), что и у
  списка доверенных сетей, так что кавычки/регистр/сентинелы не могут
  развести два решения. Логика нормализации уже покрыта тестами в
  `xr-core/src/trusted.rs` (Android-слой в проекте без автотестов).
- Все четыре точки решения об авто-паузе (первичная проверка в `startVpn`,
  backstop после Connected, перепроверка в poll-loop, watcher `maybeEvaluate`)
  пропущены через один гейт `shouldAutoPause`: доверенный SSID и нет
  совпадающего override.
- Дизарм override только когда аплинк доказуемо покинул сеть
  (`maybeDisarmOverride`): читаемый SSID другой сети либо не-Wi-Fi аплинк.
  Нечитаемый SSID (обычное дело при поднятом VPN) выбор пользователя не
  отменяет. Реконнект той же сети с новым netId теперь не паузит, а уходит в
  re-bind (`nativeOnNetworkChanged`).
- `resumeOverride` обрабатывает все фазы: из `Paused` резюм как раньше; из
  `Idle`/`Error` поднимает туннель заново по сохранённому конфигу сессии
  (`lastConfigJson`); в переходных фазах взведённый override направляет уже
  идущий connect-флоу. Свежий инстанс без конфига (смерть процесса) пишет
  WARN в журнал: тут поможет только подключение из приложения.
- `ACTION_START` (обычный Connect) сбрасывает override: новая сессия живёт по
  дефолтной политике, авто-пауза в доверенной сети сохраняется.
- `VpnViewModel.resumeOnTrustedNetwork` при отсутствии binder шлёт
  `ACTION_RESUME_OVERRIDE` интентом, тап не теряется.

## Доработка по итогам проверки: путь назад в паузу

Проверка показала пробел: «Включить здесь» заработало, но вернуть авто-паузу
на той же сети можно было только передёргиванием (Disconnect/Connect), потому
что override снимался лишь при уходе из сети или полном стопе.

Два промежуточных варианта владелец забраковал: вторая карточка с кнопкой
«Вернуть паузу» (формулировка непонятная, экран корявый), затем карточка с
тумблером (метрики сползают вниз, контролов становится больше). Итоговый
дизайн: **никаких дополнительных контролов, состояния несёт главная кнопка**.

- Главная кнопка по контексту: `Connect` (выключено) -> **«Включить здесь»**
  (пауза в доверенной сети, поднимает туннель с override) -> **«Выключить
  здесь»** (туннель включён вручную, возвращает в авто-паузу) -> `Disconnect`
  (обычное подключение вне доверенной сети). Полный стоп из домашних состояний
  живёт в уведомлении («Отключить») и достигается уходом из сети.
- Контекст доверенной сети это компактные строки под статусом (по образцу
  строки «через X (резерв)»): на паузе имя сети + «трафик идёт напрямую, VPN
  включится сам при уходе из сети» + предупреждение об ограничениях; при
  ручном включении «включено вручную · доверенная сеть». Метрики и кнопка
  остаются на своих местах.
- В уведомлении пара действий «Включить здесь» / «Выключить здесь»
  (`ACTION_RESUME_OVERRIDE` / `ACTION_PAUSE_OVERRIDE`) плюс «Отключить».
- Сервис публикует взведённый override в состоянии (`overrideSsid` в
  `ServiceState`), единая точка записи `setOverrideSsid`; при выключении
  override снимается после того, как пауза встала. Если SSID в момент
  выключения не читается, паузу доставит штатный watcher, как только увидит
  доверенный SSID.

## Сценарий проверки

Нужна сеть из списка доверенных (домашний Wi-Fi) и включённая авто-пауза.

1. В доверенной Wi-Fi нажать Connect: туннель уходит в паузу (под статусом
   строка «Доверенная сеть», главная кнопка становится **«Включить здесь»**).
   Нажать её: ожидается «Подключено» без отката в паузу; в журнале «выход из
   паузы: «включить здесь»».
2. Оставить телефон подключённым в этой сети на 10+ минут (пережить
   poll-loop и хотя бы один цикл экран погас/зажёгся): туннель остаётся
   поднятым, самопроизвольной паузы нет.
3. Пока туннель поднят, выключить/включить Wi-Fi на роутере или телефоне
   (реконнект той же сети): туннель остаётся поднятым, в журнале
   «перепривязка туннеля к новому аплинку», паузы нет.
4. Уйти из сети (LTE или другой Wi-Fi) и вернуться в доверенную: туннель
   снова встаёт на паузу, то есть override не пережил уход из сети и
   дефолтная авто-пауза вернулась.
5. То же, что шаг 1, но тапом по действию «Включить здесь» в уведомлении
   (приложение свёрнуто): туннель поднимается.
6. Disconnect, затем Connect в той же сети: снова пауза (override не утёк в
   новую сессию), поведение шага 1 воспроизводится.
7. После «Включить здесь» под статусом строка «включено вручную · доверенная
   сеть», метрики на обычном месте (ничего не сползает), главная кнопка стала
   **«Выключить здесь»**. Нажать её: туннель встаёт на паузу на месте, без
   передёргивания, кнопка снова «Включить здесь». Цикл замыкается одной
   кнопкой в обе стороны.
8. То же из уведомления: действия «Включить здесь» на паузе и «Выключить
   здесь» при ручном включении, «Отключить» это полный стоп.
