# XR-115. Оконный flow control в mux (скачивание через relay обрывается)

Дизайн и реализация в [LLD-27](../lld/27-mux-flow-control.md). Ниже
подтверждённый диагноз и дизайн-бриф.

## Симптом

Скачивание файла через relay обрывается на середине: консьюмер получает
`read: error decoding response body`. Поймано на приёмке XR-103 (видео ~десятки
МБ, потребитель за NAT, путь через relay). Листинг и мелкие файлы работают.

## Причина (подтверждена)

[mux.rs:983-988](../../xr-proto/src/mux.rs#L983-L988). relay держит **один** mux
на агента (registry: pubkey -> mux), все потребители агента идут реверс-стримами
по нему. Reader mux демультиплексирует все стримы одним циклом и не имеет права
блокироваться (иначе HOL для всех стримов, весь mux встаёт). Когда агент отдаёт
байты быстрее, чем медленный потребитель за relay вычитывает, per-stream канал
(`STREAM_CHANNEL_SIZE = 1024` кадра, порядка 8-16 МБ) переполняется, и у reader
остаётся единственный вариант: убить стрим (`remove = true`). Тело обрывается,
reqwest на консьюмере отдаёт `error decoding response body`.

Не таймаут: `splice_lifetime` на relay 3600с. Зафиксировано заранее как
ограничение MVP в XR-103 ([tasks/XR-103.md](XR-103.md)).

## Почему нет дешёвого обхода

- Больше буфер (поднять 1024) только отодвигает переполнение и жрёт RAM relay на
  каждую параллельную скачку. Файл побольше всё равно рвётся.
- Блокировать reader нельзя (HOL для всех стримов).
- Троттлить агента вслепую значит резать скорость и всё равно угадывать.

## Фикс: оконный flow control в mux

Как в HTTP/2 / yamux. Приёмная сторона отдаёт кредит окна (WINDOW_UPDATE) по мере
вычитывания в потребителя; отправитель (агент) шлёт только в пределах окна и
упирается в backpressure, который доезжает до чтения HTTP на агенте. Тогда
per-stream буфер не переполняется в принципе, а скорость сама подстраивается под
самого медленного потребителя.

Выбран **полный** вариант (окно для всех стримов mux), не только для
relay-реверс-стримов.

## Радиус и выкат

`mux.rs` живёт в общем `xr-proto` и используется **и основным прокси**
(роутер <-> сервер), не только relay. Протокольное изменение на живых роутерах в
поле требует согласования возможностей (capability) или бампа версии mux и
координированного выката, чтобы старый роутер не встал. Детали в LLD-27.

## Связи

- Блокирует приёмку XR-103 (скачивание через relay это ядро фичи за NAT).
- Ортогонально XR-128 (тот убрал минутное ожидание на открытии; здесь про обрыв
  тела при передаче).
- Закрывает flow-control-половину XR-071 (тот же механизм убийства стрима на
  основном прокси); за XR-071 остаётся коалесцирование записи в writer_task.

## Что сделано

Ветка `xr-115-mux-flow-control`, реализация по LLD-27:

- окно отправки на стрим 1 МиБ, возврат кредита кадром `WindowUpdate`
  контрольным планом по мере вычитывания (порог полокна);
- согласование capability-байтом в `MuxInit`/`MuxInitAck` без бампа версии:
  пара со старым пиром живёт без окна, порядок выката произвольный;
- отправитель на исчерпанном окне засыпает и просыпается по кредиту или по
  закрытию стрима; kill-on-full оставлен санкцией за превышение окна;
- тесты: регрессия XR-115 (медленный потребитель получает 4 МБ целиком, а
  легаси-пара воспроизводит обрыв; краснота на старом коде проверена
  временным откатом окна), негоциация со старым пиром в обе стороны,
  пробуждение по Close, балк больше окна через io-адаптер.

## Проверка (Check)

Повтор сценария приёмки XR-103, на котором ловился обрыв. После выката
xr-relay на VPS и обновления агента шары за NAT:

1. С потребителя за NAT (путь через relay) скачать из шары файл в десятки МБ,
   тот же, что раньше рвался (видео).
2. Ожидаемо: файл докачивается до конца, без `read: error decoding response
   body`; размер совпадает с оригиналом на агенте.
3. Листинг и мелкие файлы работают как раньше.

До обновления агента пара «новый relay + старый агент» работает по-старому
(без окна), это штатно: окно включается, когда обновлены обе стороны mux.
