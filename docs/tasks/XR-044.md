# XR-044: редизайн синк-UX (одна ось «файл на устройстве» вместо selection/localPaths)

Синк-UX неинтуитивен (фидбек): галочка `selection` (что зеркалить) и `localPaths`
(что скачано) это две разные сущности, пользователь их путает; тап перегружен (и
выбирает, и качает, и открывает), а при single-transfer lock второй тап во время
загрузки молча делает `return`
([FilesViewModel.downloadAndOpen](../../xr-android/app/src/main/java/com/xrproxy/app/ui/files/FilesViewModel.kt#L199)),
поэтому «тапнул второй файл, он не качается».

## Редизайн

Схлопнуть `selection` и `localPaths` в одну ось «файл на устройстве (или едет)».
Заменить чекбокс на per-file контрол-кнопку с состояниями:

- не скачан -> зелёный `+` (тап ставит в общую очередь синка);
- качается -> прогресс на строке + действие «отмена» (тап отменяет, контрол
  всегда с действием, не мёртвый статус);
- скачан -> `−` (тап удаляет локальную копию);
- ошибка -> строка с красным фоном и сохранённым прогрессом на фоне + иконка
  «ретрай» (тап возобновляет с места).

Тап по строке = открыть, но только для скачанного (open-on-tap убирает только
скачивание-по-тапу). Несколько тапов кладут все файлы в очередь (чинит тихий
`return`). Папки: tri-state по конвенции established-приложений
(selective-sync/offline у Drive/Dropbox), тап докачивает недостающее, без
самодельных правил. Видимый прогресс фонового синка на строках (сейчас полоска
только для foreground). Развести «обновить список с хаба» и «синк/очередь»
(криптичная иконка круговых стрелок путает оба смысла). Существующие
`localPaths` мигрировать как selected.

## Стыки

Промпт первого выбора папки (XR-043, сейчас висит на тапе в `downloadAndOpen`)
перевесить на действие `+`, иначе первый синк перестанет спрашивать папку;
проверочные шаги XR-043 завязаны на старый тап-флоу и устареют (закрыть XR-043
до старта). Смежно с XR-056 (общий индикатор, делать следом) и XR-055 (соседняя
карточка шары в том же `FilesScreen`, но это другой виджет, не строка файла).

## Реализация

Очередь живёт на Kotlin-стороне поверх готовой нативной механики (докачка
Range по `.xrsync-part`, снапшот прогресса, отмена, лок «busy»). В xr-core
доехали две вещи: снапшот передачи получил `share_id` (иначе строки двух шар
с одинаковым относительным путём делили чужой прогресс и отмену), а алгебра
снятия выбора переехала в `sync::expand_deselect` рядом с планировщиком, под
юнит-тесты; Kotlin только маршалит JSON через `nativeExpandDeselect`.

- `FilesViewModel`: FIFO-очередь `QueueItem` (identity-класс, не data) с
  последовательным воркером поверх `downloadOne`; на «busy» воркер ждёт лок
  фонового зеркала и пропускает файл, который зеркало уже успело скачать.
  Ошибка складывается в `FailedDownload` с размером partial на диске, ретрай
  кладёт файл обратно (натив продолжает с места). Отмена любой строки идёт
  через один `cancelDownload`: снятие с выбора, изъятие из очереди по
  identity, нативная отмена только если свежий снапшот показывает ровно этот
  share+file. Воркер трактует пропажу элемента из очереди как отмену.
- Единая ось: `selection` держится в ногу с контролами; при открытии шары
  скачанное вливается в выбор (`adoptLocalIntoSelection`, partial-файлы
  отфильтрованы), а выбранное-но-отсутствующее встаёт в очередь
  (`enqueueMissingSelected`): старые галочки и очередь, потерянная со смертью
  процесса, становятся видимыми строками, а не тихой закачкой зеркала. В
  листинг подмешиваются локальные файлы, пропавшие из манифеста, чтобы минус
  был доступен и без фонового синка. Офлайн раскрытие папки не делается
  (локальный манифест неполный), снимаются только точные вхождения.
- Один поллер снапшота (строки, фоновый синк, карточка миграции), медленный
  такт в простое; по концу передачи локальный список перечитывается.
  `removeFolder` сперва рвёт нативную запись под папкой и ждёт ухода писателя,
  потом удаляет поддерево.
- Экран: чекбоксы файлов заменены контролом состояния (зелёный `+` / отмена /
  минус / ретрай на красной строке с сохранённым прогрессом фоном), тап по
  строке открывает только скачанное, папки через `TriStateCheckbox` (состояние
  из диска + очереди, тап докачивает недостающее или сносит поддерево);
  счётчики папок и принадлежность строк очереди считаются одним проходом на
  смену состояния. В шапке вместо круговых стрелок синка отдельное «Обновить
  список», фоновый синк остался тумблером.
- Промпт первой папки (XR-043) перевешан с тапа на `+` и на выбор папки
  (`Pending.Enqueue`/`EnqueueFolder`).

Отличия от исходного плана: ручной «синк сейчас» и с ним весь флоу XR-135
(подтверждение «удалить всё» при пустом выборе) убраны, у `syncOnce` пропал
параметр `deleteAll`: очистка теперь адресная, через минус и снятие папок.
Раскрытие папки при снятии файла замораживает её охват на видимых ветках:
файлы, добавленные на сервере позже, в выбор уже не попадают (осознанная цена
prefix-модели, задокументирована у `expand_deselect`). Общий индикатор очереди
сознательно не делался, это смежная XR-056.

## Проверка

Тапнуть `+` на трёх файлах подряд -> все три встают в очередь и качаются по
очереди с прогрессом; отменить качающийся -> возвращается `+`; скачанный
тапнуть -> открывается; `−` на скачанном -> файл удаляется, иконка -> `+`;
оборвать сеть в процессе -> красная строка + ретрай, тап -> докачивает.

- Отмена во время busy-пробы убивала чужую передачу: флаг activeInFlight взводился до захвата нативного лока, а голова очереди оставалась и докачивалась уже снятой с выбора: исправлено
- Строка узнавала передачу по одному относительному пути: при совпадении путей в двух шарах прогресс и отмена били по чужой шаре: исправлено
- Гонки dequeue/cancelActive: элемент, ставший головой между тапом и корутиной, продолжал качаться уже снятым с выбора, зеркало потом удаляло файл: исправлено
- QueueItem как data class: повторная постановка того же файла удалялась вместе с завершённым близнецом по value-равенству: исправлено
- adoptLocalIntoSelection усыновлял .xrsync-part: partial попадал в выбор, а зачистка сирот сносила резюм-файл: исправлено
- Офлайн-снятие раскрывало папку по локальному манифесту и молча теряло нескачанные ветки выбора: исправлено
- removeFolder удалял поддерево под пишущим нативом: partial воскресал после deleteRecursively: исправлено
- Очередь перекачивала файл, который зеркало успело скачать, и делала полный обход диска после каждого файла: исправлено
- expandDeselect: чистая логика выбора без тестов, риск разъехаться с планировщиком в xr-core: исправлено
- Выбранное-но-не-скачанное стало невидимым (старые галочки, гибель процесса с очередью): зеркало качало бы то, чего строки не показывают: исправлено
- Файлы, пропавшие с сервера, при выключенном синке нечем удалить: строк нет, минуса нет: исправлено
- Ошибка загрузки не давала фидбека вне папки файла, вердикт agent_offline терялся: исправлено
- Тап по скачанной строке при пропавшем с диска файле был немым no-op: исправлено

- Пере-скан манифеста и очереди на каждую строку каждые 500 мс плюс холостой поллинг открытого экрана: исправлено
- Два дублирующих поллера прогресса (launchPolling и построчный) с расходящейся математикой скорости: исправлено
- Первый замер скорости у уже идущей передачи давал всплеск: весь счётчик байт делился на миллисекунды: исправлено
- Busy-ожидание поверх нативного лока вместо очереди в xr-core; отмена одного файла зеркала гасит весь проход: отклонено, нативная очередь и точечная отмена файла зеркала не по скоупу задачи; busy-ожидание с пропуском уже скачанного задокументировано, остаток прохода зеркала возвращается следующим циклом
