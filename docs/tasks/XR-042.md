# XR-042: персистентный журнал логов вместо ленты из транзиентных источников

Логи не должны зависеть от состояния (сеть/режим), это просто журнал записей.

## Проблема

Лента собиралась из транзиентных источников (engine `recentErrors` +
`probeLog`): при смене сети движок перезапускается и recentErrors обнуляется,
после переключения доверенная -> обычная тянутся пробы доверенной. Файловые
события (скачивания, синки, ошибки шар) в ленту не попадали вовсе.

## Что сделано

Единый append-only журнал `xr_core::journal::Journal`: пишет в файл
(`filesDir/journal/journal.log` + ротированные хвосты `.1`, `.2`, ...) и держит
хвост в памяти для вкладки Log. Свойства перенесены из старого ринга `Stats`:
свёртка подряд идущих дубликатов в `(×N)` (и в памяти, и в файле), приоритетный
drain хвоста (ERROR не вытесняются INFO/WARN-шумом). Ротация по размеру файла,
кол-во файлов настраивается; параметры в настройках приложения (вкладка
Servers, секция «Журнал»), меняются на лету.

Все источники пишут в один буфер, каждая запись несёт тег источника:

- `[vpn]` движок (`Stats::add_log/warn/error` делегируют в Journal; JNI
  подключает общий журнал к движку при старте) плюс жизненный цикл сервиса
  (туннель поднят/остановлен, ошибки старта и движка);
- `[net]` смены сети и режима: пауза на доверенной сети, выход из паузы (с
  причиной), перепривязка туннеля к новому аплинку;
- `[probe]` строки пробы ограничений доверенной сети;
- `[files]` файловые события в JNI-слое: скачивание (успех/ошибка), сводка
  синка (получено/удалено/ошибки), недоступный манифест (WARN), перенос
  директории хранения.

Журнал живёт на уровне процесса (`static JOURNAL` в JNI, поднимается в
`XrApp.onCreate` до любых компонентов), на диске переживает перезапуск
приложения. UI читает его напрямую (`nativeJournalTail`, полл раз в секунду на
переднем плане), поэтому лента видна и без запущенного движка; из снапшота
статистики поле `errors` убрано. Экспорт/копирование/шаринг лога отдают полное
содержимое с диска (`nativeJournalDump`), не только видимый хвост.

Тесты: `xr-core/src/journal.rs` (персист через reopen, ротация и dump,
свёртка на диске, drain, очистка, орфаны при уменьшении кол-ва файлов,
оборванная крэшем строка) и `xr-core/src/stats.rs`
(`external_journal_survives_stats_recreation`: регрессия «лента обнуляется при
перезапуске движка»).

## Проверка

1. Подключить VPN, дождаться записей на вкладке Log (`[vpn]` туннель поднят,
   mux relay ...).
2. Переключить сеть Wi-Fi <-> LTE. Ожидается: в ленте запись `[net]` про
   перепривязку, прежние записи НЕ пропали (раньше лента обнулялась).
3. Войти в доверенную сеть (авто-пауза), выйти из неё. Ожидается: записи
   `[net]` про паузу и выход из паузы, строки пробы `[probe]`; после выхода из
   паузы лента продолжает старую, пробы доверенной сети остаются в истории
   ниже, а не «тянутся» поверх нового режима.
4. Убить приложение (свайп из recents), открыть заново. Ожидается: лента на
   вкладке Log на месте (читается с диска).
5. Скачать файл из шары / дёрнуть синк при недоступном агенте. Ожидается:
   записи `[files]` про скачивание и WARN про манифест.
6. Servers -> секция «Журнал»: сменить размер/кол-во файлов; кнопка Clear на
   вкладке Log очищает ленту и файлы.
