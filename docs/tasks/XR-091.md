# XR-091: CLI-обвязка доски задач (taskctl)

## Мотивация

Каждая операция с доской сегодня это ручная хирургия markdown: прочитать
`TASKS.md`, точечно поправить 1-3 места, при закрытии ещё дописать строку в
`TASKS-archive.md` и перенести файл задачи в `tasks/archive/<год>/`. За месяц
доску правили ~110 раз. Самая частая операция (закрытие в архив) заодно и самая
многошаговая, а инварианты доски (сортировка Backlog по R, формат архивной
строки, год в пути архива, не забытый перенос файла) держатся только на
аккуратности исполнителя. Для агента это ещё и лишние токены: каждое чтение
доски стоит ~10 тысяч, пересортировка Backlog переписывает целую секцию.

## Решение

Утилита на Go, stdlib без зависимостей. Написана здесь как `tools/taskctl`,
затем по решению владельца перенесена в общий репозиторий `devkit`
(github.com/dronrider/devkit, клонируется рядом с проектами): доска и
`RANKING.md` общие для всех проектов, инструменту место рядом с ними. Бинарь
ставится в PATH: `cd ../devkit/taskctl && go build -o ~/go/bin/taskctl .`.
Думающая часть
(заголовок, слагаемые ранга по `RANKING.md`, сценарий проверки) остаётся за
исполнителем и передаётся аргументами, утилита делает только механику.

Подкоманды:

- `add --title ... --type bug|task|LLD --rank "а+б+в+г+д" [--link ...] [--status ...] [--id XR-NNN]`
  сумма и бакет P считаются из разбивки, строка встаёт в Backlog сразу в
  правильное место сортировки (R по убыванию, при равенстве ID по возрастанию);
- `move XR-NNN backlog|in-progress|check|blocked [--reason ...]` переносит
  строку между секциями, для Blocked причина обязательна и дописывается к
  заголовку;
- `close XR-NNN [--commit sha[,sha]] [--date ГГГГ-ММ-ДД]` вырезает строку с
  доски, дописывает в архив (колонка R при этом отбрасывается, как заведено в
  архиве), переносит файл задачи в `tasks/archive/<год>/` через `git mv` и
  ставит ссылку уже на новое место;
- `sort` пересортировывает Backlog;
- `lint` проверяет инварианты: разбор всех строк, число колонок, сумма ранга
  сходится с разбивкой, P соответствует бакету, Backlog отсортирован, ID
  уникальны по доске и архиву, локальные markdown-ссылки ведут на существующие
  файлы;
- `id` печатает следующий свободный ID.

Ключевое требование к надёжности: утилита не переформатирует то, чего не
трогала. Нетронутые строки и весь текст вне таблиц сохраняются байт в байт
(строка хранится как есть и пересобирается только при изменении), а строка,
которую не удалось разобрать, валит операцию с внятной ошибкой до какой-либо
записи на диск.

## DoD

- Подкоманды add / move / close / sort / lint / id работают на реальных
  `docs/TASKS.md` и `docs/TASKS-archive.md` без порчи формата.
- Нетронутые части файлов не меняются байт в байт (round-trip тест).
- Юнит-тесты на разбор строки, сортировку, бакеты P, вставку, закрытие с
  переносом файла; интеграционный прогон цикла add -> move -> close на
  временной копии доски.
- `lint` на текущей доске и архиве проходит (или найденные им реальные огрехи
  исправлены).
- Упоминание утилиты в проектном `CLAUDE.md`, короткий README с примерами.

## Сценарий проверки (Check)

1. `cd ~/projects/devkit/taskctl && go test ./...` - все тесты зелёные.
2. `taskctl lint` из корня xr-proxy - доска и архив проходят без ошибок.
3. Прогнать цикл на живых файлах: `taskctl add --title "тестовая задача"
   --type task --rank "0+1+1+0+1" --link "x"`, убедиться по `git diff`, что
   строка встала в Backlog по сортировке и больше ничего не изменилось; затем
   `taskctl move` её в in-progress, `taskctl close` без коммита, убедиться,
   что строка ушла в архив с сегодняшней датой; откатить (`git checkout docs/`).
4. Сама задача XR-091 переведена в Check и потом закрыта уже утилитой, diff
   этих операций читается как обычная ручная правка доски.
