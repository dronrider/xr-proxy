# LLD-13. Автоустановка сервера и роутера (zero-touch provisioning)

**Статус:** Draft
**Область:** новый бинарь/скрипт `xr-setup` (идемпотентная установка
xr-server+xr-hub на свежий VPS и xr-client на свежий OpenWRT-роутер),
`xr-android` (обёртка по SSH: ввод доступа, запуск установщика, приём инвайта),
`xr-hub` (выдача инвайта по завершении — переиспользуем LLD-01/04).
**Зависимости:** [LLD-01](01-control-plane.md) — xr-hub, генерация инвайтов,
ed25519-ключи, формат пресетов. [LLD-04](04-onboarding-qr-uri.md) — приём инвайта
(`nativeApplyInvite`, TOFU): автоустановка **заканчивается** выдачей такого инвайта,
дальше работает существующий онбординг. [LLD-08](08-android-multi-server.md) —
`ServerProfile`: свежеподнятый сервер добавляется как профиль.
[LLD-10](10-client-multi-vps-failover.md) — поднятый сервер попадает в пул как
backup/primary; конфиг роутера генерируется в формате `[[servers]]`.
[LLD-17](17-hub-router-registry.md): после установки роутера установщик
**регистрирует** его на хабе одноразовым enrollment-токеном и пишет `[control]`
(per-router ключ) в конфиг: это шов «установка → реестр и удалённое управление».
**Связанные документы:** `local-docs/routers.md` (реальный путь деплоя, доступы),
memory [[reference_deploy_handoff]] (SSH, cross-build, ловушки nftables/sysctl),
[ARCHITECTURE.md §3](../ARCHITECTURE.md).

Поднять новый xr-server или настроить новый OpenWRT-роутер сейчас — это десятки
ручных шагов по SSH (`reference_deploy_handoff`, `routers.md §3-4`): собрать бинарь,
разложить файлы, написать конфиг, ключи, systemd/procd, sysctl, nftables. Этот LLD
сводит обе цели к **идемпотентному установщику**, запускаемому из приложения по
SSH: ввёл host + доступ — через минуту получил готовый сервер (с инвайтом для
онбординга) или настроенный роутер. Цель — «минимум ручной настройки» (история 4
про VPS, история 11 про роутер родителям за серым IP — всё в LAN).

---

## 1. Текущее состояние

- **VPS.** Развёртывание вручную: сборка `xr-server`/`xr-hub` (часто прямо на VPS
  из-за glibc — `routers.md §3.2`), раскладка бинарей, `server.toml` с ключом/salt,
  systemd-юниты, sysctl (bbr/fq/буферы), Let's Encrypt для хаба. Десятки шагов,
  легко ошибиться, не идемпотентно.
- **Роутер.** Cross-build `xr-client` (musl/aarch64, только на Mac с docker —
  `routers.md §7.5`), копирование бинаря, `config.toml`, procd init, watchdog-cron,
  nftables (TPROXY, QUIC-block), fd-limit, dnsmasq на Quad9. Всё руками по SSH-8822.
- **Инвайты.** LLD-01/04 уже умеют сгенерировать одноразовый инвайт и принять его в
  приложении (QR/URI, TOFU). Но **создать сам сервер**, который этот инвайт
  раздаёт, по-прежнему нужно руками.
- Итог: «новый сервер за минуту» и «новый роутер родителям» упираются не в
  онбординг (он готов), а в **установку самой инфраструктуры**.

## 2. Целевое поведение

### 2.1 Этап 1: идемпотентный установщик (ядро MVP)

Единый артефакт `xr-setup`, запускаемый на целевой машине (VPS или роутер),
с подкомандой по цели:

```
xr-setup server --hub-domain xr-hub.example.com [--with-hub] [--key <b64>]
xr-setup router --server <addr:port> --key <b64> [--preset russia]
```

Свойства:

- **Идемпотентность.** Повторный запуск приводит систему в то же целевое состояние
  без поломки уже настроенного (проверяет наличие/версию бинаря, не перетирает
  существующий ключ без `--force`, доустанавливает недостающее). Можно гонять
  многократно.
- **Самодостаточность.** Не требует тулчейна на цели: установщик несёт/качает
  готовый бинарь под нужную арку (x86_64-musl для VPS, aarch64-musl для роутера),
  кладёт конфиг, ставит сервис-менеджер (systemd на VPS, procd на OpenWRT),
  применяет sysctl/nftables/fd-limit/watchdog (для роутера — из
  `reference_deploy_handoff`/`routers.md`).
- **`server`-цель:** генерирует ключ/salt (или принимает `--key`), пишет
  `server.toml`, ставит `xr-proxy-server.service`, sysctl (bbr/fq/буферы), при
  `--with-hub` — `xr-hub` + конфиг + (опц.) TLS. По завершении **печатает
  одноразовый инвайт** (URI/QR) через хаб (LLD-01) → его принимает приложение
  (LLD-04). Это «шов» между установкой и онбордингом.
- **`router`-цель:** кладёт `xr-client` + `config.toml` (из переданного
  сервера/ключа/пресета, в формате `[[servers]]` LLD-10), procd init + watchdog,
  nftables (auto_redirect/QUIC-block — xr-client ставит сам), fd-limit, переключает
  dnsmasq на Quad9 (`routers.md §4.2`), `enable`+`start`.
  - **Раздаваемый Wi-Fi SSID** (`--ssid <name> [--wifi-pass <psk>]`): задаёт имя
    домашней сети, которую роутер вещает (`uci set wireless...; commit`). Цель
    пользователя — «роутер начал раздавать интернет с обходом», для чего и нужно
    имя сети. **Применяется отложенно/последним шагом** (`uci commit; wifi reload`
    в фоне с задержкой), иначе смена SSID оборвёт ту самую Wi-Fi-сессию, через
    которую приложение настраивает роутер (§5.9). Опционально (`--ssid` не задан —
    Wi-Fi не трогаем).
  - **Enrollment в хаб (шов с LLD-17):** при наличии `--enroll-token` установщик
    после старта клиента регистрирует роутер на хабе (`POST /api/v1/enroll`),
    получает `{router_id, secret, command_pubkey}` и пишет секцию `[control]` в
    `config.toml` → роутер появляется в реестре хаба и открывает исходящий
    control-канал. Без токена — роутер работает автономно (как сегодня), без
    удалённого управления.

### 2.2 Этап 2 — обёртка в Android по SSH

Приложение даёт UI «Добавить сервер с нуля» / «Настроить роутер»:

1. Ввод: `host`, порт SSH, креды (root-пароль **или** приватный ключ). Для роутера
   — это локальный адрес в его LAN (работает за серым IP, всё в Wi-Fi роутера,
   история 11).
2. Приложение по SSH:
   - проверяет доступность/арку цели;
   - доставляет `xr-setup` (scp/потоком) **или** запускает `curl|sh`-обёртку,
     тянущую установщик с хаба;
   - запускает нужную подкоманду, стримит прогресс в UI (шаги: connect → deliver →
     install → configure → start → verify).
3. По завершении:
   - **server:** приложение получает инвайт (его выдал установщик) -> прогоняет
     существующий `applyInvite` (LLD-04) → добавляет `ServerProfile` (LLD-08),
     делает активным; новый VPS попадает в пул (LLD-10).
   - **router:** приложение показывает «роутер настроен на сервер X», проверяет,
     что трафик за роутером идёт через прокси (опц. health-пинг).
4. Креды **не сохраняются** — живут только в памяти на время SSH-сессии,
   стираются по завершении. Конечный артефакт доверия — выданный инвайт / факт
   настройки, не пароль.

### 2.3 Что НЕ хранится и не утекает

- Root-пароль/ключ цели — только in-memory на время сессии, не в prefs, не в логах.
- Базовый obfuscation-ключ генерируется на сервере (или задаётся владельцем) и
  доезжает до приложения **через инвайт** (подписанный канал LLD-04), а не вводится
  руками в SSH-обёртке.

## 3. Дизайн-решения

### 3.1 Делаем оба этапа, но MVP = этап 1 (движок установки)

Открытый вопрос problems.md: оба этапа или только этап 1 как MVP. **Решение:**
проектируем оба, **реализуем этап 1 первым и отдельно проверяемым** (запускается
вручную по SSH владельцем — уже огромная экономия против десятков ручных шагов),
этап 2 (Android SSH-обёртка) идёт поверх готового идемпотентного установщика. Если
время поджимает — этап 1 самоценен и закрывает «сервер за минуту» для владельца с
SSH; этап 2 добавляет «из приложения». Разбивка снижает риск: вся хрупкая
системная логика живёт в тестируемом установщике, Android лишь его драйвер.

Сам этап 1 реализуется двумя инкрементами: сначала каркас шагов и
server-профиль (самоценен, заканчивается выданным инвайтом, проверка по п. 1
плана в разделе 6), затем router-профиль поверх готового каркаса (п. 2 там
же). У них разные цепочки зависимостей (server упирается в LLD-01, router
тянет enrollment из LLD-17) и разные стенды проверки, поэтому и на доске это
две задачи. Чтобы каркас не оказался заточен под сервер, цель `router` в CLI
и enum профилей закладывается с первого дня.

### 3.2 Общий движок установки, две цели, а не две кодовые базы

Открытый вопрос: роутер и VPS — общий движок или две ветки. **Решение:** один
бинарь `xr-setup` с общим каркасом «список идемпотентных шагов» и двумя
**профилями** (server / router), различающимися набором шагов (сервис-менеджер,
конфиг, sysctl/nftables). Общее: доставка бинаря, запись конфига, idempotency-
проверки, verify. Это избегает дублирования и держит ловушки
(`reference_deploy_handoff`: musl-константы, nftables `ip`-family, procd respawn)
в одном месте.

### 3.3 Установщик как отдельный артефакт, а не `curl|sh`-только

`curl | sh` удобен для ручного запуска, но непрозрачен и плохо тестируется.
**Решение:** ядро это компилируемый `xr-setup` (можно как подкоманда
существующего бинаря или отдельный маленький крейт), с тонкой `curl|sh`-обёрткой
поверх для ручного сценария (она просто качает и запускает установщик). Так логика
идемпотентна и покрыта тестами на чистых функциях (рендер конфига, парс арки,
планирование шагов), а не размазана по shell.

### 3.4 SSH из Android — проверенной библиотекой, не самописом

Android-обёртка использует зрелую SSH-библиотеку (sshj/JSch-класс) для exec/scp.
Не пишем свой SSH. Поддержка пароля и ключа. Стримим stdout шагов в UI. Хост-ключ
цели — TOFU-приёмка с показом fingerprint'а (как у любого первого SSH), чтобы не
ловить MITM в чужой сети.

### 3.5 Завершение через инвайт, а не прямой импорт конфига

Установка сервера заканчивается выдачей **инвайта** (LLD-01/04), а не отдаёт
приложению голый конфиг по SSH. Так переиспользуется весь готовый, подписанный,
TOFU-защищённый онбординг-путь, и SSH-канал не становится доверенным источником
секретов подключения. Один шов (инвайт) вместо второго пути импорта.

## 5. Риски и edge-кейсы

1. **Не-идемпотентный повтор затирает рабочий сервер.** Каждый шаг — `check`
   перед `apply`; ключ/salt не перегенерируются при наличии (без `--force`);
   конфиг мёржится, а не перезаписывается вслепую. Тест `test_step_idempotent`.
2. **Креды в логах/памяти.** Пароль/ключ не пишем в лог (маскируем), не сохраняем
   в prefs, стираем буфер после сессии. Установщик не эхоит секреты.
3. **MITM в чужой Wi-Fi при настройке роутера.** Host-key TOFU с показом
   fingerprint; для роутера это его LAN (история 11), риск ниже, но приёмку ключа
   показываем явно.
4. **Несовпадение арки/ОС цели.** `detect_arch` + ранний fail с понятным
   сообщением, если арка не поддержана; не доставляем несовместимый бинарь.
5. **Полу-настроенная цель при обрыве SSH.** Идемпотентность спасает: повторный
   запуск доводит до целевого состояния. UI предлагает «повторить».
6. **OpenWRT-специфика** (`reference_deploy_handoff`/`routers.md`): nftables
   `ip`-family (не `inet`), procd `respawn 3600 15 0`, fd-limit `nofile 16384`,
   dnsmasq на Quad9, QUIC-block — всё в router-профиле; легко забыть шаг → verify
   проверяет (процесс жив, nft-таблица есть, DNS резолвит не-127.0.0.1).
7. **TLS для хаба при `--with-hub`.** Let's Encrypt требует домен/80-порт;
   если недоступно, то установщик ставит хаб на self-signed/HTTP с предупреждением,
   полноценный TLS — ручной follow-up (не блокирует сервер-часть).
8. **Серый IP VPS / firewall.** Установка сервера идёт по SSH к публичному VPS; если
   порты 8443/443 закрыты провайдером — verify это поймает, сообщит.
9. **Смена SSID обрывает сессию настройки.** Приложение настраивает роутер по его
   же Wi-Fi; `wifi reload` с новым SSID рвёт это соединение на полпути. Поэтому
   SSID применяется **последним**, отложенно (`commit` + фоновый `wifi reload`
   через задержку), уже после `enable`/`start`/enrollment; verify-пинг — до смены
   SSID. UI предупреждает «сеть переименуется, переподключитесь к <name>».

## 6. План проверки

1. **Этап 1, server (ручной).** На чистом VPS: `xr-setup server --with-hub`
   → за <2 мин поднят xr-server+xr-hub, выдан инвайт. Повторный запуск —
   идемпотентен (ничего не сломал, ключ тот же).
2. **Этап 1, router (ручной).** На свежем OpenWRT: `xr-setup router --server
   <aeza> --ssid HomeNet --enroll-token <t>` → xr-client стоит, procd enabled,
   nft-таблица `xr_proxy` есть, QUIC-block есть, dnsmasq на Quad9, трафик за
   роутером проксируется (curl geo-blocked → 200); роутер **вещает SSID `HomeNet`**
   (применён последним, сессия не оборвалась раньше времени) и **виден в реестре
   хаба** `online` (LLD-17).
3. **Этап 2, server из приложения.** «Добавить сервер с нуля» → ввод host/пароль
   → степпер до verify → инвайт принят → `ServerProfile` добавлен и активен →
   Connect работает. Креды не в prefs (проверить).
4. **Этап 2, router из приложения** (за роутером по Wi-Fi, серый IP): «Настроить
   роутер» прогоняет установщик, verify-пинг показывает проксирование.
5. **Идемпотентность из приложения.** Повторный прогон на уже настроенной цели —
   «уже настроено», ничего не ломается.
6. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Создание самого VPS** (заказ у хостера через API). Установщик работает с уже
  существующей машиной с SSH. Авто-провижн облака — отдельная большая фича.
- **Полная автоматизация TLS/домена для хаба** — base-случай делаем, edge (DNS,
  CAA, проброс 80) — ручной follow-up (§5.7).
- **Обновление уже поднятых компонентов** (in-place upgrade сервера/клиента) —
  установщик ставит/доводит; rolling-upgrade флота это отдельный LLD.
- **Хранение списка серверов на хабе** (раздача флоту) — упомянуто в LLD-10 §3.2
  как follow-up; автоустановка лишь *создаёт* сервер и отдаёт инвайт.
- **Автоустановка для iOS/desktop**. Этот LLD про Android-обёртку + универсальный
  установщик; до других платформ дойдём позже.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Делать оба этапа или только этап 1?* → **оба**, но MVP и первая имплементация
  это этап 1 (идемпотентный установщик), этап 2 поверх него (раздел 3.1).
- *`curl|sh` или бинарь?* -> **бинарь `xr-setup`** (тестируемый, идемпотентный)
  + тонкая `curl|sh`-обёртка для ручного запуска (§3.3).
