# LLD-20. Удалённое управление роутерами (подписанные команды поверх реестра)

**Статус:** Draft
**Область:** `xr-hub` (очередь команд, подпись, выдача команд в ответе poll,
действия и история команд в разделе «Роутеры»), `xr-client` (приём, верификация
и применение команд в control-цикле), `xr-proto` (типы команды и результата,
подпись/верификация).
**Зависимости:** [LLD-17](17-hub-router-registry.md) это база: реестр роутеров,
идентичность, per-router секрет и **исходящий poll** (роутер -> хаб). Здесь поверх
него добавляются команды. [LLD-01](01-control-plane.md): xr-hub, ed25519, Admin
SPA, JSON-стор. [LLD-16](16-manual-server-hub-rules.md): назначение пресета
(команда `apply_preset`). [LLD-10](10-client-multi-vps-failover.md): поля
`[[servers]]` для `update_config`. [LLD-12](12-android-apk-self-update.md):
отделённый офлайн-ключ подписи как образец (компрометация VPS не даёт права
подписывать команды).
**Связанные документы:** [ARCHITECTURE.md §3 (хаб)](../ARCHITECTURE.md),
memory [[reference_deploy_handoff]].

Реестр (LLD-17) даёт хабу видеть парк роутеров и их статус, но не **управлять**
ими. Этот LLD добавляет управление: хаб ставит роутеру **подписанные команды**
(закрытый набор, не произвольный shell), роутер забирает их тем же исходящим
poll'ом, что и для статуса (LLD-17 §2.2), проверяет подпись закреплённым ключом и
применяет, результат репортит обратно. Это самая весомая по безопасности фича
экосистемы (хаб меняет конфиг на роутерах в чужих квартирах), поэтому центральное
здесь это модель угроз и её ограничение.

---

## 1. Предпосылка (из LLD-17)

- Реестр, идентичность роутера, per-router секрет и исходящий poll-канал уже есть
  (LLD-17). Inbound к роутеру невозможен (серый IP за NAT), поэтому всё идёт через
  тот же исходящий poll.
- Сейчас poll несёт только статус-отчёт и обновляет `last_seen`. Здесь его **ответ**
  расширяется списком pending-команд, а тело отчёта это результаты их выполнения.

## 2. Целевое поведение

### 2.1 Команды (least-privilege, не произвольный shell)

**Закрытый enum** типов команд, не «выполни shell»:

| Команда | Действие |
|---|---|
| `apply_preset { name }` | назначить/сменить пресет (роутер тянет его, LLD-01/16) |
| `update_config { patch }` | мёрж в **белый список** полей: `block_quic`, `mux_pool_size`, `on_server_down`, `log_level`, набор `[[servers]]` (LLD-10) |
| `reload` / `restart` | перечитать конфиг / перезапустить процесс (через procd) |
| `deregister` | роутер стирает `[control]`, перестаёт поллить |

Каждая команда **подписана** ed25519-ключом хаба, который роутер проверяет
закреплённым ключом (TOFU при enrollment, LLD-17 §2.1). Команда несёт `id` и
идемпотентна; результат роутер репортит в следующем poll (`acked`/`failed`).
`update_client` (удалённое обновление бинаря, аналог APK-self-update LLD-12 для
роутера) это **вне скоупа MVP** (§7), но enum и канал спроектированы под него.

### 2.2 Доставка команд через poll (расширение LLD-17 §2.2)

В ответе на `POST /api/v1/router/poll` хаб дополнительно отдаёт список
pending-команд и ack'и уже выполненного. Роутер их применяет, а результаты кладёт
в `command_results` следующего отчёта (поле `RouterReport`, LLD-17 §2.4):

```
command_results: Vec<{ id, ok, detail }>
```

Pull-природа канала задаёт цену: задержка применения команды <= `control_poll_interval`
(default 30с). Realtime (websocket/long-poll) вне скоупа MVP (§7).

### 2.3 Действия в админке (карточка роутера, LLD-17 §2.5)

Раздел «Роутеры» (список и статус роутера) живёт в LLD-17. Управление добавляет в
**карточку роутера**:

- историю команд (pending / acked / failed);
- кнопки действий (apply preset, reload, restart), каждая ставит **подписанную**
  команду в очередь;
- аудит-лог команд на хабе (кто, что, когда поставил и когда acked).

### 2.4 Безопасность (центральный раздел)

Хаб получает право менять конфиг на роутерах в чужих квартирах, и это требует
самой строгой защиты в экосистеме:

- **Подписанные команды** ed25519, закреплённый ключ на роутере (TOFU при
  enrollment, как public-key LLD-04). Тот же принцип, что отделённый release-ключ
  в LLD-12: **компрометация транспорта/VPS не равна RCE на роутерах** без
  приватного ключа подписи команд.
- **Least-privilege:** закрытый enum (§2.1), нет произвольного shell или доступа к
  файлам, `update_config` только по белому списку полей.
- **Аудит-лог** команд на хабе.
- Per-router секрет и его отзыв, rate-limit poll, валидация `router_id` это уже
  реестр (LLD-17 §2.6); здесь они переиспользуются.

## 3. Дизайн-решения

### 3.1 Команды подписаны; ключ подписи в идеале офлайн

MVP: подписываем команды **пресет-ключом LLD-01** (он уже закреплён у клиента),
меньше движущихся частей. Усиление: **отдельный офлайн command-ключ** (как
release-ключ LLD-12), чтобы взлом VPS не давал подписывать команды, follow-up
(§7). Trade-off зафиксирован осознанно.

### 3.2 Закрытый enum команд, не shell

Минимизируем поверхность RCE: хаб может только то, что явно перечислено и покрыто
белым списком, а не «выполни произвольную команду».

## 4. Изменения в коде

| Файл | Что меняется |
|---|---|
| `xr-proto/src/...` | Типы `Command { id, kind, payload, signature }` и результат команды; сериализация (чистые функции); подпись/верификация команды (переиспользует ed25519 LLD-01). |
| `xr-hub/src/api/...` | Очередь команд в ответе `POST /api/v1/router/poll`; Admin `POST /admin/routers/:id/commands` (подпись + очередь); аудит-лог команд. |
| `xr-hub/admin-ui/` | В карточке роутера: история команд + кнопки действий. |
| `xr-client/src/...` | В control-цикле: приём команд из ответа poll, **верификация подписи** закреплённым ключом, применение (`apply_preset` через `PresetCache`; `update_config` мёрж по белому списку; `reload`/`restart` через procd), ack в следующем poll. |
| `docs/ARCHITECTURE.md` §3 | Control-канал команд поверх реестра. |

Тесты (Rust, чистые функции, без реальной машины/сети):

- `test_command_sign_verify` (валидная подпись принимается; чужой ключ/подмена -> reject).
- `test_command_config_whitelist` (`update_config` с полем вне белого списка -> reject).
- Android/админка: без автотестов ([[feedback_android_tests]]), ручной план §6.

## 5. Риски и edge-кейсы

1. **Компрометация хаба -> RCE на домашних роутерах.** Главный риск. Митигация:
   подписанные команды (закреплённый ключ §2.4), закрытый enum, least-privilege;
   цель это офлайн command-ключ (§3.1, follow-up).
2. **Replay команд.** `id` + ack только после успеха + идемпотентный apply; опц.
   nonce/timestamp в подписи.
3. **Задержка команд из-за pull.** Команда применяется <= `control_poll_interval`.
   Для «срочно выключить обходы» это медленно, принимаем как лимит pull-MVP;
   `on_server_down=direct` и так fail-open ([[project_failopen_breaker]]).
4. **Полу-применённая команда (обрыв питания/SSH).** Идемпотентность + ack после
   успеха; повторный poll доводит. Конфиг мёржится, не перезатирается вслепую.
5. **Рассинхрон закреплённого command-ключа при ротации.** Ротация ключа подписи
   требует повторного pin на роутерах (через подписанную старым ключом команду или
   re-enroll); процедуру при ротации описать.

## 6. План проверки

Rust (подпись/whitelist) автотестами; админка и клиент-апплай вручную.

1. **Команда.** `apply_preset` из админки -> роутер применяет <= poll-interval,
   `acked` в карточке; трафик идёт по новому пресету.
2. **Подпись.** Подменить тело команды / подписать чужим ключом -> роутер
   отвергает, пишет в лог; конфиг не меняется.
3. **Least-privilege.** `update_config` с полем вне белого списка -> reject.
4. **Идемпотентность.** Повторная доставка той же `id` -> применяется один раз.
5. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Realtime канал** (websocket/long-poll): pull достаточно для MVP, follow-up.
- **`update_client`** (удалённое обновление бинаря xr-client, аналог LLD-12 для
  роутера): enum/канал готовы под него, сама раскатка бинаря по арке отдельно.
- **Отдельный офлайн command-ключ**: усиление поверх MVP (§3.1); сначала
  переиспользуем пресет-ключ LLD-01.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Произвольный shell или закрытый enum?* -> **закрытый enum + белый список полей**
  (§3.2, §2.1).
- *Чем подписывать команды?* -> **пресет-ключом LLD-01 в MVP**, офлайн
  command-ключ как follow-up (§3.1).
