# LLD-24: локальный индекс хэшей для синка шары (XR-098)

**Статус:** Implemented (в main с 2026-07-11, проверка владельцем пройдена, XR-098 в архиве)
**Область:** `xr-core` (`sync.rs`), `xr-android-jni` (`nativeSyncShare`), `xr-android` (`NativeBridge`, `ShareRepository`)
**Зависимости:**
- [LLD-19](19-file-sharing-agent.md) задаёт движок синка (`plan_sync` + `scan_local_dir`) и агентский `HashCache`, образец для этого индекса.
- XR-043 (перенос директории хранения) диктует относительные ключи: rename сохраняет mtime, но меняет корень.
- XR-097 (пустые `sha256` холодного кэша агента) определяет, какие скачанные файлы можно класть в индекс сразу.

**Связанные документы:** [tasks/archive/2026/XR-098.md](../tasks/archive/2026/XR-098.md) (разбор корня и сценарий проверки), [xr-share/src/manifest.rs](../../xr-share/src/manifest.rs) (агентский `HashCache`).

---

## 1. Зачем

`sync_share_selected` перед диффом зовёт `scan_local_dir`, а тот на каждый синк заново считает SHA-256 каждого локального файла шары: однопоточно, чтение по 64 КБ, на выбранной пользователем папке Android ещё и через FUSE-слой. На шаре в гигабайты «Подготовка…» растягивается на десятки секунд, и тот же полный пересчёт регулярно гоняет фоновый WorkManager-синк, впустую тратя батарею. У агента эта же проблема уже решена кэшем `(size, mtime) -> sha256`; потребителю нужен такой же, но переживающий перезапуск процесса.

## 2. Решение: персистентный `HashIndex`

Локальный индекс `относительный путь -> (size, mtime, sha256)` в `xr-core/src/sync.rs`, рядом со `scan_local_dir`.

- **Ключи относительные** (forward-slash, как в манифесте). Миграция XR-043 переносит файлы rename'ом: mtime сохраняется, корень меняется, поэтому абсолютные ключи протухли бы всей пачкой, а относительные переживают перенос без пересчёта.
- **Формат на диске** это JSON `{"version":1,"entries":{"rel/path":{"size":..,"mtime":..,"sha256":".."}}}`.
- **Загрузка не доверяет файлу**: отсутствующий, битый или чужой файл, как и незнакомая `version`, дают пустой индекс и полный пересчёт. Ошибок наружу нет.
- **Запись атомарная**: временный файл рядом плюс rename. Ошибка сохранения синк не валит, индекс это ускоритель, а не источник правды.
- **Скан с индексом** (`scan_local_dir_indexed`): тот же обход, что у `scan_dir`, но файл перехэшируется только когда его `(size, mtime)` разошлись с индексом, иначе хэш берётся из записи. По ходу обхода строится новая карта, поэтому записи исчезнувших файлов отпадают сами. Тёплый скан сводится к stat-обходу.

## 3. Поток синка

`sync_share_selected` получает новый параметр `index_path: Option<&Path>` (пустая строка из JNI превращается в `None`, старое поведение):

1. загрузить индекс, отсканировать через `scan_local_dir_indexed`;
2. **сохранить сразу после скана**: прогретый индекс переживает упавший или отменённый трансфер, а dry_run прогревает кэш для следующего боевого синка;
3. дифф и `apply_plan` как раньше;
4. после применения дополнить индекс: для каждого `report.fetched` с непустым `entry.sha256` хэш уже верифицирован при скачивании, достаточно stat финального файла; записи с пустым хэшем (холодный кэш агента, XR-097) не пишутся, их один раз дохэширует следующий скан; пути из `report.deleted` удаляются; сохранить.

Файл индекса живёт в приватном `filesDir` приложения (`share-index/<shareId>.json`), а не в директории шары: директорию шары обходят `localPaths`/`localManifest` для UI, true-mirror-delete снёс бы посторонний файл, а в выбранной пользователем папке файл был бы виден и читался через FUSE. Ключ по `shareId` вместе с относительными путями внутри означает, что смена директории хранения индекс не инвалидирует.

## 4. Компромиссы и границы

- **Подмена файла с теми же (size, mtime) не ловится**, ровно как у агентского `HashCache`. Осознанный компромисс из постановки задачи.
- **Разовое скачивание** (`nativeDownloadFile`) индекс не трогает: такой файл один раз хэшируется на следующем скане и оседает в индексе.
- **Параллельные сканы** (фоновый воркер и ручной тап до захвата transfer-лока) могут одновременно сохранить индекс; атомарный rename гарантирует целый файл, побеждает последняя запись, содержимое в обоих случаях корректно.
- Окно между rename скачанного файла и его stat теоретически может зафиксировать чужой mtime; это подмножество компромисса выше.

## 6. Тесты

1. Регрессионный: после тёплого скана содержимое файла подменяется при том же размере, mtime возвращается через `File::set_modified`; повторный скан обязан вернуть старый хэш из индекса, то есть файл не перечитывался. Без индексного lookup тест красный.
2. Персистентность: скан -> save -> load новым экземпляром -> тёплый скан; битый JSON и чужая `version` дают пустой индекс и полный пересчёт без ошибок.
3. Prune: после удаления локального файла его запись пропадает из сохранённого индекса.
4. Переезд корня: rename директории A -> B, скан B с тем же индексом берёт хэши из кэша (относительные ключи).
5. Интеграционный: канированный HTTP-сервер (манифест + файл), `sync_share_selected` с `index_path`; после синка индекс содержит хэш скачанного файла, повторный синк даёт пустой план.

## 7. Проверка владельцем

Сценарий в [tasks/archive/2026/XR-098.md](../tasks/archive/2026/XR-098.md): тёплый повторный синк держит «Подготовку» меньше секунды-двух; изменение одного файла на сервере перекачивает только его; первый синк после обновления приложения допустимо долгий, второй уже быстрый.

## 8. Вне скоупа

- `xr-share pull` (локальный скан не делает) и SAF-путь `nativePlanSync` (локальное состояние поставляет Kotlin).
- Параллельное хэширование холодного скана: первый прогон остаётся однопоточным, индекс убирает все последующие.
- Общий кэш между шарами: индекс на шару, по файлу на `shareId`.
