# LLD-12 — Самообновление Android-APK

**Статус:** Implemented (2026-06-18; `xr-core/update.rs` +6 тестов, эндпоинты
`/api/v1/app/{latest,download/:ver}`, CLI `xr-hub sign-release`/`gen-release-key`,
`UpdateManager.kt` + UI-баннер + JNI `nativeCheckUpdate`/`nativeVerifyApk`. Rust
зелёный, 0 warnings; Android-сборка/device-verify §6 — за владельцем на Mac.)
**Область:** `xr-hub` (раздача APK + подписанного манифеста версии, отдельный
release-ключ), `xr-core` (загрузка/проверка манифеста и APK — переиспользуем
reqwest-клиент онбординга), `xr-android` (UI обновления, `PackageInstaller`,
`REQUEST_INSTALL_PACKAGES`), `xr-android-jni` (проброс проверки манифеста).
**Зависимости:** [LLD-01](01-control-plane.md) — xr-hub, ed25519-подпись пресетов,
HTTPS (axum-server), `rust-embed` / статика. [LLD-04](04-onboarding-qr-uri.md) —
`onboarding.rs` reqwest-клиент + TOFU public-key (модель «доверенный ключ хаба»,
которую тут **сознательно не переиспользуем** для релиза, см. §3.1).
**Связанные документы:** [ARCHITECTURE.md §5.3 (control plane)](../ARCHITECTURE.md),
problems.md задача 7.

Обновление приложения через `adb install` неудобно — каждый раз нужен MacBook,
кабель и ручная сборка ([[feedback_android_build]]). Этот LLD позволяет xr-hub
раздавать APK + подписанный манифест версии, а приложению — проверять подпись и
предлагать установку. **Критично по безопасности:** компрометация VPS не должна
превращаться в RCE на телефоне. Поэтому ключ подписи **релиза ≠ ключ сервера/хаба**,
и публичный release-ключ **зашит в приложение** на этапе сборки.

---

## 1. Текущее состояние

- xr-hub раздаёт пресеты и инвайты по HTTPS (`/api/v1/...`), умеет ed25519-подпись
  пресетов **серверным** ключом (LLD-01). Этот ключ живёт на VPS — компрометация
  VPS = доступ к нему.
- Приложение TOFU'ит **серверный** публичный ключ при онбординге
  (`trusted_public_key`, LLD-04) — для доверия пресетам, не для кода.
- Версия приложения известна как `BuildConfig.VERSION_NAME/CODE`; механизма
  «узнать о новой версии и поставить» нет — только `adb install -r`.
- Статика хаба встроена в бинарь (`rust-embed`); раздачи бинарных артефактов
  (APK) сейчас нет.

## 2. Целевое поведение

### 2.1 Манифест версии (подписанный)

xr-hub отдаёт `GET /api/v1/app/latest` → JSON-манифест:

```json
{
  "version_code": 12,
  "version_name": "0.2.0",
  "min_sdk": 29,
  "apk_url": "https://xr-hub.zoobr.top/api/v1/app/download/0.2.0",
  "apk_sha256": "<hex>",
  "size_bytes": 7340032,
  "release_notes": "Multi-VPS failover, панель здоровья",
  "released_at": "2026-06-20"
}
```

Подпись — **отдельным** detached-полем: эндпоинт отдаёт `{ "manifest": <канонические
байты>, "signature": "<base64 ed25519>" }`, либо манифест + заголовок
`X-Signature`. Подпись считается **release-ключом** над каноническими байтами
манифеста (включая `apk_sha256`). APK отдаётся отдельным `GET /api/v1/app/download/:ver`
(application/vnd.android.package-archive).

### 2.2 Цепочка доверия

1. Release-ключ (ed25519) — **приватная половина у владельца офлайн** (ноут/
   менеджер ключей), не на VPS. Публичная — **зашита в приложение** на этапе
   сборки (`BuildConfig.RELEASE_PUBLIC_KEY` из gradle/CI, не из сети).
2. Приложение скачивает манифест → проверяет подпись **зашитым** release-ключом.
   Подпись не сходится → **отказ**, обновление не предлагается, WARN в лог.
3. Скачивает APK → считает SHA-256 → сверяет с `apk_sha256` из (уже проверенного
   подписью) манифеста. Не сходится → отказ, файл удаляется.
4. Только при обоих успехах — предлагает установку.

Компрометация VPS позволяет подменить APK и манифест, но **не подделать подпись**
(приватного ключа на VPS нет) → приложение отвергнет. Это и есть «VPS-compromise
≠ RCE».

### 2.3 UX обновления

- Фоновая проверка `GET /api/v1/app/latest` раз в `update_check_interval` (default
  24ч) и по ручному «Проверить обновления» в Servers/Settings.
- `version_code` манифеста > текущего → ненавязчивый баннер/диалог: версия,
  release notes, размер, кнопки «Обновить» / «Позже».
- «Обновить» → скачивание APK (прогресс) → проверка SHA → системный установщик
  через `PackageInstaller` (или `ACTION_VIEW`/`FileProvider` как fallback).
  Требуется `REQUEST_INSTALL_PACKAGES`; если не выдано — ведём пользователя в
  системный экран «разрешить установку из этого источника».
- Установка того же `signingCert` (одинаковая подпись APK как Android-пакета) →
  обновление поверх без удаления данных. Разные подписи → Android откажет;
  документируем, что релизы должны подписываться одним keystore.

### 2.4 Откат и отказоустойчивость

- Плохая подпись манифеста / несошедшийся SHA → обновление **не предлагается**,
  старая версия продолжает работать. Никаких частичных установок.
- Прерванная загрузка → temp-файл в `cacheDir`, докачка не обязательна, просто
  повтор; недокачанный файл удаляется.
- Хаб недоступен → тихо пропускаем проверку, ретрай по интервалу.

## 3. Дизайн-решения

### 3.1 Release-ключ отдельный и зашитый, не TOFU и не серверный

TOFU-ключ из LLD-04 доверяет тому, что отдал хаб при первом контакте — для кода
это недопустимо: подменивший VPS при первом онбординге подсунул бы свой ключ.
Release-ключ **зашивается в сборку** (pin), приватная половина живёт офлайн у
владельца. Серверный ключ (подпись пресетов) тоже не годится — он на VPS. Три
разных ключа с разными границами доверия:

| Ключ | Приватная половина | Подписывает | Доверие у клиента |
|---|---|---|---|
| Серверный (LLD-01) | на VPS | пресеты | TOFU при онбординге |
| Release (этот LLD) | офлайн у владельца | манифест APK | pinned в сборке |
| APK signing (Android) | офлайн keystore | сам .apk-пакет | системой при install |

### 3.2 Подпись над манифестом, SHA — внутри манифеста

Подписываем **манифест** (маленький, канонизируемый JSON), а целостность
большого APK гарантируем через `apk_sha256` **внутри** подписанного манифеста.
Так не нужно подписывать многомегабайтный бинарь и можно проверить APK
поточно при скачивании. Каноникализация — фиксированный порядок полей
(переиспользуем подход подписи пресетов LLD-01) либо подпись над сырыми байтами
ровно того JSON, что отдан (хаб хранит и отдаёт `manifest` как непрозрачную
строку + `signature`).

### 3.3 Проверка подписи/SHA — в Rust (xr-core), установка — в Kotlin

ed25519-verify и SHA-256 живут в `xr-core` (рядом с `onboarding.rs`, тот же
reqwest-клиент и формат) — единая криптопроверка, покрытая unit-тестами без
Android. Kotlin отвечает только за то, что нельзя сделать в Rust: скачать в файл с
прогрессом и вызвать `PackageInstaller`. JNI — `nativeCheckUpdate(hubUrl,
currentVersionCode, pinnedKey) -> JSON {available, manifest, error}` и
`nativeVerifyApk(path, expectedSha256) -> bool`.

### 3.4 APK раздаёт хаб с диска, не из rust-embed

APK — большой и меняется каждый релиз; встраивать в бинарь хаба (rust-embed)
нельзя. Кладём в `/var/lib/xr-hub/releases/<version>.apk` + `manifest.json` +
`manifest.sig`; эндпоинты читают с диска. Загрузка нового релиза — владелец
кладёт файлы и подписывает локально (`xr-hub sign-release` CLI или офлайн-скрипт,
см. §4). Хаб **не подписывает** релиз сам (у него нет приватного release-ключа).

## 5. Риски и edge-кейсы

1. **Компрометация VPS** (главный риск, ради него весь дизайн). Атакующий
   подменяет APK+манифест, но не имеет приватного release-ключа → подпись не
   сходится → приложение отвергает. Защита держится, **пока release-ключ не на
   VPS**. Документируем жирно (§3.1, HUB-DEPLOY).
2. **Утечка приватного release-ключа.** Тогда атакующий может подписать
   вредонос. Митигация: ключ офлайн, ротация = смена pinned-ключа = новая сборка
   приложения (раздаётся прежним доверенным каналом). Считаем редким событием.
3. **Разные APK-signing keystore между релизами.** Android откажет в установке
   поверх. Требование: один keystore на все релизы; первый переход с `adb`-сборки
   на self-update должен быть тем же keystore, иначе пользователь переустанавливает
   вручную один раз. Документируем.
4. **`REQUEST_INSTALL_PACKAGES` не выдан / отозван.** `PackageInstaller` бросит /
   система покажет экран разрешения. Ведём пользователя туда, не падаем.
5. **Скачан битый/обрезанный APK** (как баг C2 на download, но тут — обычный HTTP).
   SHA-256 над полным файлом ловит обрыв → reject + повтор.
6. **Понижение версии / replay старого манифеста.** Проверяем `version_code >
   current`; подписанный старый манифест с меньшим кодом не предлагается. (Полный
   anti-rollback с монотонным счётчиком — follow-up, для self-hosted одиночного
   флота избыточно.)
7. **MITM на HTTPS.** Хаб уже за TLS (LLD-01); даже при компрометации TLS подпись
   манифеста — независимый слой (защита не опирается только на транспорт).

## 6. План проверки

1. **Happy path.** Поднять версию в манифесте, подписать `sign-release`, выложить
   на хаб. В приложении (ручная проверка) → баннер; «Обновить» → скачивание →
   SHA ОК → установщик → версия обновилась, данные/настройки сохранены.
2. **Битая подпись.** Руками испортить байт в `manifest.json` (не трогая `.sig`)
   → приложение **не** предлагает обновление, WARN в логе.
3. **Чужой ключ.** Подписать манифест другим ключом → reject.
4. **Битый SHA.** Подменить APK после подписи манифеста → манифест-подпись ок, но
   SHA не сходится → reject, temp-файл удалён.
5. **Старая версия.** Манифест с `version_code` ≤ текущего → апдейт не предлагается.
6. **Нет разрешения на установку.** Отозвать `REQUEST_INSTALL_PACKAGES` → корректный
   переход в системный экран, без краша.
7. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Автоматическая (silent) установка без подтверждения** — Android этого без
  device-owner не даёт, и небезопасно; всегда явный системный диалог.
- **Дельта-обновления / патчи** — качаем полный APK; дельты — преждевременная
  оптимизация для редких релизов.
- **Раздача через Google Play / F-Droid** — self-hosted канал по дизайну.
- **Push-нотификация «вышло обновление»** — поллинг по интервалу; push — общий
  follow-up (см. LLD-11 §3.4).
- **Anti-rollback с монотонным счётчиком версий на сервере** — §5.6 follow-up.
- **Обновление самого xr-server/xr-client/xr-hub** — это provisioning/деплой
  (LLD-13), не клиентский self-update.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Формат манифеста?* → JSON `{version_code, version_name, min_sdk, apk_url,
  apk_sha256, size, notes, released_at}` + detached ed25519-подпись (§2.1).
- *Канал раздачи?* → хаб-эндпоинты `/app/latest` + `/app/download/:ver`, файлы на
  диске в `releases/` (§3.4).
- *UX и `REQUEST_INSTALL_PACKAGES`?* → баннер → скачивание+SHA → `PackageInstaller`,
  ведём в системный экран при отсутствии разрешения (§2.3).
- *Откат при плохой подписи?* → не предлагаем обновление, старая версия работает
  (§2.4).
- *VPS-compromise ≠ RCE?* → отдельный офлайн release-ключ, pinned в сборке;
  подпись не подделать без приватной половины (§2.2, §3.1).
