# LLD-10 — Multi-VPS failover на роутере (пул серверов)

**Статус:** Draft
**Область:** `xr-proto` (новый `ServerPool` поверх нескольких `MuxPool`,
расширение `config.rs`), `xr-client` (чтение списка серверов, проброс пула в
`ProxyState`), `xr-core` (тот же пул доступен Android-движку, но UI выбора —
LLD-08).
**Зависимости:** [LLD-09](09-multi-mux-pool.md) — `MuxPool` с N слотами,
round-robin, failover **внутри одного сервера**; обобщаем до пула *серверов*.
Circuit-breaker `is_server_down()` / `MuxPool::recycle()` из C1/C4
(`xr-proto/src/mux_pool.rs`) — готовая «память о падении» и пересоздание пула.
**Связанные документы:** [ARCHITECTURE.md §3 (mux)](../ARCHITECTURE.md),
[ARCHITECTURE.md §6 (конфиг)](../ARCHITECTURE.md), [CLAUDE.md](../../CLAUDE.md),
`local-docs/routers.md` (два VPS: Aeza + Timeweb).

Один VPS — единая точка отказа. Инцидент с неоплаченным хостингом (задача 1a)
показал это на практике: пока сервер был мёртв, проксируемый трафик деградировал
до тех пор, пока breaker (C1) не уводил его в Direct — но **резервного сервера,
на который можно переехать, не было**. Этот LLD добавляет на роутере (xr-client)
пул из ≥2 серверов с автоматическим failover'ом: падение активного сервера
переводит проксируемый трафик на резервный за секунды, а не в Direct.

---

## 1. Текущее состояние

- Конфиг клиента описывает **ровно один** сервер
  ([config.rs](../../xr-proto/src/config.rs), `[server] address/port`,
  `routers.md §5`). Резервный VPS существует физически (Aeza + Timeweb), но
  переключение между ними — ручной `sed -i s/IP1/IP2/ config.toml` + restart
  (`routers.md §3.2`).
- `MuxPool` (LLD-09) держит N (default 4) параллельных TCP-туннелей **к одному
  серверу**, балансирует стримы round-robin, failover'ит между слотами и
  переподнимает мёртвый слот. Это failover *внутри* сервера, не *между*
  серверами.
- Circuit-breaker (C1): после полного провала обхода слотов `MuxPool` помечает
  сервер `down` на cooldown, `open_stream` мгновенно отдаёт `Err` →
  `on_server_down="direct"` уводит соединение в Direct. `is_server_down()` —
  публичный хук состояния; `recycle()` (C4) пересоздаёт пул и снимает breaker.
- Итог: при мёртвом активном VPS трафик идёт **Direct** (часть заблокированных
  ресурсов не откроется), хотя живой резервный сервер мог бы его принять.

## 2. Целевое поведение

### 2.1 Список серверов в конфиге

`[server]` (одиночный) обобщается до упорядоченного `[[servers]]` с приоритетом:

```toml
[[servers]]
name = "aeza"               # человекочитаемый лейбл для логов
address = "85.192.38.29"
port = 8443
priority = 0                # меньше = выше приоритет (0 = primary)

[[servers]]
name = "timeweb"
address = "201.51.16.159"
port = 8443
priority = 1                # backup
# key = "..."               # опционально: override общего [obfuscation].key,
                            # если у резерва другой ключ; по умолчанию — общий
```

- Обфускация по умолчанию **общая** (`[obfuscation]` как сейчас) — оба наших VPS
  делят ключ. Per-server `key`/`salt`/`modifier` — опциональный override на
  будущее (разные ключи на разных провайдерах), в MVP не обязателен.
- `mux_pool_size` (LLD-09) остаётся **глобальным** — каждый сервер поднимает свой
  `MuxPool` этого размера.
- **Обратная совместимость:** старый `[server]` парсится как `[[servers]]` из
  одного элемента с `priority=0`. Конфиги обоих боевых роутеров продолжают
  работать без правки до миграции.

### 2.2 Модель пула серверов

```
┌──────────────────────────────────────────────────────────┐
│ ServerPool                                               │
│   active: AtomicUsize  ──┐                               │
│   ┌─────────────────────▼──────────┐  ┌────────────────┐ │
│   │ ServerSlot #0 (priority 0)     │  │ ServerSlot #1  │ │
│   │   MuxPool(N) ── breaker        │  │   MuxPool(N)   │ │
│   │   health: Up / Down(since,cls) │  │   health: ...  │ │
│   └────────────────────────────────┘  └────────────────┘ │
└──────────────────────────────────────────────────────────┘
        ↑ open_stream → active server's MuxPool
```

- `ServerPool` держит по одному `MuxPool` на сервер (прямое обобщение LLD-09:
  LLD-09 раскладывал стримы по N TCP одного сервера, LLD-10 — по M серверам, у
  каждого свои N TCP).
- `open_stream(target)` идёт в `MuxPool` **активного** сервера. Если его breaker
  взведён (`is_server_down()`), пул выбирает следующий по приоритету **здоровый**
  сервер, делает его активным и идёт в него. Если все серверы down — возвращает
  `Err`, и `on_server_down="direct"` уводит соединение в Direct (как сейчас, но
  теперь это происходит только когда исчерпан **весь** пул).

### 2.3 Политика: primary/backup, не балансировка

**Решение открытого вопроса (problems.md 1b):** серверы **не равноправны** —
строгий приоритетный порядок (primary/backup), а не балансировка нагрузки.
Причины:

- Балансировка по серверам осмысленна только при сопоставимой ёмкости и цене
  трафика; у нас два разных VPS разных провайдеров, цель — *отказоустойчивость*,
  а не суммарная полоса.
- Sticky-to-primary даёт предсказуемый «домашний» сервер: меньше прыжков
  fingerprint'а между IP, проще диагностика по логам.
- Балансировку внутри одного провайдера уже даёт `MuxPool` (N TCP). Пул серверов
  — это слой *резервирования*, а не *масштабирования*.

**Инвариант выбора активного:** активен сервер с наименьшим `priority` среди
здоровых. Падение primary → активным становится следующий здоровый.

### 2.4 Health-check: пассивный детект + активный пробинг

Два механизма, дополняющие друг друга:

- **Пассивный (основной триггер failover'а)** — breaker из C1. Любой реальный
  стрим, упавший в `is_server_down()`, мгновенно переключает активный сервер на
  следующий здоровый. Бесплатно, без лишнего трафика, реагирует ровно на ту
  деградацию, что видит пользователь.
- **Активный (для возврата и pre-warm резерва)** — лёгкий фоновый пробер
  (`ServerPool::health_loop`), раз в `health_interval_secs` (default 15с)
  проверяющий серверы **выше по приоритету активного** и сам активный. Проба =
  переиспользуем probe-lease механику breaker'а (C1): один прободатель ставит
  mux-handshake. Зачем активная проба, если есть пассивная: пассивная узнаёт о
  *восстановлении* primary только когда туда случайно пойдёт стрим — а при
  sticky-to-primary стримы туда не идут, пока он «down». Активный пробер
  закрывает этот провал.

### 2.5 Возврат на primary (failback) с hold-down

После восстановления primary активный трафик надо вернуть на него, но **без
флаппинга** на нестабильной связи:

- Прободатель видит primary здоровым → запускает таймер `failback_hold_secs`
  (default 60с).
- Если primary остаётся здоровым непрерывно весь hold-down — `ServerPool`
  переключает `active` обратно на него (failback). Если за время hold-down
  primary снова мигнул в down — таймер сбрасывается.
- Failback атомарен на уровне `active` индекса; живые стримы на backup
  дорабатывают (их `MuxStream` не рвём), новые стримы идут на primary. Старые
  mux backup'а лениво схлопываются по `MUX_MAX_LIFETIME`.

### 2.6 Связь с Android (xr-core)

Движок Android использует тот же `xr-core::SessionContext` → `MuxPool`. После
обобщения в `xr-proto` Android-движок **автоматически** получает `ServerPool`,
если ему передан список серверов. Но пользовательский **список/выбор серверов**
на Android — это LLD-08 (UI мультисервера), где активный сервер выбирает человек.
Failover *между* серверами на Android (автоматический, как на роутере) —
включается в LLD-11 (панель здоровья) поверх этого пула. В этом LLD Android-часть
ограничена: «движок умеет принять N серверов и failover'ить», UI и политика — в
LLD-08/11. Разводим, чтобы C5 не размазал один engine-механизм по трём чатам.

## 3. Дизайн-решения

### 3.1 ServerPool как обёртка над MuxPool, а не переписывание

`MuxPool` уже умеет всё нужное на уровне одного сервера (слоты, RR, failover,
breaker, recycle). `ServerPool` — тонкий слой выбора *какой* `MuxPool`
обслуживает `open_stream`. Не трогаем wire-протокол, не трогаем логику слотов.
Это минимизирует регрессионный риск для боевых роутеров.

### 3.2 Источник списка серверов — конфиг роутера, не хаб (в MVP)

Открытый вопрос problems.md: config vs hub-пресет. **Решение:** в MVP список
серверов живёт в `client.toml` на роутере. Причины:

- Серверов мало (2), меняются редко, доступ к роутеру по SSH у владельца есть.
- Раздача *адресов серверов* через хаб — это раздача точек входа в обфусцированный
  туннель; компрометация хаба не должна перенаправлять флот на чужой VPS.
  (Адреса в инвайте уже подписаны TOFU-ключом — LLD-04; централизованная раздача
  списка серверов через подписанный пресет — отдельная фича, в этот LLD не лезем.)
- Hub-managed список серверов — естественное расширение поверх LLD-13
  (provisioning добавляет свежеподнятый сервер) и LLD-01 (подписанные пресеты).
  Помечаем как «вне скоупа, follow-up».

### 3.3 Почему failover не делает сразу Direct

Сейчас единственный сервер down → Direct. Это и есть деградация, ради ухода от
которой нужен пул. Direct остаётся **последним** рубежом (все серверы down), а не
первым: проксируемый ресурс заблокирован → Direct его не откроет, тогда как живой
backup — откроет. Порядок: primary → backup(s) → Direct.

### 3.4 Hold-down против флаппинга, без гистерезиса по латентности

Failback завязан только на «health up непрерывно N секунд», без сложного
скоринга RTT/потерь. Простая, предсказуемая политика; тонкую классификацию
«сервер деградировал, но жив» отдаём LLD-11 (мониторинг). Здесь — бинарно
up/down (как breaker) + анти-флаппинг таймером.

## 5. Риски и edge-кейсы

1. **Все серверы down одновременно** (общий провайдерский сбой / DPI на оба IP).
   Пул отдаёт `Err` → `on_server_down="direct"`. Поведение не хуже сегодняшнего;
   LLD-11 эскалирует это как «инцидент: все серверы недоступны».
2. **Флаппинг primary** (мигает up/down). Hold-down (§2.5) гасит; без него
   трафик прыгал бы между IP, ломая длинные стримы и засоряя fingerprint.
3. **Разные ключи на серверах.** Если у backup другой `[obfuscation].key`, а
   override не указан — handshake к backup провалится и он будет вечно «down».
   Валидация конфига при старте + WARN в логе «у servers[i] нет ключа и общий ключ
   к нему не подходит» (детектится первой failed-пробой). В MVP оба VPS делят ключ
   — не стреляет, но документируем.
4. **Рост ESTAB на серверах.** Пул держит mux к **каждому** серверу (warmup) →
   на backup тоже висят N TCP, даже когда он неактивен. Это сознательно (мгновенный
   failover без cold-start). Учесть `max_connections=256` на VPS (как в LLD-09 §6).
   Опция: держать backup «холодным» (mux поднимать только при failover) —
   компромисс латентности; оставляем тёплый warmup, но с меньшим `mux_pool_size`
   для backup как возможный follow-up.
5. **Сетевой re-bind (C4) одновременно с failover.** `on_network_changed` зовёт
   `recycle()` — теперь надо recycle'ить пул *каждого* сервера. `ServerPool::recycle()`
   проходит по всем слотам. Следить, чтобы health_loop и recycle не гонялись за
   один и тот же слот-lock (переиспользовать существующую синхронизацию MuxPool).
6. **Конфиг с пустым `[[servers]]`.** Валидация: ≥1 сервер обязателен, иначе явная
   ошибка старта (как `source_ips` для UDP relay — отказываемся стартовать с
   небезопасной/бессмысленной конфигурацией).

## 6. План проверки

Стенд — оба боевых роутера (`routers.md §2.2`), scoped nft-drop как в C1 для
имитации мёртвого VPS.

1. **Failover primary→backup.** Конфиг с Aeza (primary) + Timeweb (backup).
   Scoped-drop 8443 к Aeza. Проксируемый curl из LAN (`claude.ai`) должен
   продолжить работать через Timeweb за секунды; в логе — «server aeza down →
   active=timeweb». Direct-трафик не затронут.
2. **Failback.** Снять drop с Aeza. В течение `failback_hold_secs` активный
   остаётся Timeweb, затем переключается обратно на Aeza; новые стримы идут на
   Aeza (`logread` / счётчики ESTAB на VPS, `routers.md §7.4`).
3. **Все серверы down.** Drop 8443 к обоим. Проксируемый curl падает в Direct
   за ~0.5с (как C1); непроксируемый Direct не затронут.
4. **Анти-флаппинг.** Drop/undrop Aeza с периодом < hold-down — активный не должен
   прыгать; остаётся на backup до стабилизации primary.
5. **Legacy-конфиг.** Роутер со старым `[server]` стартует без правки, ведёт себя
   как пул из одного сервера (Direct при его падении — как сегодня).
6. `cargo test --workspace` зелёный, 0 warnings (CLAUDE.md).

## 7. Вне скоупа

- **Раздача списка серверов через xr-hub-пресет** (подписанный, hot-swap). Будет
  естественно поверх LLD-13 (provisioning добавляет сервер) + LLD-01. См. §3.2.
- **Балансировка нагрузки между серверами** (active-active). Сознательно не
  делаем — пул про резервирование, не масштабирование (§2.3).
- **UI выбора/списка серверов на Android** — LLD-08. **Авто-failover + панель
  здоровья на Android** — LLD-11.
- **Классификация классов сбоя** («DPI» vs «сервер не отвечает» vs «auth») —
  `HealthState.class` здесь только заведён как поле, наполняет его LLD-11.
- **Гео-/латентностный выбор «ближайшего» сервера** — нет, строгий приоритет.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Равноправны или primary/backup?* → **primary/backup**, строгий приоритет (§2.3).
- *Липкость после восстановления?* → **да**, sticky-to-primary + failback с
  hold-down (§2.5).
- *Где список серверов — config или hub?* → **config роутера** в MVP; hub —
  follow-up поверх LLD-13 (§3.2).
- *Health-check пассивный или активный?* → **оба**: пассивный breaker как триггер
  failover'а, активный пробер для failback и pre-warm (§2.4).
