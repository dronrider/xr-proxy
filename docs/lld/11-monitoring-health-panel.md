# LLD-11 — Мониторинг, уведомления и панель здоровья

**Статус:** Draft
**Область:** `xr-proto` (классификация сбоя в breaker/`ServerPool`), `xr-core`
(агрегирование health в snapshot, инцидент-таймлайн), `xr-android` (слои
индикатора вместо смайлика, панель здоровья по серверам, локальные уведомления,
напоминание об оплате), `xr-hub` (опциональный `/api/v1/health`).
**Зависимости:** [LLD-06](06-android-visual.md) — `ShieldArrowIcon`, `HealthFace`,
палитра, раскладка статистики (этот LLD заменяет смайлик слоями статуса).
[LLD-08](08-android-multi-server.md) — `ServerProfile`/`ServerRepository`; health
и дата оплаты — атрибуты профиля. [LLD-10](10-router-multi-vps-failover.md) —
`ServerPool` + `HealthState`: источник «сервер up/down/класс сбоя».
[LLD-02](02-android-reliability.md) — `recentErrors`, `StatsSnapshot`, foreground
notification, `POST_NOTIFICATIONS`.
**Связанные документы:** [ARCHITECTURE.md §8 (наблюдаемость)](../ARCHITECTURE.md),
problems.md задачи 6 и 10.

При сбое пользователю **непонятно, кто виноват**: провайдер, неоплаченный хостинг,
упавший процесс или DPI. Сейчас единственный индикатор — смайлик-`HealthFace`
(LLD-06), который не различает классы сбоя и не объясняет причину. Этот LLD
превращает приложение в **панель здоровья**: явные слои статуса вместо эмоции,
классификация сбоев по симптому, локальные уведомления при падении/восстановлении
и напоминание об оплате хостинга — прямой ответ на P0-инцидент 1a.

**Решение об объединении задач 6 и 10 (открытый вопрос problems.md):** делаем
**один** LLD. Слои индикатора (задача 10) — это и есть пользовательская
поверхность панели здоровья (задача 6); классификация сбоя и напоминание об
оплате — её содержание. Разводить по двум экранам нечего: это одна ось «насколько
здорово соединение и почему».

---

## 1. Текущее состояние

- Индикатор соединения — единственный `HealthFace` (😊/😐/☹️) из LLD-06,
  выводится эвристикой из `relay_errors`/скорости. Не отвечает на вопрос «почему
  плохо».
- `StatsSnapshot` (LLD-02, [stats.rs](../../xr-core/src/stats.rs)) несёт
  bytes up/down, connections, uptime, debug-метрики и `recent_errors` (журнал
  строк). Нет понятия «инцидент», «класс сбоя», «здоровье *конкретного* сервера».
- Уведомление одно — foreground-notification сервиса (LLD-02): «VPN активен», без
  реакции на деградацию.
- Breaker (C1) знает «сервер down», но **не различает причину** падения (connect
  fail / handshake reset / auth reject) — у `HealthState.class` (LLD-10) пока
  только `Unknown`.
- Даты оплаты хостинга нигде нет; инцидент 1a случился именно из-за молчаливого
  истечения оплаты.

## 2. Целевое поведение

### 2.1 Классификация классов сбоя

В точке, где breaker фиксирует падение сервера, различаем три класса по симптому:

| Класс | Симптом | Вероятная причина |
|---|---|---|
| `ServerUnreachable` | TCP connect к `:8443` не устанавливается (timeout/refused) | сервер лежит / хостинг не оплачен / сеть до VPS |
| `HandshakeReset` | TCP connect ОК, но obf/mux-handshake рвётся (RST / молчание после connect) | **DPI/провайдер** режет туннель по сигнатуре, либо заглушка хостера принимает TCP но не отвечает |
| `AuthFailed` | сервер отвечает, но отвергает (deobfuscate/MuxInit reject) | рассинхрон ключа/salt, протухший конфиг |

Класс пишется в `HealthState::Down { since, class }` (поле заведено в LLD-10) и
поднимается в snapshot. Различение `ServerUnreachable` vs `HandshakeReset` —
ровно тот сигнал, что отделяет «заплати за хостинг» от «провайдер давит туннель».

**Порог «инцидент начался»:** инцидент = активный сервер перешёл в `Down`
(breaker взведён) И прошёл grace-период `incident_grace_secs` (default 10с),
чтобы единичный мигающий сбой не поднимал тревогу. `since` берётся от момента
breaker-trip, отображается как «инцидент длится N мин».

### 2.2 Слои индикатора (замена смайлика)

`HealthFace` заменяется на **три слоя статуса**, рендерящиеся в
`ConnectionSection` главного экрана (LLD-06/08 раскладка):

1. **Туннель** — установлен ли канал: `Phase` (Idle/Connecting/Connected) +, в
   Connected, состояние активного сервера (`Up` / `Down(class)` / `failover на
   backup`).
2. **Трафик** — идёт ли поток: live up/down скорость (уже есть в snapshot), флаг
   «тихо N сек» если скорость нулевая при Connected.
3. **Качество** — RTT до активного сервера (из health-пробы LLD-10) + доля
   ретраев/failover'ов за окно. Грубая шкала: хорошо / средне / плохо — но с
   **подписью-причиной**, а не эмоцией.

Визуально: компактная строка-чип под `ShieldArrowIcon` («Подключено · timeweb ·
12 МБ/с · RTT 80мс») + разворачиваемая карточка с детализацией по слоям. Иконка
`ShieldArrowIcon` сохраняет анимацию по фазам (LLD-06), но её «настроение»
выводится из слоёв, не из отдельной эвристики.

### 2.3 Панель здоровья по серверам

На вкладке Servers (LLD-08) каждая карточка сервера получает строку здоровья:

- Индикатор `Up`/`Down(class)`/`не проверялся`, цвет по классу.
- Время последней успешной пробы / длительность инцидента.
- Для активного — RTT и текущая роль (primary активен / failover на этот backup).
- Дата оплаты хостинга (§2.5) и «осталось N дней».

Это поверхность для `ServerPool::server_health()` (LLD-10) на роутерном движке и
для per-profile health на Android.

### 2.4 Уведомления о падении/восстановлении

Локальные уведомления (без сети, без push-инфраструктуры в MVP):

- **Падение активного сервера** (инцидент прошёл grace): уведомление с **классом
  и подсказкой** — «Сервер timeweb не отвечает. Возможно, не оплачен хостинг» /
  «Соединение режется на handshake — похоже на DPI провайдера» / «Сервер отверг
  подключение — проверьте ключ». Если есть живой backup — «переключились на
  aeza».
- **Восстановление** — «Сервер timeweb снова доступен» (снимаем/обновляем
  уведомление инцидента).
- **Все серверы down** — «Нет доступных серверов, проксируемые ресурсы идут
  Direct» (эскалация поверх LLD-10 §5.1).

Отдельный notification channel `IMPORTANCE_DEFAULT` «Состояние соединения»
(не глушит foreground-канал сервиса). Требует `POST_NOTIFICATIONS` (уже
запрашивается, LLD-02).

### 2.5 Напоминание об оплате хостинга

**Решение открытого вопроса (где живёт дата):** поле в `ServerProfile` (LLD-08),
не в хабе. Сервер «принадлежит» профилю на клиенте, дата оплаты — локальный
атрибут владельца, хаб тут ни при чём.

- Новое опциональное поле `paidUntil: String?` (RFC 3339 date) в `ServerProfile`,
  редактируется в `ServerEditScreen` (LLD-08 §3.6).
- За `reminder_lead_days` (default 5) до даты — локальное уведомление «Хостинг
  сервера X оплачен до DD.MM, осталось N дней». На дату/после — «Хостинг X истёк
  — оплатите, иначе сервер отключится».
- Проверка — раз в сутки из сервиса (или при старте туннеля), дешёвая дата-сверка,
  сети не требует.

### 2.6 Серверная часть: опциональный health-эндпоинт xr-hub

`GET /api/v1/health` на xr-hub — отдаёт `{ status: "ok", version, uptime_secs }`
самого хаба (liveness). **Опционально** хранит короткую историю аптайма по
известным серверам, если их пингует (follow-up). В MVP — только liveness хаба:
приложение может отличить «хаб жив, но мой VPS лёг» от «всё лежит». Не делаем
агрегированный мониторинг флота — это отдельная инфраструктура.

## 3. Дизайн-решения

### 3.1 Классификация — в Rust, рядом с breaker'ом

Класс сбоя определяется там, где происходит сам сбой connect/handshake — в
`MuxPool`/`ServerPool` (`xr-proto`), а не угадывается в Kotlin по тексту ошибки.
Это даёт единый источник и для роутера (логи/crash.log), и для Android (snapshot),
и переиспользуется LLD-10 failover-логикой. Чистая классификация (по типу
`io::Error` / стадии handshake) покрывается unit-тестами без сети.

### 3.2 Снимок здоровья едет тем же каналом, что и stats

Расширяем `StatsSnapshot` секцией `servers: Vec<ServerHealth>` + `incident:
Option<Incident>`. Android читает их теми же `nativeGetStats()`, что и метрики
(LLD-02 §4.6) — никакого нового JNI-поллинга. `ServerHealth { name, state, class,
since, rtt_ms, role }`.

### 3.3 Слои вместо смайлика — почему

Смайлик кодирует *одно* число в эмоцию и теряет причину. Пользователь в инциденте
1a видел «сеть без интернета» и не знал, что виноват неоплаченный хостинг. Слои
+ подпись-причина прямо отвечают «что сломано и что делать». `HealthFace` из
LLD-06 убирается (или прячется за debug) — конфликт двух индикаторов недопустим;
согласовать с LLD-08 §2.4 (там HealthFace выносился inline в строку статуса) —
**этот LLD её замещает слоями**, отметить в обоих документах при имплементации.

### 3.4 Уведомления локальные, не push

Push (FCM) тянет Google-инфраструктуру, токены, серверную рассылку — избыточно
для self-hosted одиночного пользователя. Все сигналы (падение/восстановление/
оплата) детектируются **на устройстве** из health-снимка и таймера. Push как
канал — вне скоупа (follow-up, если появятся несколько устройств владельца).

### 3.5 Инцидент-таймлайн минимальный

Храним один «текущий инцидент» на активный сервер (начало, класс, был ли
failover), не полную историю. Полноценный uptime-лог — серверная фича (§2.6
follow-up). Цель MVP — ответить «что сейчас не так и сколько длится», не вести
журнал аптайма.

## 5. Риски и edge-кейсы

1. **Ложная классификация `HandshakeReset` при «неоплаченном хостинге».** Заглушка
   хостера может принять TCP и молчать → выглядит как DPI. Различаем по таймауту:
   молчание после connect без RST в первые мс → ближе к `ServerUnreachable`
   (хостер-заглушка), быстрый RST на handshake → `HandshakeReset` (DPI). Граница
   эвристична; подпись уведомления формулируем осторожно («возможно…»), не
   категорично.
2. **Спам уведомлений при флаппинге.** Дебаунс: уведомление о падении — только
   после grace; о восстановлении — только если до этого было падение; rate-limit
   на канал (не чаще раза в `notif_min_interval`). Failback (LLD-10) не должен
   плодить пары «упал/встал».
3. **`paidUntil` в прошлом из-за неверной таймзоны.** Сравниваем по локальной дате
   устройства, без времени; формат RFC 3339 date (YYYY-MM-DD). Дата-only снимает
   путаницу TZ.
4. **Health-слои на Android при одном сервере.** Если на Android пул из одного
   сервера (LLD-08 активный), failover-слой просто не показывается; слои трафика/
   качества работают как обычно.
5. **Конфликт двух индикаторов.** `HealthFace` (LLD-06/08) и новые слои не должны
   сосуществовать — удаляем/прячем смайлик (§3.3), синхронно правим LLD-06 §3.5a и
   LLD-08 §2.4 при имплементации.
6. **Notification permission отозван.** Мягкая деградация: уведомления молча не
   показываются (как foreground LLD-02), панель здоровья в UI всё равно работает.

## 6. План проверки

Android — ручная (правило [[feedback_android_tests]]); Rust-классификация и
инцидент-грейс — автотестами.

1. **Класс `ServerUnreachable`.** Активный VPS down (scoped-drop 8443) → панель и
   уведомление «не отвечает, возможно хостинг», слой «Туннель» красный, при наличии
   backup — «переключились на …».
2. **Класс `HandshakeReset`.** Поднять на стенде заглушку, рвущую соединение после
   TCP-connect → класс `HandshakeReset`, подпись «похоже на DPI».
3. **Восстановление.** Вернуть сервер → уведомление «снова доступен», слои зелёные,
   инцидент закрыт.
4. **Все серверы down.** Drop оба → «нет серверов, идём Direct»; непроксируемое
   работает.
5. **Напоминание об оплате.** Выставить `paidUntil` через 3 дня → в течение суток
   приходит «осталось 3 дня»; дата в прошлом → «истёк».
6. **Слои индикатора.** Connected с трафиком → «трафик идёт, RTT N»; обрубить
   трафик → слой «тихо»; смайлик отсутствует.
7. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Push-уведомления (FCM)** — локальных достаточно для одного владельца (§3.4).
- **Агрегированный мониторинг флота / история аптайма на хабе** — только liveness
  в MVP (§2.6); history — follow-up.
- **Авто-действия по классу сбоя** (напр. авто-смена ключа при `AuthFailed`) — UI
  только показывает причину; чинит человек.
- **Серверные алерты владельцу по e-mail/телеграм** — отдельная инфраструктура.
- **Графики/таймсерии трафика** — панель показывает мгновенное состояние, не
  историю.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Какие классы сбоя различаем?* → три: `ServerUnreachable` / `HandshakeReset` /
  `AuthFailed` (§2.1).
- *Где порог «инцидент начался»?* → breaker-trip активного сервера + grace
  `incident_grace_secs` (§2.1).
- *Объединять задачи 6 и 10 в один LLD?* → **да** — слои индикатора суть
  поверхность панели здоровья (преамбула, §3.3).
- *Дата оплаты — в профиле (LLD-08) или в хабе?* → **в `ServerProfile`** (§2.5,
  §3.4).
