# LLD-17. Хаб-реестр роутеров

**Статус:** Draft
**Область:** `xr-hub` (реестр роутеров, enrollment, приём статус-отчётов, раздел
«Роутеры» в Admin SPA), `xr-client` (per-router идентичность, исходящий poll:
отчёт о статусе), `xr-proto` (типы записи роутера, креденшела, отчёта),
`xr-bootstrap` (enrollment во время установки, шов с LLD-13).
**Зависимости:** [LLD-01](01-control-plane.md): xr-hub, инвайты, ed25519,
Admin SPA, axum, JSON-стор: расширяем реестром.
[LLD-13](13-zero-touch-provisioning.md): bootstrap регистрирует роутер и пишет
per-router ключ в конфиг (это «шов» установка -> учёт).
[LLD-10](10-client-multi-vps-failover.md): `ServerPool`/`HealthState`, статус,
который роутер репортит наверх. [LLD-11](11-monitoring-health-panel.md):
классификация сбоя и `ServerHealth`, содержимое отчёта. [LLD-16](16-manual-server-hub-rules.md):
per-router назначение пресетов через хаб.
**Связанные документы:** [ARCHITECTURE.md §3 (хаб), §8 (наблюдаемость)](../ARCHITECTURE.md),
memory [[reference_deploy_handoff]], [[project_failopen_breaker]] (`is_server_down`
как источник статуса). **Удалённое управление** (подписанные команды поверх этого
реестра) это отдельный [LLD-20](20-router-remote-management.md) (XR-048).
Метрики/история/Grafana это [LLD-18](18-fleet-metrics-grafana.md) (XR-026), здесь
только транспорт отчёта и его хранение «последним снимком».

Сейчас «роутер, зарегистрированный в сети хаба» невозможен: хаб видит всех
клиентов **анонимно** (общий obfuscation-ключ, публичные пресеты, никакой записи о
роутере, LLD-01), клиент только **тянет** пресеты, а хаб за NAT до роутера (серый
IP) **не достучится** ни при установке, ни в работе. Этот LLD добавляет (1)
идентичность роутера и enrollment, (2) **исходящий** poll-канал роутер -> хаб, по
которому роутер репортит статус, (3) раздел «Роутеры» в админке. Хаб начинает
**знать и видеть** свой парк роутеров. Управление этим парком (команды) строится
поверх в LLD-20.

---

## 1. Текущее состояние

- **Хаб не аутентифицирует клиентов** (LLD-01): общий obfuscation-ключ, пресеты
  публичны, нет записи роутера, `router_id`, `last_seen`, версии, статуса.
- **Клиент только pull'ит пресеты** ([xr-client/src/main.rs](../../xr-client/src/main.rs),
  `PresetCache` раз в `refresh_interval`). Хаб не знает, жив ли роутер.
- **NAT.** Роутер за серым IP; хаб (VPS) не может инициировать к нему соединение
  ни при установке, ни потом. Inbound невозможен в принципе.
- **Деплой/настройка руками по SSH** ([[reference_deploy_handoff]],
  `routers.md`); LLD-13 автоматизирует установку, но не даёт идентичности роутера
  на хабе.
- Итог: «зарегистрирован в сети хаба» сегодня = «где-то стоит клиент с тем же
  ключом», без видимости.

## 2. Целевое поведение

### 2.1 Идентичность и enrollment

Запись роутера на хабе:

```
RouterRecord {
  router_id,        // случайный, не переиспользуется
  name,             // человекочитаемое («роутер родителям»)
  secret_hash,      // per-router секрет (хранится хэшем, как admin-пароль LLD-01)
  enrolled_at, last_seen,
  client_version,   // версия xr-client (из отчёта)
  assigned_preset,  // какой пресет роутеру назначен (LLD-16)
  last_report,      // последний снимок статуса (§2.3), для админки
}
```

**Enrollment во время установки (шов с LLD-13).** Чтобы не класть admin-креды
хаба в приложение/bootstrap, переиспользуем механику инвайта (LLD-01):

1. Владелец в админке создаёт **одноразовый enrollment-токен** (TTL, как инвайт),
   либо приложение получает его тем же путём, что и инвайт.
2. `xr-bootstrap` (LLD-13) на LAN после установки клиента дёргает
   `POST /api/v1/enroll {token, name, arch, version}` -> хаб создаёт
   `RouterRecord`, возвращает `{router_id, secret, command_pubkey}`, помечает
   токен `consumed`.
3. Bootstrap пишет `[control]` в `config.toml` роутера: `hub_url`, `router_id`,
   `secret` (chmod 600), закреплённый `command_pubkey` (TOFU, как public-key
   LLD-04). Ключ нужен уже здесь, чтобы команды LLD-20 потом проверялись.
4. Клиент стартует, открывает poll-канал (§2.2) -> роутер виден в реестре
   `online`.

### 2.2 Исходящий poll-канал

`xr-client` периодически (`control_poll_interval`, default 30с) шлёт
`POST /api/v1/router/poll` с аутентификацией (`router_id` + `secret`, bearer/HMAC)
и телом = краткий отчёт (§2.3). Хаб:

- обновляет `last_seen` и сохраняет `last_report`;
- отвечает версией назначенного пресета (роутер сам тянет полный пресет
  существующим путём LLD-01, если устарел).

Pull выбран сознательно: NAT-дружелюбен (только исходящее), переиспользует
существующий polling-движок. Тот же канал в LLD-20 расширяется доставкой команд;
здесь он несёт только статус.

### 2.3 Отчёт статуса (шов к XR-026)

Тело poll-запроса это компактный JSON, переиспользует `StatsSnapshot`/`ServerHealth`
(LLD-10/11), без нового формата:

```
RouterReport {
  version, uptime_secs,
  servers: Vec<ServerHealth>,   // state/class/rtt/role (LLD-11 §3.2)
  incident: Option<Incident>,
  bytes_up, bytes_down, connections,   // счётчики
  applied_preset_version,
  command_results: Vec<{ id, ok, detail }>,   // заполняется в LLD-20
}
```

Хаб хранит **последний** снимок на роутер (для админки). Опц. короткое кольцо
последних N как задел под пайплайн XR-026; **полная история/TSDB/Grafana это LLD-18**,
не здесь.

### 2.4 Админка: раздел «Роутеры»

Новая страница в Admin SPA (LLD-01):

- **Список:** имя, `online`/`offline` (по `last_seen` + grace), версия, активный
  сервер `Up`/`Down(class)` (LLD-11), live-скорость, «last seen N сек назад».
- **Карточка роутера:** слои статуса (как LLD-11), назначенный пресет. Кнопки
  действий и история команд это уже управление (LLD-20).
- **Deregister/Delete:** отзыв секрета -> последующий poll роутера отвергается 401.

### 2.5 Безопасность реестра

- **Per-router секрет**, не общий; на роутере chmod 600, на хабе хэш. Отзыв в
  админке.
- **Закреплённый `command_pubkey`** кладётся при enrollment (TOFU), чтобы команды
  LLD-20 было чем проверять; сам механизм команд и их подпись в LLD-20 §2.4.
- Rate-limit poll, валидация `router_id`; enrollment-токен одноразовый, с TTL.

## 3. Дизайн-решения

### 3.1 Pull, не push/inbound

Хаб за NAT до роутера не достучится, единственный рабочий вариант это исходящий
канал роутер -> хаб. Pull переиспользует существующий polling (PresetCache уже
ходит на хаб). Realtime как follow-up (LLD-20 §7).

### 3.2 Enrollment через одноразовый токен, а не admin-креды в приложении

Приложение/bootstrap не обязаны быть админом хаба. Переиспользуем механику
инвайта (LLD-01): токен -> `{router_id, secret}`. Один шов, знакомый паттерн.

### 3.3 Реестр в JSON-сторе, как пресеты/инвайты

Консистентно с LLD-01 (`tempfile` + atomic rename, in-memory `RwLock`). БД нужна при
росте флота (follow-up §7), сейчас оверкилл.

### 3.4 Отчёт переиспользует `StatsSnapshot`/`ServerHealth`

Не плодим второй формат метрик: то же, что Android читает через `nativeGetStats()`
(LLD-11 §3.2), едет на хаб.

## 5. Риски и edge-кейсы

1. **Утечка per-router секрета.** Уникален, chmod 600, хэш на хабе, отзыв в
   админке. Утечка одного не равна компрометации флота.
2. **Рост флота -> JSON-стор медленный.** Сейчас единицы роутеров, ок; БД как
   follow-up (§3.3, §7).
3. **Приватность отчёта.** В отчёте только операционные метрики (скорость,
   up/down, версия, инцидент), **без поведенческой телеметрии** (какие домены).
   Полноценная политика приватности в XR-026/LLD-18.

## 6. План проверки

1. **Enrollment.** Bootstrap/приложение с одноразовым токеном -> роутер в админке
   `online`, версия/арка видны; повтор токена -> 410.
2. **Статус.** Роутер репортит -> в админке скорость/активный сервер/last-seen
   обновляются; убить сервер (scoped-drop) -> класс сбоя виден (LLD-11).
3. **Offline.** Выключить роутер -> через grace в админке `offline`.
4. **Revoke.** Delete роутера -> его poll отвергается 401, исчезает из активных.
5. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Удалённое управление** (команды): отдельный [LLD-20](20-router-remote-management.md).
- **Realtime канал** (websocket/long-poll): pull достаточно для MVP, follow-up.
- **TSDB / Grafana / история метрик / алертинг**: [LLD-18](18-fleet-metrics-grafana.md) (XR-026).
- **БД вместо JSON**: при росте флота.
- **Реестр VPS-серверов** (не роутеров) через тот же механизм, возможно позже.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Pull или push/realtime?* -> **pull** (§3.1), realtime как follow-up.
- *Как enroll без admin-кред в приложении?* -> **одноразовый enrollment-токен**
  (механика инвайта LLD-01, §3.2).
- *Где хранить реестр?* -> **JSON-стор, как пресеты/инвайты** (§3.3).
- *Метрики/история/Grafana здесь?* -> **нет**, только last-snapshot для админки;
  fleet-наблюдаемость в LLD-18.
