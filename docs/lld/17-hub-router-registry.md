# LLD-17. Хаб-реестр роутеров и удалённое управление (pull control-plane)

**Статус:** Draft
**Область:** `xr-hub` (реестр роутеров, enrollment, очередь команд, приём
статус-отчётов, раздел «Роутеры» в Admin SPA), `xr-client` (per-router
идентичность, исходящий control-канал: poll → отчёт + приём/верификация/
применение команд), `xr-proto` (типы записи роутера, креденшела, команды,
отчёта), `xr-bootstrap` (enrollment во время установки, шов с LLD-13).
**Зависимости:** [LLD-01](01-control-plane.md): xr-hub, инвайты, ed25519,
Admin SPA, axum, JSON-стор: расширяем реестром и очередью команд.
[LLD-13](13-zero-touch-provisioning.md): bootstrap регистрирует роутер и пишет
per-router ключ в конфиг (это «шов» установка→управление).
[LLD-10](10-router-multi-vps-failover.md): `ServerPool`/`HealthState`, статус,
который роутер репортит наверх. [LLD-11](11-monitoring-health-panel.md):
классификация сбоя и `ServerHealth`, содержимое отчёта. [LLD-16](16-manual-server-hub-rules.md):
per-router назначение пресетов через хаб.
**Связанные документы:** [ARCHITECTURE.md §3 (хаб), §8 (наблюдаемость)](../ARCHITECTURE.md),
memory [[reference_deploy_handoff]], [[project_failopen_breaker]] (`is_server_down`
как источник статуса). Метрики/история/Grafana это отдельный
[LLD-18](18-fleet-metrics-grafana.md) (XR-026), здесь только транспорт отчёта и
его хранение «последним снимком».

Сейчас «роутер, зарегистрированный в сети хаба» и «удалённое управление»
невозможны: хаб видит всех клиентов **анонимно** (общий obfuscation-ключ,
публичные пресеты, никакой записи о роутере, LLD-01), клиент только **тянет**
пресеты, а хаб за NAT до роутера (серый IP) **не достучится** ни при установке,
ни в работе. Этот LLD добавляет (1) идентичность роутера и enrollment, (2)
**исходящий** control-канал роутер→хаб (pull), по которому роутер репортит статус
и забирает подписанные команды, (3) раздел «Роутеры» в админке. Закрывает
пользовательскую цель «получить рабочий роутер, зарегистрированный в сети хаба, в
котором обновляются правила и которым можно управлять».

---

## 1. Текущее состояние

- **Хаб не аутентифицирует клиентов** (LLD-01): общий obfuscation-ключ, пресеты
  публичны, нет записи роутера, `router_id`, `last_seen`, версии, статуса.
- **Клиент только pull'ит пресеты** ([xr-client/src/main.rs](../../xr-client/src/main.rs),
  `PresetCache` раз в `refresh_interval`). Хаб не знает, жив ли роутер, и не может
  ничего ему отправить.
- **NAT.** Роутер за серым IP; хаб (VPS) не может инициировать к нему соединение
  ни при установке, ни потом. Inbound невозможен в принципе.
- **Деплой/настройка руками по SSH** ([[reference_deploy_handoff]],
  `routers.md`); LLD-13 автоматизирует установку, но не даёт ни идентичности
  роутера на хабе, ни удалённого управления.
- Итог: «зарегистрирован в сети хаба» сегодня = «где-то стоит клиент с тем же
  ключом», без видимости и управляемости.

## 2. Целевое поведение

### 2.1 Идентичность и enrollment

Запись роутера на хабе:

```
RouterRecord {
  router_id,        // случайный, не переиспользуется
  name,             // человекочитаемое («роутер родителям»)
  secret_hash,      // per-router секрет (хранится хэшем, как admin-пароль LLD-01)
  enrolled_at, last_seen,
  client_version,   // версия xr-client (из отчёта)
  assigned_preset,  // какой пресет роутеру назначен (LLD-16)
  last_report,      // последний снимок статуса (§2.4), для админки
}
```

**Enrollment во время установки (шов с LLD-13).** Чтобы не класть admin-креды
хаба в приложение/bootstrap, переиспользуем механику инвайта (LLD-01):

1. Владелец в админке создаёт **одноразовый enrollment-токен** (TTL, как инвайт),
   либо приложение получает его тем же путём, что и инвайт.
2. `xr-bootstrap` (LLD-13) на LAN после установки клиента дёргает
   `POST /api/v1/enroll {token, name, arch, version}` → хаб создаёт
   `RouterRecord`, возвращает `{router_id, secret, command_pubkey}`, помечает
   токен `consumed`.
3. Bootstrap пишет `[control]` в `config.toml` роутера: `hub_url`, `router_id`,
   `secret` (chmod 600), pinned `command_pubkey` (TOFU, как public-key LLD-04).
4. Клиент стартует, открывает control-канал (§2.2) → роутер виден в реестре
   `online`.

### 2.2 Исходящий control-канал (pull)

`xr-client` периодически (`control_poll_interval`, default 30с) шлёт
`POST /api/v1/router/poll` с аутентификацией (`router_id` + `secret`, bearer/HMAC)
и телом = краткий отчёт (§2.4). Хаб:

- обновляет `last_seen` и сохраняет `last_report`;
- отвечает: версия назначенного пресета (роутер сам тянет полный пресет
  существующим путём LLD-01, если устарел) + список pending-команд (§2.3) +
  ack'и уже выполненного.

Pull выбран сознательно: NAT-дружелюбен (только исходящее), переиспользует
существующий polling-движок. Realtime (websocket/long-poll) вне скоупа MVP
(§7); цена pull это задержка команды ≤ `control_poll_interval`.

### 2.3 Команды (least-privilege, не произвольный shell)

**Закрытый enum** типов команд, не «выполни shell»:

| Команда | Действие |
|---|---|
| `apply_preset { name }` | назначить/сменить пресет (роутер тянет его, LLD-01/16) |
| `update_config { patch }` | мёрж в **белый список** полей: `block_quic`, `mux_pool_size`, `on_server_down`, `log_level`, набор `[[servers]]` (LLD-10) |
| `reload` / `restart` | перечитать конфиг / перезапустить процесс (через procd) |
| `deregister` | роутер стирает `[control]`, перестаёт поллить |

Каждая команда **подписана** ed25519-ключом хаба, который роутер проверяет
pinned-ключом (§2.1.3). Команда несёт `id` + идемпотентна; результат роутер
репортит в следующем poll (`acked`/`failed`). `update_client` (удалённое
обновление бинаря, аналог APK-self-update LLD-12 для роутера) это **вне скоупа MVP**
(§7), но enum и канал спроектированы под него.

### 2.4 Отчёт статуса (шов к XR-026)

Тело poll-запроса это компактный JSON, переиспользует `StatsSnapshot`/`ServerHealth`
(LLD-10/11), без нового формата:

```
RouterReport {
  version, uptime_secs,
  servers: Vec<ServerHealth>,   // state/class/rtt/role (LLD-11 §3.2)
  incident: Option<Incident>,
  bytes_up, bytes_down, connections,   // счётчики
  applied_preset_version,
  command_results: Vec<{ id, ok, detail }>,
}
```

Хаб хранит **последний** снимок на роутер (для админки). Опц. короткое кольцо
последних N как задел под пайплайн XR-026; **полная история/TSDB/Grafana это LLD-18**,
не здесь.

### 2.5 Админка: раздел «Роутеры»

Новая страница в Admin SPA (LLD-01):

- **Список:** имя, `online`/`offline` (по `last_seen` + grace), версия, активный
  сервер `Up`/`Down(class)` (LLD-11), live-скорость, «last seen N сек назад».
- **Карточка роутера:** слои статуса (как LLD-11), назначенный пресет, история
  команд (pending/acked/failed), кнопки действий (apply preset, reload, restart),
  каждая ставит **подписанную** команду в очередь.
- **Deregister/Delete:** отзыв секрета → последующий poll роутера отвергается 401.

### 2.6 Безопасность (центральный раздел)

Хаб получает право менять конфиг на роутерах в чужих квартирах это **самая
весомая по безопасности фича экосистемы**. Защита:

- **Per-router секрет**, не общий; на роутере chmod 600, на хабе хэш. Отзыв в
  админке.
- **Подписанные команды** ed25519, pinned-ключ на роутере (TOFU при enrollment,
  как public-key LLD-04). Тот же принцип, что отделённый release-ключ в LLD-12:
  **компрометация транспорта/VPS ≠ RCE на роутерах** без приватного ключа
  подписи команд.
- **Least-privilege:** закрытый enum (§2.3), нет произвольного shell/файлов,
  `update_config` только по белому списку полей.
- **Аудит-лог** команд на хабе (кто/что/когда поставил, когда acked).
- Rate-limit poll, валидация `router_id`; enrollment-токен одноразовый, с TTL.

## 3. Дизайн-решения

### 3.1 Pull, не push/inbound

Хаб за NAT до роутера не достучится, единственный рабочий вариант это исходящий
канал роутер→хаб. Pull переиспользует существующий polling (PresetCache уже ходит
на хаб), задержка команды ≤ интервал. Realtime как follow-up (§7).

### 3.2 Команды подписаны; ключ подписи в идеале офлайн

MVP: подписываем команды **пресет-ключом LLD-01** (он уже pinned у клиента),
меньше движущихся частей. Усиление: **отдельный офлайн command-ключ** (как
release-ключ LLD-12), чтобы взлом VPS не давал подписывать команды, follow-up
(§7). Trade-off зафиксирован осознанно.

### 3.3 Enrollment через одноразовый токен, а не admin-креды в приложении

Приложение/bootstrap не обязаны быть админом хаба. Переиспользуем механику
инвайта (LLD-01): токен → `{router_id, secret}`. Один шов, знакомый паттерн.

### 3.4 Закрытый enum команд, не shell

Минимизируем поверхность RCE: хаб может только то, что явно перечислено и
покрыто белым списком, а не «выполни произвольную команду».

### 3.5 Реестр в JSON-сторе, как пресеты/инвайты

Консистентно с LLD-01 (`tempfile` + atomic rename, in-memory `RwLock`). БД нужна при
росте флота (follow-up §7), сейчас оверкилл.

### 3.6 Отчёт переиспользует `StatsSnapshot`/`ServerHealth`

Не плодим второй формат метрик: то же, что Android читает через `nativeGetStats()`
(LLD-11 §3.2), едет на хаб.

- `test_command_config_whitelist` (`update_config` с полем вне белого списка → reject).
- `test_report_roundtrip` (сериализация/десериализация `RouterReport`).

## 5. Риски и edge-кейсы

1. **Компрометация хаба → RCE на домашних роутерах.** Главный риск. Митигация:
   подписанные команды (pinned-ключ §2.6), закрытый enum, least-privilege; цель это
   офлайн command-ключ (§3.2, follow-up).
2. **Утечка per-router секрета.** Уникален, chmod 600, хэш на хабе, отзыв в
   админке. Утечка одного ≠ компрометация флота.
3. **Replay команд.** `id` + ack только после успеха + идемпотентный apply; опц.
   nonce/timestamp в подписи.
4. **Задержка команд из-за pull.** Команда применяется ≤ `control_poll_interval`.
   Для «срочно выключить обходы» это медленно, принимаем как лимит pull-MVP;
   `on_server_down=direct` и так fail-open ([[project_failopen_breaker]]).
5. **Полу-применённая команда (обрыв питания/SSH).** Идемпотентность + ack после
   успеха; повторный poll доводит. Конфиг мёржится, не перезатирается вслепую.
6. **Рост флота → JSON-стор медленный.** Сейчас единицы роутеров, ок; БД как
   follow-up (§3.5, §7).
7. **Приватность отчёта.** В отчёте только операционные метрики (скорость,
   up/down, версия, инцидент), **без поведенческой телеметрии** (какие домены).
   Полноценная политика приватности в XR-026/LLD-18.
8. **Рассинхрон pinned command-ключа при ротации.** Ротация ключа подписи требует
   повторного pin на роутерах (через подписанную старым ключом команду или
   re-enroll); описать процедуру при ротации.

## 6. План проверки

Rust (подпись/whitelist/offline) автотестами; админка и клиент-апплай вручную.

1. **Enrollment.** Bootstrap/приложение с одноразовым токеном → роутер в админке
   `online`, версия/арка видны; повтор токена → 410.
2. **Статус.** Роутер репортит → в админке скорость/активный сервер/last-seen
   обновляются; убить сервер (scoped-drop) → класс сбоя виден (LLD-11).
3. **Команда.** `apply_preset` из админки → роутер применяет ≤ poll-interval,
   `acked` в карточке; трафик идёт по новому пресету.
4. **Подпись.** Подменить тело команды / подписать чужим ключом → роутер
   отвергает, пишет в лог; конфиг не меняется.
5. **Least-privilege.** `update_config` с полем вне белого списка → reject.
6. **Offline.** Выключить роутер → через grace в админке `offline`.
7. **Revoke.** Delete роутера → его poll отвергается 401, исчезает из активных.
8. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Realtime канал** (websocket/long-poll): pull достаточно для MVP, follow-up.
- **`update_client`** (удалённое обновление бинаря xr-client, аналог LLD-12 для
  роутера): enum/канал готовы под него, сама раскатка бинаря по арке отдельно.
- **TSDB / Grafana / история метрик / алертинг**: [LLD-18](18-fleet-metrics-grafana.md) (XR-026).
- **БД вместо JSON**: при росте флота.
- **Отдельный офлайн command-ключ**: усиление поверх MVP (§3.2); сначала
  переиспользуем пресет-ключ LLD-01.
- **Реестр VPS-серверов** (не роутеров) через тот же механизм, возможно позже.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Pull или push/realtime?* → **pull** (§3.1), realtime как follow-up.
- *Как enroll без admin-кред в приложении?* → **одноразовый enrollment-токен**
  (механика инвайта LLD-01, §3.3).
- *Произвольный shell или закрытый enum?* → **закрытый enum + белый список полей**
  (§3.4, §2.3).
- *Чем подписывать команды?* → **пресет-ключом LLD-01 в MVP**, офлайн
  command-ключ как follow-up (§3.2).
- *Где хранить реестр?* → **JSON-стор, как пресеты/инвайты** (§3.5).
- *Метрики/история/Grafana здесь?* → **нет**, только last-snapshot для админки;
  fleet-наблюдаемость в LLD-18.
