# LLD-18. Fleet-метрики и Grafana-дашборды

**Статус:** Draft (follow-up)
**Область:** `xr-hub` (накопление кольца метрик из poll-отчётов + экспозиция
Prometheus-формата), деплой (VictoriaMetrics + Grafana на VPS, provisioned
дашборды), опц. `xr-android` (sparse opt-in операционная телеметрия).
**Зависимости:** [LLD-17](17-hub-router-registry.md) (XR-025): роутеры уже
**пушат** компактный `RouterReport` на хаб по исходящему control-каналу: транспорт
данных готов, этот LLD лишь **накапливает и визуализирует** их.
[LLD-11](11-monitoring-health-panel.md): классификация сбоя/`ServerHealth`
(содержимое метрик). [LLD-01](01-control-plane.md): xr-hub.
**Связанные документы:** [LLD-11 §7](11-monitoring-health-panel.md) (вынес
«агрегированный мониторинг флота / историю аптайма на хабе» как follow-up, это
он), [ARCHITECTURE.md §8](../ARCHITECTURE.md), `docs/HUB-DEPLOY.md`.

LLD-17 даёт в админке **мгновенный** снимок каждого роутера. Нет **истории,
трендов, графиков и алертинга** по флоту. LLD-11 это наблюдаемость on-device для
одного владельца, не флот. Этот LLD добавляет time-series поверх того, что роутеры
(и опц. приложения) уже шлют хабу: дашборды скорости/аптайма/инцидентов и алерты
«роутер offline / инцидент длится / сервер down». **«Просто графана» это лёгкая
часть (визуализация); тяжёлая часть, пайплайн за NAT, закрыта в LLD-17.**

---

## 1. Текущее состояние

- **LLD-17** хранит `last_report` на роутер → админка показывает текущее
  состояние, но не историю.
- **LLD-11**: локальные слои индикатора + уведомления на устройстве владельца,
  без агрегации по нескольким роутерам/устройствам.
- Time-series/графиков/алертинга нет нигде. NAT не даёт скрейпить роутеры напрямую
  (это и решено в LLD-17 через push роутер→хаб).

## 2. Целевое поведение

### 2.1 Хаб как единственный экспортёр

Хаб накапливает кольцо метрик per-router из `RouterReport` (LLD-17 §2.4) и
экспонирует их в **Prometheus-формате**: `GET /api/v1/metrics` (под токеном/в
приватной сети). Роутеры **не** скрейпятся напрямую, они за NAT и уже пушат хабу;
скрейпится один публично достижимый хаб.

### 2.2 Стек на VPS

- **VictoriaMetrics** (single-binary, лёгкая по RAM альтернатива Prometheus)
  скрейпит `/api/v1/metrics` хаба.
- **Grafana** с provisioned-дашбордами (JSON в репозитории):
  - per-router: скорость up/down, RTT, uptime, инциденты по времени;
  - сводка по флоту: сколько online, агрегированная скорость, аптайм-хитмап;
  - классы сбоя (`ServerUnreachable`/`HandshakeReset`/`AuthFailed`, LLD-11) во
    времени.
- Опц. алертинг (vmalert/Grafana): роутер `offline` > N мин, инцидент > N мин,
  все серверы down.

### 2.3 Метки и кардинальность

Метки: `router_id`/`name`, `server`, `class`. Без PII, без доменов/destination.
Флот это единицы-десятки роутеров → кардинальность низкая, retention базовый.

### 2.4 Приложения (опц., по умолчанию выкл.)

Android может слать **sparse opt-in** операционную телеметрию (версия, аптайм
сессии, класс сбоя, агрегаты), не поведенческую. По умолчанию **off**; включает
владелец. Это отдельный осознанный подпункт, не обязательный для MVP дашбордов
(роутеры дают основную картину).

## 3. Дизайн-решения

### 3.1 Хаб как единственная точка экспозиции

Роутеры за NAT не скрейпятся; они и так пушат хабу (LLD-17). Значит VictoriaMetrics
скрейпит **один** хаб, минимум движущихся частей, никакого push-gateway и агентов
на роутерах.

### 3.2 VictoriaMetrics, не полный Prometheus

Single-binary, мало RAM, дешёвый self-host. Под масштаб «домашний флот» полный
Prometheus избыточен. Совместим с Prometheus-форматом и PromQL/Grafana.

### 3.3 Приватность как явный раздел, не побочка

Это **антицензурный инструмент**. Централизованно собираем только **операционные**
метрики (скорость, up/down, версия, инциденты), **никогда** не поведенческие
(какие домены/адреса посещаются). App-телеметрия opt-in, по умолчанию выключена,
агрегирована. Не встраиваем слежку в инструмент обхода слежки.

### 3.4 Дашборды в репозитории, provisioned

JSON-дашборды и datasource лежат в `deploy/`, разворачиваются вместе со стеком
(Grafana provisioning), а не кликаются руками, что воспроизводимо и под review.

## 4. Изменения в коде

| Файл | Что меняется |
|---|---|
| `xr-hub/src/...` | Кольцо последних N `RouterReport` per-router (в памяти/на диске); `GET /api/v1/metrics`, Prometheus-экспозиция (gauge скорость/RTT/connections, counter bytes, up/down, инцидент-флаги) с метками `router_id`/`name`/`server`/`class`. Auth на эндпоинт. |
| `deploy/observability/` (новое) | compose/systemd для VictoriaMetrics + Grafana; scrape-конфиг на хаб; provisioned datasource + дашборды (JSON); опц. vmalert-правила. |

| `xr-android/...` (опц.) | Sparse opt-in телеметрия в хаб (по умолчанию off), отдельный подпункт. |

Тесты:
- `xr-hub`: `test_metrics_exposition_format` (валидный Prometheus-формат из
  `RouterReport`), `test_metrics_labels_no_pii` (нет доменов/PII в метках).
- Стек VM+Grafana проверяется вручную (§6).

## 5. Риски и edge-кейсы

1. **Публичная экспозиция метрик.** `/api/v1/metrics` под токеном/TLS или скрейп
   только из приватной сети VPS; не отдаём флот-телеметрию миру.
2. **Приватность.** См. §3.3: только операционные метрики; app-телеметрия opt-in.
3. **Ops-вес.** Ещё два сервиса на VPS (VM + Grafana), взвесить против масштаба;
   для единиц роутеров админ-вид LLD-17 уже закрывает ~80%, стек включаем, когда
   нужны история/тренды/алерты.
4. **Кардинальность/диск.** Низкая при малом флоте; retention базовый, чистка
   старого.
5. **Дыра при оффлайне хаба.** Метрики копятся в кольце ограниченно; при долгом
   простое будет провал в графиках (приемлемо, это не биллинг).

## 6. План проверки

1. Поднять VictoriaMetrics + Grafana по `deploy/observability/`; datasource и
   дашборды подхватились provisioning'ом.
2. Хаб с парой роутеров (LLD-17) → дашборд показывает live-скорость/uptime по
   каждому.
3. Убить активный сервер роутера → на графике инцидент + класс сбоя (LLD-11).
4. Роутер offline > порога → алерт срабатывает.
5. `GET /api/v1/metrics` без токена → отказ; метки без доменов/PII.
6. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Поведенческая аналитика приложений** (домены, destination): принципиально нет
  (§3.3).
- **Managed-cloud Grafana / внешние SaaS**: self-host на своём VPS.
- **Долгое хранение / downsampling**: базовый retention; тюнинг позже.
- **Биллинг/SLA-отчёты** это операционная наблюдаемость, не учёт.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Scrape хаба или remote-write от агентов?* → **scrape одного хаба** (он публичен,
  роутеры за NAT уже пушат ему; §3.1).
- *VictoriaMetrics или Prometheus?* → **VictoriaMetrics** (single-binary, лёгкий;
  §3.2).
- *Метрики приложений?* → **opt-in, по умолчанию off, только операционные** (§2.4,
  §3.3).
- *Делать ли стек сейчас?* → **транспорт в LLD-17; стек и дашборды это данный LLD как
  follow-up**, включается когда нужны история/тренды/алерты.
