# LLD-16 — Правила из хаба для серверов, добавленных вручную

**Статус:** Draft
**Область:** `xr-android` (карточка сервера: выбор источника правил + список
пресетов с хаба, TOFU публичного ключа), `xr-android-jni` (проброс списка
пресетов / ключа), опционально `xr-core` (верификация подписи пресета —
рекомендованное усиление). Движок фонового обновления пресета —
**переиспользуется без изменений**.
**Зависимости:** [LLD-01](01-control-plane.md) — хаб, именованные пресеты,
эндпоинты `/api/v1/presets`, `/api/v1/presets/:name`, `/api/v1/public-key`,
ed25519-подпись пресетов. [LLD-04](04-onboarding-qr-uri.md) — TOFU-модель
доверенного ключа хаба (`trusted_public_key`), reqwest-клиент онбординга.
[LLD-08](08-android-multi-server.md) — `ServerProfile`/`ServerRepository`
(поля `hubUrl`, `hubPreset`, `trustedPublicKey`, `routingPreset`).
[LLD-12](12-android-apk-self-update.md) — редактируемое поле «Адрес хаба» в
карточке сервера (точка входа этой фичи) и ed25519-verify в `xr-core`.
**Связанные документы:** [ARCHITECTURE.md §5.3 (control plane)](../ARCHITECTURE.md),
[ARCHITECTURE.md §4.x (preset cache)](../ARCHITECTURE.md).

Сейчас «живые», централизованно управляемые правила маршрутизации с хаба
получают только серверы, заведённые **по инвайту**. Сервер, добавленный
**вручную**, заморожен на локальном пресете (зашитый `PRESET_RUSSIA` /
`proxy_all` / ручной Custom-список) и не подхватывает изменения с хаба. Этот
LLD делает ручной сервер полноценным: позволяет привязать его к пресету на
хабе и получать правила так же, как инвайт-сервер.

---

## 1. Текущее состояние

- Хаб (LLD-01) хранит именованные пресеты (наборы правил маршрутизации),
  отдаёт `GET /api/v1/presets` (список `PresetSummary` с версиями) и
  `GET /api/v1/presets/:name` (полный `Preset`, поле `signature`).
- Движок (`xr-core`: `engine.rs` + `presets.rs::PresetCache`) при заданных
  `hub_url` + `hub_preset` (+ `hub_cache_dir`) грузит пресет с диска и фоном
  раз в `hub_refresh_interval_secs` (default 300) дотягивает новую версию
  (`fetch_if_stale`), **горячо подменяя** роутер. Включается, только если
  заданы **И** `hub_url`, **И** `hub_preset` (см. сборку конфига в
  `VpnViewModel`, ветка `hubFields`).
- `ServerProfile` (LLD-08) уже несёт поля `hubUrl`, `hubPreset`,
  `trustedPublicKey`, `routingPreset`/`customDomains`/`customIpRanges` —
  **миграция схемы не нужна**.
- Инвайт-онбординг (LLD-04) при Apply заполняет `hubUrl` + `hubPreset` (и
  TOFU-ит `trustedPublicKey`) → инвайт-серверы получают живые правила.
- Ручной сервер (`ServerEditScreen`, `source = Manual`): `hubPreset` пуст →
  правила берутся из локального `routingPreset`. С LLD-12 `hubUrl`
  редактируем, но у ручного сервера влияет **только** на self-update
  (правил-рефреш не включается без `hubPreset`).
- **Факт по коду (важно для §3.4):** `PresetCache::fetch_if_stale`
  **не проверяет** подпись пресета — поле `signature` есть, `trustedPublicKey`
  хранится в профиле, но в верификации **не участвует**. Доверие к правилам
  сейчас фактически на уровне TLS, и это так **у всех** серверов, включая
  инвайт. → Фича не вводит нового разрыва доверия относительно сегодняшнего
  поведения.

## 2. Целевое поведение

- В карточке сервера (`ServerEditScreen`), когда задан адрес хаба, появляется
  переключатель **«Источник правил»**: `Локальный пресет` (как сейчас) либо
  `Пресет с хаба`.
- При `Пресет с хаба`:
  - показывается выпадающий список пресетов, полученный с хаба
    (`GET /api/v1/presets` → имена + версии); выбор пишет `hubPreset`;
  - локальные чипы `Russia / Proxy all / Custom` скрываются/деактивируются
    (источник правил — хаб, а не они);
  - если список не дотянулся — допускается ручной ввод имени пресета (fallback).
- После сохранения существующий движок тянет правила с хаба и обновляет их
  фоном — у ручного сервера так же, как у инвайт-сервера.
- Источник правил явно виден и в карточке списка серверов (бейдж
  «правила: хаб · `<preset>`» vs «правила: локально»).
- Отказоустойчивость: хаб недоступен → работаем на последнем кэше; пресет на
  хабе отсутствует/переименован → остаёмся на кэше + понятное сообщение, без
  падения (движок уже так делает в `fetch_if_stale`).

## 3. Дизайн-решения

### 3.1 Движок переиспользуем без изменений
Вся механика (диск-кэш, фоновый `fetch_if_stale`, горячая подмена роутера,
fallback на кэш) уже работает для инвайт-серверов. Нужен только UI, который
заполнит `hubPreset` у ручного сервера. Бэкенд маршрутизации **не трогаем**.

### 3.2 Явный выбор источника правил, не угадывание
Переключатель `Локальный / Хаб` — не выводить источник неявно из «заполнен ли
`hubPreset`». Когда выбран хаб-пресет, он имеет приоритет, а локальные поля
прячутся — чтобы у пользователя не было сомнений, какие правила реально
применяются. Локальный Custom-список при переключении на «Хаб»
**не затираем** в профиле (чтобы можно было вернуться на «Локальный»).

### 3.3 Список пресетов — с хаба, с ручным fallback
Тянем `GET /api/v1/presets` (переиспользуем reqwest-клиент онбординга;
добавляем `xr_core::onboarding::list_presets` и JNI `nativeListPresets(hubUrl,
timeoutMs) -> JSON` по образцу `nativeFetchInviteInfo`). Если список не
получен — поле ручного ввода имени пресета.

### 3.4 Доверие к пресету у ручного сервера (ключевое решение)
Граница доверия: правила ≠ код. Худший случай подмены правил — трафик пошёл
`direct` вместо `proxy` (приватность/доступность), **не RCE**. Поэтому:
- **MVP:** паритет с инвайт-серверами — правила тянутся по TLS. Дополнительно
  при привязке к хабу **TOFU-им** публичный ключ (`GET /api/v1/public-key` →
  `trustedPublicKey`), тихо, как при онбординге; отпечаток ключа показываем в
  карточке для прозрачности.
- **Рекомендованное усиление (в этом же LLD или отдельной задачей):** реально
  **проверять подпись пресета** в `PresetCache::fetch_if_stale` переданным
  `trusted_public_key` (тот же ed25519-verify, что в `xr-core::update`,
  LLD-12). Закрывает «скомпрометированный/MITM-нутый хаб подменил правила».
  Это **единственное** место, где может потребоваться правка `xr-core`, и оно
  заодно усиливает инвайт-серверы (которые сегодня тоже не верифицируют). Если
  хаб отдаёт неподписанные пресеты — помечаем «правила без подписи».

### 3.5 TLS по голому IP
Если адрес хаба — IP без домена, TLS-сертификат (выписанный на домен) не
сойдётся. Рекомендуем домен хаба; та же грабля, что в LLD-12 §5 (по IP может
потребоваться домен/корректный сертификат).

## 5. Риски и edge-кейсы
1. **Неоднозначность источника правил** → §3.2 явный селектор + бейдж.
2. **Хаб недоступен / пресет удалён/переименован** → остаёмся на кэше, не
   падаем (поведение `fetch_if_stale` уже такое); показать сообщение.
3. **Подпись пресета** — сегодня не верифицируется ни у кого (не регресс), но
   §3.4 усиление желательно; учесть ротацию ключа (смена `trustedPublicKey`).
4. **TLS по IP** (§3.5) — рекомендовать домен.
5. **Потеря локального Custom-списка** при переключении на «Хаб» — не затирать
   `customDomains`/`customIpRanges` в профиле.
6. **Версия пресета ≤ локальной** → не обновляем (уже учтено в `fetch_if_stale`).

## 6. План проверки
1. Ручной сервер → «Источник правил: Хаб» → выбрать пресет из списка →
   Connect → маршрутизация соответствует пресету хаба.
2. На хабе изменить пресет (bump version) → за ≤5 мин правила на устройстве
   обновились **без** переустановки/ручной правки.
3. Хаб недоступен → работаем на последнем кэше, без падения.
4. Переключение Хаб → Локальный → Хаб сохраняет оба набора правил.
5. (Если усиление) подделать/испортить подпись пресета → отвергается,
   остаёмся на кэше, WARN в лог.
6. `cargo test --workspace` зелёный, 0 warnings (если трогаем `xr-core`).

## 7. Вне скоупа
- Полноценный **редактор** правил на клиенте (LLD-05) и на хабе (LLD-14) — тут
  только **потребление** готового пресета ручным сервером.
- Несколько пресетов на сервер / композиция правил — один пресет на профиль.
- Серверная модель ключей/подписи — как в LLD-01.
- Превращение этого в полный pull всей конфигурации (адрес/ключ/salt) с хаба
  для ручного сервера — это уже онбординг (LLD-04), не данный LLD.

## 8. Открытые вопросы (закрыты в этом дизайне)
- *Как выбирать источник правил?* → явный переключатель Локальный/Хаб (§3.2),
  при «Хаб» локальные чипы прячутся.
- *Откуда брать список пресетов?* → `GET /api/v1/presets` + новый JNI (§3.3);
  ручной ввод имени как fallback.
- *Доверие к пресету у ручного сервера?* → MVP: паритет с инвайтом (TLS) +
  TOFU `public-key`; **рекомендуется** включить реальную верификацию подписи
  пресета по TOFU-ключу (§3.4) — единственное касание `xr-core`, усиливает и
  инвайт-серверы.
