# LLD-10: Multi-VPS failover клиента (роутер + Android)

**Статус:** Draft (расширен: изначально был только про роутер, теперь покрывает
любого клиента общего движка)
**Область:** `xr-proto` (новый `ServerPool` поверх нескольких `MuxPool`,
расширение `config.rs` и `preset.rs`), `xr-client` (список серверов из конфига,
проброс пула в `ProxyState`), `xr-core` + `xr-android-jni` (пул в
`SessionContext`, список серверов в JNI-конфиге), `xr-android` (список адресов
внутри профиля, миграция, редактор, индикация резерва), `xr-hub` (список
серверов в инвайт-payload и дефолтах хаба).
**Зависимости:** [LLD-09](09-multi-mux-pool.md) (Implemented): `MuxPool` с N
слотами, round-robin, failover **внутри одного сервера**; обобщаем до пула
*серверов*. [LLD-08](08-android-multi-server.md) (Implemented): `ServerProfile`
и вкладка Servers; пул встраивается внутрь профиля. [LLD-04](04-onboarding-qr-uri.md):
инвайт-payload, который расширяется списком серверов. Circuit-breaker
`is_server_down()` / `MuxPool::recycle()` из C1/C4 (`xr-proto/src/mux_pool.rs`):
готовая «память о падении» и пересоздание пула.
**Связанные документы:** [ARCHITECTURE.md §3 (mux)](../ARCHITECTURE.md),
[ARCHITECTURE.md §6 (конфиг)](../ARCHITECTURE.md), XR-068 (энергоэффективность
мобильного клиента), `local-docs/routers.md` (два VPS: Aeza + Timeweb).

Один VPS это единая точка отказа для **всех** клиентов, не только роутера.
Инцидент с неоплаченным хостингом (задача 1a) показал это на практике: пока
сервер был мёртв, проксируемый трафик деградировал до тех пор, пока breaker (C1)
не уводил его в Direct, а резервного сервера, на который можно переехать, не
было. Этот LLD добавляет в общий движок пул из >=2 серверов с автоматическим
failover'ом и включает его на обоих клиентах: падение активного сервера
переводит проксируемый трафик на резервный за секунды, а не в Direct. Роутер
берёт список из конфига, Android держит его внутри профиля и получает по
подписанному инвайту.

---

## 1. Текущее состояние

- Конфиг роутера описывает **ровно один** сервер
  ([config.rs](../../xr-proto/src/config.rs), `[server] address/port`,
  `routers.md §5`). Резервный VPS существует физически (Aeza + Timeweb), но
  переключение между ними ручное: `sed -i s/IP1/IP2/ config.toml` + restart
  (`routers.md §3.2`).
- На Android то же самое на уровне профиля: `ServerProfile` хранит один
  `serverAddress`/`serverPort`
  ([ServerProfile.kt:8](../../xr-android/app/src/main/java/com/xrproxy/app/data/ServerProfile.kt#L8)),
  JNI-конфиг принимает одно поле `server_address`
  ([lib.rs:206](../../xr-android-jni/src/lib.rs#L206)), `SessionContext` держит
  один `server_addr` ([session.rs:59](../../xr-core/src/session.rs#L59)).
  Мультисервер LLD-08 это список *профилей* (независимых подключений), а не пул
  резервов: активен ровно один, автоматики между ними нет.
- Инвайт-payload несёт один адрес: `InvitePayload { server_address, server_port,
  ... }` ([preset.rs:62](../../xr-proto/src/preset.rs#L62)).
- `MuxPool` (LLD-09) держит N (default 4) параллельных TCP-туннелей **к одному
  серверу**, балансирует стримы round-robin, failover'ит между слотами и
  переподнимает мёртвый слот. Это failover *внутри* сервера, не *между*
  серверами.
- Circuit-breaker (C1): после полного провала обхода слотов `MuxPool` помечает
  сервер `down` на cooldown, `open_stream` мгновенно отдаёт `Err`, и
  `on_server_down="direct"` уводит соединение в Direct. `is_server_down()` это
  публичный хук состояния; `recycle()` (C4) пересоздаёт пул и снимает breaker.
- Итог: при мёртвом активном VPS трафик идёт **Direct** (часть заблокированных
  ресурсов не откроется), хотя живой резервный сервер мог бы его принять. Это
  одинаково больно на роутере и на телефоне.

## 2. Целевое поведение

### 2.1 Список серверов в конфиге роутера

`[server]` (одиночный) обобщается до упорядоченного `[[servers]]` с приоритетом:

```toml
[[servers]]
name = "aeza"               # человекочитаемый лейбл для логов
address = "85.192.38.29"
port = 8443
priority = 0                # меньше = выше приоритет (0 = primary)

[[servers]]
name = "timeweb"
address = "201.51.16.159"
port = 8443
priority = 1                # backup
# key = "..."               # опционально: override общего [obfuscation].key,
                            # если у резерва другой ключ; по умолчанию общий
```

- Обфускация по умолчанию **общая** (`[obfuscation]` как сейчас), оба наших VPS
  делят ключ. Per-server `key`/`salt`/`modifier` это опциональный override на
  будущее (разные ключи на разных провайдерах), в MVP не обязателен.
- `mux_pool_size` (LLD-09) остаётся **глобальным**: каждый сервер поднимает свой
  `MuxPool` этого размера.
- **Обратная совместимость:** старый `[server]` парсится как `[[servers]]` из
  одного элемента с `priority=0`. Конфиги обоих боевых роутеров продолжают
  работать без правки до миграции.

### 2.2 Модель пула серверов

```
+----------------------------------------------------------+
| ServerPool                                               |
|   active: AtomicUsize                                    |
|   +--------------------------------+  +----------------+ |
|   | ServerSlot #0 (priority 0)     |  | ServerSlot #1  | |
|   |   MuxPool(N) + breaker         |  |   MuxPool(N)   | |
|   |   health: Up / Down(since,cls) |  |   health: ...  | |
|   +--------------------------------+  +----------------+ |
+----------------------------------------------------------+
      open_stream -> MuxPool активного сервера
```

- `ServerPool` держит по одному `MuxPool` на сервер (прямое обобщение LLD-09:
  LLD-09 раскладывал стримы по N TCP одного сервера, LLD-10 раскладывает по M
  серверам, у каждого свои N TCP).
- `open_stream(target)` идёт в `MuxPool` **активного** сервера. Если его breaker
  взведён (`is_server_down()`), пул выбирает следующий по приоритету **здоровый**
  сервер, делает его активным и идёт в него. Если все серверы down, возвращает
  `Err`, и `on_server_down="direct"` уводит соединение в Direct (как сейчас, но
  теперь это происходит только когда исчерпан **весь** пул).

### 2.3 Политика: primary/backup, не балансировка

**Решение открытого вопроса (problems.md 1b):** серверы **не равноправны**,
действует строгий приоритетный порядок (primary/backup), а не балансировка
нагрузки. Причины:

- Балансировка по серверам осмысленна только при сопоставимой ёмкости и цене
  трафика; у нас два разных VPS разных провайдеров, цель это
  *отказоустойчивость*, а не суммарная полоса.
- Sticky-to-primary даёт предсказуемый «домашний» сервер: меньше прыжков
  fingerprint'а между IP, проще диагностика по логам.
- Балансировку внутри одного провайдера уже даёт `MuxPool` (N TCP). Пул серверов
  это слой *резервирования*, а не *масштабирования*.

**Инвариант выбора активного:** активен сервер с наименьшим `priority` среди
здоровых. Падение primary делает активным следующий здоровый.

### 2.4 Health-check: пассивный детект + активный пробинг

Два механизма, дополняющие друг друга:

- **Пассивный (основной триггер failover'а)**: breaker из C1. Любой реальный
  стрим, упавший в `is_server_down()`, мгновенно переключает активный сервер на
  следующий здоровый. Бесплатно, без лишнего трафика, реагирует ровно на ту
  деградацию, что видит пользователь.
- **Активный (для возврата и pre-warm резерва)**: лёгкий фоновый пробер
  (`ServerPool::health_loop`), раз в `health_interval_secs` (default 15с)
  проверяющий серверы **выше по приоритету активного** и сам активный. Проба
  переиспользует probe-lease механику breaker'а (C1): один прободатель ставит
  mux-handshake. Зачем активная проба, если есть пассивная: пассивная узнаёт о
  *восстановлении* primary только когда туда случайно пойдёт стрим, а при
  sticky-to-primary стримы туда не идут, пока он «down». Активный пробер
  закрывает этот провал.

Это базовая (роутерная) политика. Мобильная отличается, см. §2.7.

### 2.5 Возврат на primary (failback) с hold-down

После восстановления primary активный трафик надо вернуть на него, но **без
флаппинга** на нестабильной связи:

- Прободатель видит primary здоровым и запускает таймер `failback_hold_secs`
  (default 60с).
- Если primary остаётся здоровым непрерывно весь hold-down, `ServerPool`
  переключает `active` обратно на него (failback). Если за время hold-down
  primary снова мигнул в down, таймер сбрасывается.
- Failback атомарен на уровне `active` индекса; живые стримы на backup
  дорабатывают (их `MuxStream` не рвём), новые стримы идут на primary. Старые
  mux backup'а лениво схлопываются по `MUX_MAX_LIFETIME`.

### 2.6 Android: пул внутри профиля

**Решение владельца:** failover-пул на Android живёт **внутри** `ServerProfile`,
между профилями автоматики нет. Профиль это хаб-аккаунт (XR-058): свой ключ,
свои шары, свои правила; молчаливый прыжок между аккаунтами смешал бы
identity-контексты. Наши Aeza + Timeweb в этой модели один профиль с двумя
адресами.

- `ServerProfile` получает список эндпоинтов
  `endpoints: List<ProfileEndpoint(name, address, port, priority)>`. Легаси-поля
  `serverAddress`/`serverPort` мигрируют в `endpoints[0]` (priority 0) при
  первом чтении prefs; ключ/`modifier`/`salt` остаются полями профиля, общими
  на все эндпоинты (per-endpoint ключи в UI не тащим, роутерный TOML-override
  покрывает этот кейс на будущее).
- **Edit профиля** (LLD-08 `ServerEditScreen`): секция «Серверы» со списком
  адресов, добавление/удаление/порядок (порядок в списке и есть приоритет).
  Минимум один адрес, как валидация `[[servers]]` на роутере.
- Движок: JNI-конфиг вместо одного `server_address` принимает массив `servers`
  (легаси-поле продолжает работать), `SessionContext` держит `Arc<ServerPool>`
  вместо `server_addr` + `mux_pool`. Failover/failback внутри движка, без
  пересоздания VPN-сервиса и рестарта туннеля.
- **Индикация, минимальная в этом LLD:** строка статуса на главном экране
  «через `<name>` (резерв)», когда активен не-primary, плюс записи в лог при
  failover/failback. Полная панель здоровья, классификация сбоев и уведомления
  остаются в LLD-11.

### 2.7 Пробы и энергия на мобильном

Роутерную политику (§2.4: пробер каждые 15с + тёплый warmup всех серверов) на
телефон переносить нельзя: это регулярные пробуждения радио в простое, ровно то,
с чем борется XR-068. Сравнение вариантов:

| | Как на роутере | Экономная (выбрана) |
|---|---|---|
| Активный пробер | всегда, каждые 15с | только пока активен не-primary (нужен для failback) или всё down; интервал 60с |
| Backup-пул | тёплый (N TCP постоянно) | холодный: mux поднимается при failover |
| Цена failover | ~0 (пул готов) | +1 RTT mux-handshake (сотни мс) |
| Радио в простое | лишние промоушены каждые 15с | ноль сверх обычного keepalive |
| ESTAB на VPS | N TCP с каждого телефона на **каждый** сервер | только на активный |

Секундная задержка первого стрима при failover это редкое событие (падение
сервера), а пробуждения радио это каждый день у каждого пользователя, поэтому
**выбрана экономная**. В здоровом состоянии (active == primary) телефон не
пробит и не греет backup вообще; пробер живёт только в деградированном
состоянии, где он и нужен для failback, и его тик по возможности коалесцируется
с mux keepalive (XR-068). Реализация: `ServerPool` принимает
`PoolProfile { warm_backups: bool, probe_when_healthy: bool, probe_interval }`;
роутер конфигурирует `{true, true, 15s}`, Android `{false, false, 60s}`.

### 2.8 Раздача списка серверов: инвайт и пресет хаба

**Решение владельца:** Android получает список не только руками, но и по
инвайту.

- `InvitePayload` дополняется полем `servers: Vec<PayloadServer { name, address,
  port, priority }>` (serde-default пустой). Легаси-поля
  `server_address`/`server_port` остаются и продолжают заполняться primary'м:
  старое приложение незнакомое поле игнорирует и работает как раньше, новое при
  пустом `servers` строит пул из одного легаси-адреса. Подпись ed25519 и TOFU
  (LLD-04) покрывают payload целиком, так что список приезжает
  аутентифицированным.
- Хаб: одиночные `defaults.server_address/port` в конфиге хаба обобщаются до
  списка (та же схема `[[servers]]`, что на роутере); `create`/`claim` инвайта
  кладут его в payload. Admin SPA не трогаем, список задаётся конфигом хаба.
- Apply инвайта на Android (LLD-04 §3.6) переносит `servers` в
  `ServerProfile.endpoints`.
- Список фиксируется в момент применения инвайта, дальше пользователь правит его
  руками. **Живое** обновление списка с хаба (hot-swap через рефреш пресета)
  сознательно не здесь: это follow-up поверх LLD-13 (provisioning добавляет
  свежеподнятый сервер) и LLD-16 (рефреш с хаба), см. §7.
- Роутер инвайты не потребляет, его источник остаётся конфиг (§2.1).

## 3. Дизайн-решения

### 3.1 ServerPool как обёртка над MuxPool, а не переписывание

`MuxPool` уже умеет всё нужное на уровне одного сервера (слоты, RR, failover,
breaker, recycle). `ServerPool` это тонкий слой выбора, *какой* `MuxPool`
обслуживает `open_stream`. Не трогаем wire-протокол, не трогаем логику слотов.
Это минимизирует регрессионный риск для боевых роутеров и телефона.

### 3.2 Один механизм в движке, разные источники списка

Пул живёт в `xr-proto` и одинаков для всех клиентов (роутер, Android, будущий
macOS-клиент XR-017 получит его бесплатно). Различаются только источник списка
и энергетический профиль:

- роутер: `client.toml`, тёплый профиль (§2.1, §2.4);
- Android: профиль приложения, наполняемый инвайтом и руками, холодный профиль
  (§2.6-2.8).

Раздачу списка *роутерам* через хаб по-прежнему не делаем: компрометация хаба не
должна перенаправлять флот роутеров на чужой VPS, а роутерный канал доверия к
хабу (реестр, подписанные команды) появится только в LLD-17/20. Для Android
инвайт уже подписан и TOFU-закреплён (LLD-04), поэтому там раздача безопасна.

### 3.3 Почему failover не делает сразу Direct

Сейчас единственный сервер down означает Direct. Это и есть деградация, ради
ухода от которой нужен пул. Direct остаётся **последним** рубежом (все серверы
down), а не первым: проксируемый ресурс заблокирован, и Direct его не откроет,
тогда как живой backup откроет. Порядок: primary -> backup(s) -> Direct.

### 3.4 Hold-down против флаппинга, без гистерезиса по латентности

Failback завязан только на «health up непрерывно N секунд», без сложного
скоринга RTT/потерь. Простая, предсказуемая политика; тонкую классификацию
«сервер деградировал, но жив» отдаём LLD-11 (мониторинг). Здесь бинарно up/down
(как breaker) плюс анти-флаппинг таймером.

### 3.5 Failover между профилями не делаем

Профиль это аккаунт (свой хаб, ключ, шары, правила), автопрыжок между аккаунтами
незаметно менял бы identity-контекст пользователя. Если два VPS обслуживают одну
экосистему, правильная модель это один профиль с двумя эндпоинтами, и ровно её
этот LLD и вводит.

## 4. Изменения в коде

| Файл | Что меняется |
|---|---|
| `xr-proto/src/config.rs` | `ServerAddress` (single) обобщается в `Vec<ServerEntry>`; новый `ServerEntry { name, address, port, priority, key/salt/modifier: Option }`. Парсер принимает и legacy `[server]`, и `[[servers]]` (serde untagged / ручной merge), сортирует по `priority`. Дефолты, валидация (>=1 сервер; при дублях приоритета стабильная сортировка по порядку). |
| `xr-proto/src/server_pool.rs` (новый) | `ServerPool { slots: Vec<ServerSlot>, active: AtomicUsize, profile: PoolProfile }`, `ServerSlot { entry, pool: Arc<MuxPool>, health: HealthState }`. Методы: `open_stream(target)` (active + failover к следующему здоровому), `health_loop()` (фоновая проба + failback с hold-down, поведение по `PoolProfile`), `warmup()` (только тёплый профиль), `recycle()` (все слоты, для C4), `active_server()`/`server_health()` (хуки для LLD-11 и индикации §2.6). `HealthState { Up, Down { since, class } }`; `class` пока `Unknown`, наполняется в LLD-11. |
| `xr-proto/src/mux_pool.rs` | Публичные геттеры под `ServerPool`: подтвердить, что `is_server_down()` и `recycle()` доступны; добавить `probe()` (probe-lease для активной health-пробы без пользовательского стрима), если ещё нет отдельной точки входа. |

| `xr-proto/src/lib.rs` | Реэкспорт `ServerPool`, `ServerEntry`, `HealthState`, `PoolProfile`. |
| `xr-hub` (конфиг + `api/invites.rs`) | Дефолты хаба обобщаются до списка серверов; `payload()` заполняет и `servers`, и легаси-поля (primary). |

## 5. Риски и edge-кейсы

1. **Все серверы down одновременно** (общий провайдерский сбой / DPI на оба IP).
   Пул отдаёт `Err`, дальше `on_server_down="direct"`. Поведение не хуже
   сегодняшнего; LLD-11 эскалирует это как «инцидент: все серверы недоступны».
2. **Флаппинг primary** (мигает up/down). Hold-down (§2.5) гасит; без него
   трафик прыгал бы между IP, ломая длинные стримы и засоряя fingerprint.
3. **Разные ключи на серверах.** Если у backup другой `[obfuscation].key`, а
   override не указан, handshake к backup провалится и он будет вечно «down».
   Валидация конфига при старте + WARN в логе (детектится первой failed-пробой).
   В MVP оба VPS делят ключ, не стреляет, но документируем. На Android ключ
   общий на профиль by design (§2.6).
4. **Рост ESTAB на серверах.** Тёплый профиль роутера держит mux к **каждому**
   серверу, на backup висят N TCP даже без трафика. Это сознательно (мгновенный
   failover без cold-start), учесть `max_connections=256` на VPS (LLD-09 §6).
   Телефоны нагрузку не добавляют: холодный профиль (§2.7) держит соединения
   только к активному.
5. **Сетевой re-bind (C4) одновременно с failover.** `on_network_changed` зовёт
   `recycle()`, теперь по всем слотам пула. Следить, чтобы `health_loop` и
   `recycle` не гонялись за один слот-lock (переиспользовать существующую
   синхронизацию MuxPool).
6. **Конфиг с пустым `[[servers]]` / профиль без эндпоинтов.** Валидация: >=1
   сервер обязателен, иначе явная ошибка старта (как `source_ips` для UDP relay)
   и блок сохранения в Edit на Android.
7. **Миграция профилей на Android.** Легаси-поля переезжают в `endpoints[0]`
   лениво при чтении; откат на старую версию приложения не теряет primary
   (легаси-поля продолжаем писать зеркально с `endpoints[0]`).
8. **Version skew приложения и хаба.** Старое приложение + новый payload:
   лишнее поле игнорируется, работает по легаси-адресу. Новое приложение +
   старый хаб: `servers` пуст, пул из одного адреса. Ломающих комбинаций нет.

## 6. План проверки

Роутерная часть: стенд это оба боевых роутера (`routers.md §2.2`), scoped
nft-drop как в C1 для имитации мёртвого VPS.

1. **Failover primary -> backup.** Конфиг с Aeza (primary) + Timeweb (backup).
   Scoped-drop 8443 к Aeza. Проксируемый curl из LAN (`claude.ai`) продолжает
   работать через Timeweb за секунды; в логе «server aeza down -> active=timeweb».
   Direct-трафик не затронут.
2. **Failback.** Снять drop с Aeza. В течение `failback_hold_secs` активный
   остаётся Timeweb, затем переключается обратно на Aeza; новые стримы идут на
   Aeza (`logread` / счётчики ESTAB на VPS, `routers.md §7.4`).
3. **Все серверы down.** Drop 8443 к обоим. Проксируемый curl падает в Direct
   за ~0.5с (как C1); непроксируемый Direct не затронут.
4. **Анти-флаппинг.** Drop/undrop Aeza с периодом < hold-down: активный не
   прыгает, остаётся на backup до стабилизации primary.
5. **Legacy-конфиг.** Роутер со старым `[server]` стартует без правки, ведёт
   себя как пул из одного сервера (Direct при его падении, как сегодня).

Android-часть (device-verify):

6. **Failover на телефоне.** Профиль с двумя адресами (Aeza primary + Timeweb).
   Уронить primary (остановить `xr-server` на VPS). Проксируемый сайт продолжает
   открываться, статус показывает «через timeweb (резерв)», в логе failover.
   Туннель и VPN-сервис не перезапускались.
7. **Failback на телефоне.** Поднять primary, после hold-down статусная строка
   резерва исчезает, новые стримы идут на primary.
8. **Энергопрофиль.** В здоровом состоянии на VPS-backup нет ESTAB с телефона;
   в простое нет периодических проб (по логу движка).
9. **Инвайт со списком.** Инвайт от хаба с двумя серверами: Apply создаёт
   профиль с двумя эндпоинтами. Легаси-инвайт создаёт профиль с одним.
10. **Миграция.** Обновление приложения поверх существующего профиля: адрес
    сохранился как `endpoints[0]`, подключение работает без правок.
11. `cargo test --workspace` зелёный, 0 warnings (CLAUDE.md).

## 7. Вне скоупа

- **Живое обновление списка серверов с хаба** (hot-swap через рефреш пресета,
  раздача роутерам). Естественно ляжет поверх LLD-13 (provisioning добавляет
  сервер), LLD-16 (рефреш с хаба) и LLD-17/20 (канал доверия роутер-хаб). См.
  §2.8, §3.2.
- **Балансировка нагрузки между серверами** (active-active). Сознательно не
  делаем: пул про резервирование, не масштабирование (§2.3).
- **Панель здоровья, уведомления, классификация классов сбоя** («DPI» vs
  «сервер не отвечает» vs «auth»): `HealthState.class` здесь только заведён как
  поле, наполняет его LLD-11. Индикация в этом LLD ограничена строкой статуса.
- **Failover между профилями на Android** (§3.5).
- **Per-endpoint ключи в Android UI** (роутерный TOML-override покрывает кейс).
- **Гео-/латентностный выбор «ближайшего» сервера**: нет, строгий приоритет.

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Равноправны или primary/backup?* **Primary/backup**, строгий приоритет (§2.3).
- *Липкость после восстановления?* **Да**, sticky-to-primary + failback с
  hold-down (§2.5).
- *Роутер или любой клиент?* **Любой клиент** общего движка; роутер и Android в
  этом LLD, macOS (XR-017) получит бесплатно (решение владельца, §3.2).
- *Что считать пулом на Android?* **Пул внутри профиля**, между профилями
  автоматики нет (решение владельца, §2.6, §3.5).
- *Откуда список на Android?* **Ручной ввод + подписанный инвайт/пресет хаба**;
  живое обновление с хаба это follow-up (решение владельца, §2.8).
- *Пробы на мобильном?* **Экономная политика**: пассивный детект всегда, пробер
  только в деградации, backup холодный (сравнение и выбор в §2.7).
- *Где список для роутера: config или hub?* **Config роутера** в MVP; hub это
  follow-up поверх LLD-13/16/17 (§3.2).
- *Health-check пассивный или активный?* **Оба**: пассивный breaker как триггер
  failover'а, активный пробер для failback и pre-warm (§2.4), на мобильном
  урезанный (§2.7).
