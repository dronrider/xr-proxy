# LLD-14 — Гибридный редактор правил в xr-hub

**Статус:** Draft
**Область:** `xr-hub` (TOML как источник правды для пресета, фрагмент-редактор в
Admin SPA, line-surgical правки с сохранением комментариев, компиляция в
раздаваемый JSON), `xr-proto` (общая модель «фрагмента правила» с LLD-05).
**Зависимости:** [LLD-01](01-control-plane.md) — xr-hub, Admin SPA (Vue+PrimeVue),
хранилище пресетов, подпись ed25519, раздача `GET /api/v1/presets/:name`.
[LLD-05](05-android-rules-editor.md) — Android rules editor: **обязаны разделить
единую модель правил** (формат фрагментов), иначе два несовместимых редактора.
**Связанные документы:** [ARCHITECTURE.md §6 (правила/пресеты)](../ARCHITECTURE.md),
`configs/routing-russia.toml` (эталон пресета), `routers.md §6` (структура
`russia.json`), problems.md задача 8.

Конфигуратор пресета в Admin UI неудобен — владелец правит правила **руками в
TOML/JSON** (`routers.md §6`: «vim russia.json, bump version, restart»). Идея: TOML
— **первичный источник правды**, мастер редактирует его *фрагменты* (категорию,
правило), **не отнимая** возможности ручной правки и **не затирая комментарии и
порядок**. Модель правил при этом должна быть **единой** с LLD-05 (Android), чтобы
не возникло двух несовместимых редакторов.

## 1. Текущее состояние

- Пресет хранится как **JSON** на VPS (`/var/lib/xr-hub/presets/russia.json`,
  `routers.md §6`), раздаётся `GET /api/v1/presets/:name` (LLD-01). Структура —
  `{ name, version, updated_at, rules: { default_action, rules: [ {action,
  domains[], ip_ranges[]} ] } }`.
- Эталон-источник правил живёт **отдельно** как `configs/routing-russia.toml` (с
  комментариями-категориями), но он **не связан** с боевым JSON автоматически —
  это просто референс, который владелец переносит руками.
- Admin SPA (LLD-01) умеет CRUD пресетов, но «по-крупному»; тонкая правка идёт
  через прямой `vim russia.json` + `bump version` + `systemctl restart xr-hub`
  (`routers.md §6`). Комментарии-категории в JSON не живут.
- LLD-05 (Android) вводит свою модель `UserRule { action, pattern, kind }` (одно
  правило = один паттерн). Пресет же — **категориями** (одно правило = действие +
  список доменов). **Две разные формы правила** — риск рассинхрона редакторов.

## 2. Целевое поведение

### 2.0 Конкретная боль (ответ пользователя, 2026-06-17)

Не абстрактное «неудобно», а две предметные претензии к текущей Admin SPA:

1. **Админка устроена «доисторически».** В разделе редактирования правил на одной
   бесконечной «портянке» (один скролл) свалены **и дефолтные параметры**
   (`default_action` и пр.), **и полотно всех правил** для предпросмотра. Нет
   разделения «настройки пресета» / «данные правил»; всё редактируется в одном
   простыне-тексте.
2. **Ожидаемая модель — современная админка-таблица.** Правила должны быть
   **таблицей** с **панелью управления сверху** (toolbar): добавить запись,
   выделить одну/несколько, отредактировать/удалить выделенное, искать. Дефолтные
   параметры пресета — **отдельно** от таблицы правил, а не в общей простыне.

Вывод для дизайна: ведущая поверхность редактора — **таблица правил + toolbar**
(§2.2), `default_action`/мета пресета вынесены в отдельную панель «Настройки
пресета». Сырой TOML остаётся как «продвинутый» режим и резерв (§3.4), но **не**
является основным экраном — именно его «простыня» и раздражает.

### 2.1 TOML как источник правды

Пресет в хабе хранится как **TOML с комментариями** (формат
`configs/routing-russia.toml`), а раздаваемый клиентам **JSON компилируется из
него**:

```
presets/russia.toml   ← источник правды (комментарии, порядок, ручная правка)
        │  compile (on save / on load)
        ▼
presets/russia.json   ← derived, то, что отдаёт GET /api/v1/presets/russia (+ подпись)
```

- Ручная правка `russia.toml` (vim по SSH) — по-прежнему легитимна и первична.
- Admin-мастер правит **тот же** TOML фрагментарно (см. §2.2), не трогая остальное.
- Компиляция TOML→JSON детерминирована; `version`/`updated_at` бампятся при
  сохранении; JSON подписывается ed25519 (LLD-01) как сейчас.

### 2.2 Таблица правил + toolbar (ведущая поверхность)

Прямой ответ на §2.0: экран редактора пресета делится на три зоны, а не одну
простыню.

```
┌──────────────────────────────────────────────────────────────────┐
│ Пресет: russia  v20 · обновлён 13.06 · 20 правил   [Опубликовать] │ ← шапка
├──────────────────────────────────────────────────────────────────┤
│ ▸ Настройки пресета  (свёрнуто): default_action = direct          │ ← отдельная панель
├──────────────────────────────────────────────────────────────────┤
│ [+ Добавить]  [✎ Изменить]  [🗑 Удалить]  [⇅]   [поиск ⌕______]    │ ← toolbar
├──────────────────────────────────────────────────────────────────┤
│ ☐ │ Категория      │ Действие │ Доменов │ IP │ Пример паттерна    │ ← таблица
│ ☑ │ Соцсети        │ proxy    │ 18      │ 0  │ *.youtube.com      │
│ ☐ │ Geo-blocked    │ proxy    │ 4       │ 1  │ claude.com         │
│ ☐ │ Торрент-трекеры│ proxy    │ 12      │ 0  │ rutracker.org      │
└──────────────────────────────────────────────────────────────────┘
```

- **Таблица** — одна строка = одна категория-`RuleFragment` (заголовок-комментарий,
  действие, счётчики доменов/ip, превью). Чекбоксы для **выделения** (одиночного и
  множественного). Клик по строке / «Изменить» → раскрывает детали категории
  (полный список доменов/ip-ranges, редактируемый).
- **Toolbar сверху** (панель управления): `Добавить` (новая категория/правило),
  `Изменить` (выделенное), `Удалить` (выделенные — массовая операция), reorder
  (`⇅` — поднять/опустить/в начало/в конец), `поиск` по домену/категории. Действия
  применяются к **выделению** в таблице — ровно та модель, что ожидается (§2.0.2).
- **Настройки пресета** (`default_action`, имя, версия) — **отдельная** свёрнутая
  панель над таблицей, **не** в общей простыне с правилами (§2.0.1).
- **Сохранение** правит **только затронутый фрагмент** TOML (line-surgical, §3.2),
  остальные строки/комментарии не трогаются.
- **Сырой TOML** — отдельная вкладка/режим «Расширенно» (§3.4), не основной экран.

### 2.3 Единая модель правил с LLD-05

Канонический тип — **`RuleFragment`** в `xr-proto`, общий для хаба и Android:

```rust
pub struct RuleFragment {
    pub action: Action,              // Proxy | Direct
    pub domains: Vec<String>,        // exact / wildcard
    pub ip_ranges: Vec<String>,      // CIDR v4/v6
    pub comment: Option<String>,     // заголовок-категория, сохраняется в TOML
}
```

- **Пресет** (хаб) = `Vec<RuleFragment>` + `default_action` — ровно текущая
  категорийная форма, теперь с `comment`.
- **`UserRule`** из LLD-05 (одно правило = один паттерн) — **вырожденный**
  `RuleFragment` (один элемент в `domains` *или* `ip_ranges`). LLD-05 переопределяет
  свой `UserRule` как тонкую обёртку/конвертер над `RuleFragment`, либо напрямую
  использует `RuleFragment` с одиночным паттерном. Так Android-редактор и хаб-
  редактор говорят на одном типе; классификация паттерна (`classify_pattern`,
  LLD-05 §3.1) переезжает в общий модуль и переиспользуется обоими.
- `Router::from_merged` (ARCHITECTURE §6.2) уже принимает overrides + preset; меняем
  только тип элемента на `RuleFragment` (для пресета это no-op по семантике).

### 2.4 Массовые операции

- Импорт/экспорт фрагмента (вставить блок доменов в категорию).
- Поиск + bulk-toggle действия у найденных.
- Дедуп доменов внутри/между категориями (предупреждение «домен уже в категории X»).

### 2.5 Версионирование и публикация

- Save → компиляция TOML→JSON → bump `version` → подпись → атомарная запись (как
  reset-password C3: temp+rename, сохранение прав). Клиенты подхватывают по ETag
  (LLD-01, hot-swap, ARCHITECTURE §6.2).
- Невалидный TOML/фрагмент при Save → ошибка, ничего не записывается (валидация
  обратным парсом, как reset-password).

## 3. Дизайн-решения

### 3.1 TOML, а не JSON, как источник — почему

JSON не держит комментарии, а именно категории-комментарии (`# Соцсети`,
`# Geo-blocked`) — то, чем владелец ориентируется в 20+ правилах. `routing-russia.toml`
уже существует как удобочитаемый эталон. Делаем его **боевым** источником, а JSON
— производным машинным форматом для клиентов. Это совпадает с задачей дословно
(«TOML — первичный источник правды»).

### 3.2 Line-surgical правки, не пересериализация

Как в `xr-hub reset-password` (C3): структура хаба не Serialize-обратимо без потери
комментариев/форматирования. Мастер правит TOML **хирургически** — заменяет/вставляет
строки конкретного фрагмента, не переписывая файл целиком. Это сохраняет ручные
комментарии и порядок, позволяя таблично-редактору и `vim` сосуществовать.
Валидация — обратным парсом отредактированного TOML в `Vec<RuleFragment>`.

### 3.3 Единый `RuleFragment` — почему обёртка, а не два типа

Если хаб и Android держат разные типы правил, любой формат-дрейф (нормализация
паттерна, порядок, wildcard-семантика) разъедется, и «единый редактор правил»
станет фикцией. Один `RuleFragment` в `xr-proto` + общий `classify_pattern` =
один источник истины о том, что такое правило. LLD-05 при имплементации **берёт
этот тип** (его `UserRule` — частный случай). Это и есть «свести в C5».

### 3.4 Таблица — основной экран, сырой TOML — режим «Расширенно»

Гибридность ≠ «таблица и простыня TOML рядом на одном экране» — именно простыня
и раздражает (§2.0.1). Ведущая поверхность — **таблица + toolbar** (§2.2) для
частых операций (добавить/выделить/изменить/удалить/искать). Сырой TOML —
**отдельный режим «Расширенно»** для редких правок, которые таблица не покрывает,
и для power-user/`vim`-совместимости. TOML остаётся *источником правды на диске*
(§3.1), но на экране по умолчанию пользователь видит таблицу, а не текст.

## 5. Риски и edge-кейсы

1. **Рассинхрон с LLD-05.** Если LLD-05 реализуется раньше со своим `UserRule` —
   при LLD-14 придётся мигрировать. Поэтому в §9 порядок: LLD-05 и LLD-14 идут
   связкой, общий `rule_fragment.rs` вводится тем из них, что реализуется первым,
   второй его переиспользует. Отметить зависимость в обоих документах.
2. **Потеря комментариев при line-surgical правке.** Главный риск гибридности.
   Тест `test_apply_fragment_edit_keeps_comments` + правило: мастер правит только
   диапазон строк своего фрагмента, всё вне — байт-в-байт. Сырой-TOML-путь — для
   правок, которые мастер не покрывает.
3. **Конфликт мастер vs параллельный vim по SSH.** Хаб читает TOML с диска при
   загрузке редактора; если файл изменился руками между загрузкой и Save —
   детектируем (mtime/хеш) и предупреждаем «файл изменён извне, перезагрузите».
4. **Невалидный TOML после ручной правки.** Валидация обратным парсом при загрузке
   и при Save; если на диске битый TOML — редактор показывает ошибку парса и сырой
   текст для починки, не молчит.
5. **Derived JSON разошёлся с TOML** (забыли перекомпилировать). Компиляция —
   часть Save и/или ленивая при отдаче пресета (сверка mtime TOML vs JSON);
   несоответствие → перекомпиляция. Источник правды один — TOML.
6. **Wildcard/нормализация расходятся между хабом и клиентом.** Снимается общим
   `classify_pattern`/`normalize_pattern` в `xr-proto` (§3.3) — одна реализация на
   обе стороны.
7. **Большие пресеты в SPA.** 20+ категорий с длинными списками — поиск + ленивый
   рендер фрагментов; не грузим тысячи доменов в DOM разом.

## 6. План проверки

xr-hub — Rust unit-тесты + ручная проверка Admin SPA на стенде (Timeweb).

0. **Раскладка экрана (§2.0).** Настройки пресета (`default_action`) — в отдельной
   панели, НЕ в общей простыне с правилами; правила — таблицей с toolbar сверху,
   а не сплошным текстом.
1. **Round-trip.** Загрузить `russia.toml` в редактор, ничего не менять, Save →
   derived JSON эквивалентен прежнему `russia.json` (диф по семантике пуст).
2. **Добавить домен в категорию** через таблицу → в TOML появился домен, комментарий
   и порядок остальных категорий не тронуты; JSON перекомпилирован; клиент по
   auto-refresh подхватил (роутер: `=> Proxy` для нового домена).
3. **Таблица + выделение.** Выделить чекбоксами 2-3 категории → toolbar `Удалить`
   убирает их разом; `поиск` фильтрует таблицу по домену/категории мгновенно.
4. **Ручная правка vim + таблица.** Поправить TOML по SSH (добавить комментарий) →
   открыть редактор → правка видна, таблица её не затирает при следующем Save.
5. **Невалидный фрагмент.** Ввести битый паттерн → Save заблокирован, файл на диске
   не изменён.
6. **Единая модель.** Тот же домен-паттерн классифицируется одинаково в хабе и в
   Android-редакторе (LLD-05) — общий `classify_pattern`.
7. `cargo test --workspace` зелёный, 0 warnings.

## 7. Вне скоупа

- **Авто-пополнение правил из community-фидов** — это задача 13 (отдельный LLD,
  P3), здесь только ручной/мастер-редактор.
- **Версионная история пресета / diff между версиями** — bump version есть; полный
  changelog/rollback UI — follow-up.
- **Раздача пользовательских overrides через хаб** — overrides остаются клиентскими
  (LLD-05); хаб правит только централизованный пресет.
- **Многопользовательское одновременное редактирование** — single-owner модель;
  конфликт с внешним vim только детектируем (§5.3), не мержим.
- **GeoIP-правила в мастере** — feature-flag в движке; редактор работает с
  domains/ip_ranges, GeoIP — отдельно.
