# LLD-22: Балансировка устройств по VPS на роутере

**Статус:** Draft (XR-080)
**Область:** `xr-proto` (ключевой выбор сервера в `ServerPool`, `config.rs`:
секция `[balancing]` и `weight` у `[[servers]]`), `xr-client` (проброс LAN
source IP в пул, сборка правил из конфига), `configs/client.toml`
(документация).
**Зависимости:** [LLD-10](10-client-multi-vps-failover.md) (Implemented):
`ServerPool` поверх per-server `MuxPool`, failover/failback с hold-down; этот
LLD добавляет слой выбора «какой сервер домашний для устройства», не трогая
механику отказа. [LLD-09](09-multi-mux-pool.md) (Implemented): `MuxPool` и
`probe_fresh`.
**Связанные документы:** XR-016 (per-app туннель: тот же ключевой выбор с
ключом «приложение»), XR-074 (per-invite учёт на сервере), LLD-11 (панель
здоровья), `local-docs/routers.md` (парк: два VPS, два роутера).

Наблюдение владельца (2026-07-05): оба VPS живые, но sticky-to-primary из
LLD-10 гонит весь LAN через один сервер, второй простаивает до сбоя. Этот LLD
добавляет распределение **устройств** по серверам: явные правила «IP -> сервер»
плюс взвешенный хеш для остальных, при сохранении failover-гарантий LLD-10
(упавший дом не Direct-утечка, а переезд на следующий сервер; Direct только
когда исчерпан весь пул, и то по политике `on_server_down`).

Решения владельца, зафиксированные до этого LLD: (а) распределение гибкое,
правила на конкретные IP внутри сети; (б) скоуп только роутер, мобильному это
станет актуально после per-app туннеля XR-016; (в) ключ балансировки это
устройство, аутентификация не нужна (совет ассистента принят, обоснование в
§2.1).

---

## 1. Текущее состояние

- `ServerPool` ([server_pool.rs](../../xr-proto/src/server_pool.rs)) выбирает
  сервер глобально: один `active` индекс на весь роутер, sticky-to-primary,
  failover сдвигает `active`, failback с hold-down возвращает. LLD-10 §2.3
  сознательно выбрал «primary/backup, не балансировка»; пересмотр в §3.1.
- Source IP устройства в TCP-пути уже есть: `handle_connection` получает
  `client_addr` из `accept()`
  ([proxy.rs:109](../../xr-client/src/proxy.rs#L109)), но до
  `server_pool.open_stream(&target)`
  ([proxy.rs:279](../../xr-client/src/proxy.rs#L279)) он не доносится, пул
  ключа не принимает.
- Здоровье серверов пул освежает только «выше активного»: `health_tick` пробит
  кандидатов на failback, серверы ниже активного не пробятся (при
  балансировке любой сервер может быть чьим-то домом, его здоровье нужно
  держать свежим).
- UDP relay ходит на один `vps_addr` на весь relay
  ([udp_relay.rs](../../xr-client/src/udp_relay.rs)), мимо `ServerPool`
  вообще. Остаётся вне скоупа (§7).
- Правил распределения в конфиге нет; `[[servers]]` несёт `name/address/port/
  priority` ([config.rs](../../xr-proto/src/config.rs)).

## 2. Целевое поведение

### 2.1 Ключ балансировки: устройство, то есть LAN source IP

Решение владельца (пункт в). Обоснование, почему это не ограничение, а
правильный дизайн:

- Балансирует роутер, а он видит настоящий адрес устройства в PREROUTING до
  NAT. Идентичность нужна той стороне, которая не различает клиентов (сервер
  видит всех из одного туннеля); роутер различает и так, поэтому
  аутентификация для балансировки не нужна ни сейчас, ни потом. Она появится
  в этой теме только когда решения переедут на сервер (квоты per-invite,
  XR-073/074/075).
- Per-connection round-robin размазал бы соединения одного устройства по
  exit-IP разных VPS: ломаются сессии, антифрод, капчи, банки. Per-destination
  размазывает трафик одного устройства по серверам и превращает диагностику в
  угадайку. Per-device даёт стабильный exit-IP на устройство и предсказуемое
  «приставка всегда через aeza».
- На Android после XR-016 ключом станет приложение (UID); API ключа делаем
  байтовым, чтобы обобщение не потребовало ломки (§2.7).

### 2.2 Конфиг: правила плюс веса

```toml
[balancing]                 # секции нет = поведение как сегодня (все через primary)
enabled = true              # быстрый рубильник без удаления правил

[[balancing.rules]]         # явное закрепление: устройство -> сервер
source = "192.168.1.50"     # IP или CIDR ("192.168.1.0/28")
server = "timeweb"          # name из [[servers]]

[[servers]]
name = "timeweb"
address = "201.51.16.159"
port = 8443
priority = 1
weight = 1                  # вес в хеш-распределении устройств без правила;
                            # 0 = устройств по хешу не получает (только
                            # правила и failover)
```

- Устройство без правила получает дом взвешенным rendezvous-хешем по живым
  серверам с `weight > 0` (§2.4).
- Режим «только правила, остальные на primary» выражается весами без
  отдельного mode: primary `weight = 1`, остальные `weight = 0`. Все веса
  нулевые = устройства без правил идут по глобальному порядку, как сегодня.
- `priority` не меняет смысла: это порядок failover, а не доля нагрузки.

### 2.3 Домашний сервер и порядок обхода

`home_for(ip)`: первое совпавшее правило (порядок файла) -> иначе
rendezvous-хеш -> иначе None (балансировка выключена или все веса нулевые).

Порядок обхода в `open_stream` для ключевого вызова:

```
дом (если стабильно жив) -> глобальный active -> остальные по приоритету
```

- «Стабильно жив» переиспользует hold-down LLD-10 §2.5: `HealthState::Up`
  непрерывно не меньше `failback_hold`. Пока дом нестабилен, устройство едет
  по глобальному порядку; после стабилизации новые соединения возвращаются
  домой сами (это и есть per-device failback, без отдельного таймера).
  На старте пула слоты считаются стабильными оптимистично (как сегодняшний
  дефолт `HealthState::Up`), первый сбой снимает и запускает обычный цикл.
- Ключевой путь не двигает глобальный `active`: тот остаётся механикой
  failover для устройств без дома и для не-ключевых вызовов. Иначе падение
  чьего-то дома дёргало бы указатель, по которому едут все остальные.
- Отказ всего обхода отдаёт `Err`, дальше как сегодня `on_server_down`
  (дефолт block). Цепочка fail-closed (XR-077 kill-switch, XR-009 Fix A) не
  меняется ни в одной точке.

### 2.4 Взвешенный rendezvous-хеш, не ring и не mod N

Для устройств без правила дом считается weighted rendezvous (HRW): для каждого
сервера score из детерминированного хеша пары (source IP, идентичность
сервера) и веса, дом это сервер с максимальным score. Свойства, ради которых
он выбран:

- Нет состояния: ни ring-таблицы, ни карты «устройство -> сервер». Дом это
  чистая функция от (IP, состав серверов, веса), одинаковая после рестарта.
  После XR-079 отдельно ценно, что память не растёт с числом устройств.
- Минимальное перетасовывание: добавление/удаление сервера перемещает только
  устройства, чей максимум сменился (у mod N перетасовывается почти всё).
- Веса встроены в формулу (score_i = -w_i / ln(u_i), u_i это хеш, приведённый
  к (0,1)), доля устройств пропорциональна весу.

Идентичность сервера в хеше это `name` (или `address:port`, если имя пустое).
Переименование сервера перетасовывает хеш-устройства; документируем в
`client.toml` (§5.3).

### 2.5 Здоровье: пробы всех слотов

`health_tick` роутерного профиля расширяется: `probe_fresh` идёт по **всем**
слотам, а не только выше активного. Иначе здоровье дома, стоящего по
приоритету ниже активного, никто не освежает, и устройства с домом-«ниже»
навсегда зависали бы на глобальном порядке после первого сбоя дома. Цена: на
2-3 серверах это одна лишняя проба в 15 секунд, для роутера копейки.
Мобильный профиль не трогаем (балансировка роутер-only, ключевой API на
мобильном не зовётся).

### 2.6 Диагностика

- При старте сводка в лог: `balancing: 2 rules, hash weights timeweb=1
  aeza=1` (или `balancing: off`).
- Существующие события failover/failback пула остаются; отдельного лога «на
  каждое соединение» не заводим (шум), при отладке хватает
  `logread | grep server` плюс проверка exit-IP с устройства.
- Монитор XR-077 уже пишет ESTAB mux по каждому VPS (`mux[ ip=N ]`), по нему
  видно фактическое распределение.

### 2.7 Обобщение ключа под XR-016

Ключ в API это байты (`open_stream_for(key: Option<&[u8]>, target)`), роутер
передаёт октеты IPv4. Когда XR-016 введёт per-app туннель на Android, ключом
станут байты UID приложения без изменения пула. До того мобильный путь зовёт
не-ключевой вариант и ведёт себя ровно как сегодня.

## 3. Дизайн-решения

### 3.1 Пересмотр LLD-10 §2.3 («не балансировка») без слома его инвариантов

LLD-10 отвечал на вопрос отказоустойчивости и сознательно отложил
балансировку («пул про резервирование, не масштабирование», LLD-10 §7).
Наблюдение владельца добавило второй вопрос: ёмкость простаивающего резерва.
Ответ строится **над** LLD-10, а не вместо: механика отказа (breaker, bounded
open, hold-down, recycle) не меняется, добавляется только слой выбора
стартовой точки обхода. Инвариант LLD-10 «активен самый приоритетный из
здоровых» сохраняется для глобального указателя; для устройства с домом
инвариант расширяется до «дом, если стабильно жив, иначе глобальный порядок».

### 3.2 Правила первичны, хеш вторичен

Явное правило всегда сильнее хеша: владелец закрепляет критичные устройства
(приставка, рабочая машина) и получает детерминизм, а хеш лишь раскладывает
«всех остальных». Это ровно пункт (а) владельца: гибкое распределение по IP
внутри сети и правила для них.

### 3.3 Битое правило не роняет роутер

Правило с именем сервера не из `[[servers]]` или с некривым CIDR логируется
WARN и игнорируется, конфиг остаётся рабочим. Fail-fast здесь хуже: опечатка
в имени оставила бы весь LAN без прокси до ручного вмешательства (procd
рестартил бы вечно падающий клиент, kill-switch резал бы проксируемое). То же
поведение, что у недоступного пресета хаба: warn и работа на том, что есть.

### 3.4 Никакого per-device состояния

Ни карты «устройство -> текущий сервер», ни счётчиков. Вся балансировка это
функция от (IP, правила, веса, health слотов). Отсюда бесплатные свойства:
рестарт не теряет распределение, память не зависит от числа устройств, гонок
на общее состояние нет.

### 3.5 Стабильность exit-IP важнее равномерности

Выбранная модель допускает неравномерность (у одного устройства может быть
трафика больше, чем у трёх других вместе). Сознательно: выравнивание по
байтам потребовало бы миграции живых устройств между серверами, то есть
скачков exit-IP, ради проблемы, которой на нашем масштабе (единицы устройств,
два VPS) нет. Веса дают достаточную ручку грубой настройки.

Тесты (`xr-proto`):

- `test_balance_rule_matches_ip_and_cidr`: точный IP и CIDR дают дом по
  правилу, порядок правил соблюдается.
- `test_rendezvous_is_deterministic`: один ключ всегда даёт один дом (в том
  числе после пересоздания Balancer).
- `test_rendezvous_respects_weights`: на ~1000 синтетических ключей доля
  весов 2:1 держится с разумным допуском; `weight=0` не получает ключей.
- `test_rendezvous_minimal_reshuffle`: удаление одного сервера перемещает
  только его ключи, чужие дома не меняются.
- `test_keyed_open_uses_stable_home`: дом стабильно жив -> стрим идёт в него,

- `test_keyed_open_falls_back_when_home_down`: дом down -> стрим через

- `test_no_balancing_section_keeps_legacy_behavior`: без `[balancing]` (и при
  `enabled=false`) поведение бит-в-бит сегодняшнее: все через active.

## 5. Риски и edge-кейсы

1. **DHCP переезд IP.** Устройство с новым IP получает другой дом (и правило
   по старому IP перестаёт его касаться). Для устройств с правилами закреплять
   static lease на роутере; для hash-устройств смена дома редкая и безвредная
   (новый exit-IP с нового адреса). Документируем в `client.toml`.
2. **Смена состава серверов.** Rendezvous перемещает только устройства
   удалённого/добавленного максимума (§2.4), массового скачка exit-IP нет.
3. **Переименование сервера.** Идентичность в хеше это `name`, rename
   перетасовывает hash-устройства. Правила по `name` при rename тоже надо
   обновлять. Документируем; альтернатива (identity = address) страдала бы от
   смены IP сервера, что случается чаще осмысленного rename.
4. **Флап дома.** Устройства мигрируют туда-обратно не чаще hold-down
   (стабильность §2.3 переиспользует `failback_hold`), это тот же
   анти-флаппинг, что в LLD-10.
5. **Вторичный роутер/NAT в LAN.** Устройства за ним видны одним IP и делят
   один дом. Это предел ключа L3, принимаем (пункт в: ключ это устройство в
   терминах адресов LAN).
6. **`bypass_ips` и балансировка.** Bypass режется раньше в nftables
   (prerouting return), такие устройства до пула не доходят; правило
   балансировки на bypass-IP мёртвое, WARN при пересечении списков.
7. **Лишние пробы.** Пробы всех слотов (§2.5) добавляют по одному
   `probe_fresh` на сервер в 15с на роутере. На мобильном профиле ничего не
   меняется.
8. **Правила и веса при `[[servers]]` из одного сервера.** Всё вырождается в
   сегодняшнее поведение (дом совпадает с primary), отдельных веток не нужно.

## 6. План проверки

Проверка проксирования только по exit-IP (`https://www.cloudflare.com/cdn-cgi/trace`),
не по HTTP-200: Timeweb `201.51.16.159`, Aeza `85.192.38.29`, WAN роутера в
выводе значит Direct-утечку и провал теста.

1. **Правило.** На DE-роутере правило `MacBook -> timeweb` при глобальном
   primary aeza. С мака exit-IP = Timeweb, с другого устройства (без правила,
   веса aeza=1/timeweb=0) exit-IP = Aeza. Оба через прокси, не Direct.
2. **Хеш.** Веса 1/1, два устройства без правил: у каждого стабильный свой
   exit-IP, повторные соединения не прыгают между серверами; распределение
   видно в `monitor.log` (`mux[ ip=N ]`).
3. **Дом упал.** Scoped nft drop 8443 к дому мака (как в проверках C1/LLD-10).
   Новые соединения мака едут через второй VPS (exit-IP сменился на него),
   Direct-утечки нет, kill-switch не тронут.
4. **Возврат домой.** Снять drop: в пределах `failback_hold` новые соединения
   мака снова показывают exit-IP дома.
5. **Выключено.** Конфиг без `[balancing]`: все устройства через primary, как
   до этого LLD (регрессия sticky-to-primary).
6. `cargo test --workspace` зелёный, 0 warnings (CLAUDE.md).

## 7. Вне скоупа

- **UDP relay.** Ходит на один `vps_addr` мимо пула; потребитель сейчас один
  (приставка). Follow-up: `home_for` на flow с per-server туннель-сокетами,
  делать вместе с ближайшей доработкой relay.
- **Android и per-app.** После XR-016; ключевой API уже готов к этому (§2.7).
- **Раздача правил балансировки с хаба.** Правила локальны для конкретного
  LAN (адреса устройств), в пресете хаба им делать нечего; удалённое
  управление конфигом роутера это LLD-20.
- **Авто-веса по замерам** (ёмкость, латентность, цена трафика): только
  статические веса из конфига. Замеры это территория LLD-11/LLD-18.
- **Выравнивание нагрузки по байтам / least-loaded / миграция живых потоков**
  (§3.5).

## 8. Открытые вопросы (закрыты в этом дизайне)

- *Ключ балансировки?* **Source IP устройства** (решение владельца, пункт в;
  обоснование §2.1).
- *Скоуп?* **Только роутер** (пункт б); Android после XR-016 тем же API.
- *Устройства без правила?* **Weighted rendezvous** по серверам с
  `weight > 0`; все нули = глобальный порядок (§2.2, §2.4).
- *Migration живых потоков при возврате дома?* **Нет**, только новые
  соединения (как failback LLD-10: живые стримы дорабатывают).
- *Битое правило?* **WARN + игнор**, роутер продолжает работать (§3.3).
- *Отдельный failback-таймер на устройство?* **Не нужен**: «стабильность»
  слота с общим hold-down даёт per-device failback бесплатно (§2.3).
