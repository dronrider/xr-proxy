# xr-proxy: задачи (префикс XR)

Канбан-доска открытой работы и единственный источник правды по тому, что в работе.
Подробные разборы лежат в [docs/lld/](lld/) и в гитигнорнутом `local-docs/`, доска их
только индексирует, описания не дублирует. Готовую задачу из доски переносим в
[docs/TASKS-archive.md](TASKS-archive.md). ID сквозные по репозиторию и не
переиспользуются; ID ставим в subject коммита там, где уместно (`feat(core): XR-9 ...`).
Чувствительное (IP, доступы, инфра) в доску не пишем, оно остаётся в `local-docs/`.

Приоритет `P` не ставим на глаз, он выводится из ранга `R` по системе `RANKING.md`
(лежит рядом с `CLAUDE_RULES.md` над репозиторием). Формула
`R = Серьёзность(0-75) + Ценность(0-10) + Неопределённость(0-5) + Баг(+5) + Рычаг(0-5)`,
колонка `R` держит сумму и разбивку в этом же порядке слагаемых. Бакеты: P0 при
R >= 75, P1 при 50-74, P2 при 25-49, P3 при 0-24 (смысл: P0 чинить немедленно, P1
мешает ежедневному использованию, P2 заметно улучшает, P3 терпит). Backlog
отсортирован по R убыванию, при равенстве по возрастанию ID. Типы: `bug` / `task` / `LLD`.

| XR-086 | НАЙДЕН и устранён корень «проксирование зависает намертво, лечит только kill», отдельный от XR-083/084 (те реальны и закрыты, но этот эпизод не про них). Корень: head-of-line блокировка контрольных кадров в mux. Весь mux писал в ОДИН FIFO-канал writer'а (`writer_tx`, 2048): и балк-Data активных загрузок, и ConnectAck новых стримов. Под крупной загрузкой (или флудом open'ов) ConnectAck нового стрима вставал в хвост за мегабайтами Data и на медленном/деградирующем линке уходил в провод дольше `PER_SERVER_OPEN_TIMEOUT=5с` -> клиент ловил `open timed out` и с fail-closed (XR-081) блокировал. Туннель при этом ЖИВ (keepalive Ping/Pong идут, dead-link молчит, reader обеих сторон читает, Recv-Q=0), процесс спит ~1% CPU: по всем прошлым метрикам выглядел здоровым, потому долго не давался. Вся прошлая инструментация была чиста закономерно: `send_frame blocked`/`wedged` мерили постановку кадра В канал (permit), а не доставку в провод, а ConnectAck именно ДОСТАВЛЯЛСЯ поздно, а не блокировался на enqueue. Рестарт лечит, т.к. свежий mux пуст. **Фикс (коммиты 9bfe795 proto + 9d286cf core):** writer раздвоен на два плана, контрольный (`ctrl_tx`: Connect/ConnectAck/Ping/Pong) и балк (`writer_tx`: Data + Close), `writer_task` сливает их `biased`-select'ом с приоритетом контрольного, поэтому между любыми двумя Data успевают уйти все накопленные ConnectAck. HOL снят на ОБЕИХ сторонах (mux это общий крейт, серверный `mux_handler` шлёт ConnectAck тем же путём). Close сознательно оставлен в балк-плане рядом с Data того же стрима (иначе обгонит недописанные Data и обрежет выгрузку). Плюс `TCP_NODELAY` на mux-сокете клиента и Android (Connect не коалесцируется за upload-Data; на сервере nodelay уже стоял). Регресс-тест `test_writer_prioritizes_ctrl_over_bulk_data` (детерминизм: балк забит ДО ConnectAck, `biased` гонит ConnectAck первым; при развороте приоритета краснеет, проверено).
