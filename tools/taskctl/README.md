# taskctl: механика канбан-доски

Утилита для рутинных операций с `docs/TASKS.md` и `docs/TASKS-archive.md`:
завести строку, перевести между статусами, закрыть в архив, пересортировать
Backlog, проверить инварианты. Думающая часть (заголовок, слагаемые ранга по
`RANKING.md`, сценарий проверки) остаётся за исполнителем и передаётся
аргументами, утилита ничего не сочиняет.

Запуск из корня репозитория (корень это Rust-воркспейс, поэтому через `-C`):

```
go -C tools/taskctl run . <команда>
```

либо один раз собрать бинарь: `go -C tools/taskctl build` (артефакт в
гитигноре).

## Команды

```
id                                          следующий свободный ID
add --title "..." --type bug|task|LLD --rank "а+б+в+г+д"
    [--link "..."] [--status backlog|in-progress|check|blocked] [--id XR-NNN]
move <ID> <backlog|in-progress|check|blocked> [--reason "..."]
close <ID> [--commit sha1,sha2] [--date ГГГГ-ММ-ДД] [--link "..."]
sort
lint
```

Что утилита держит сама:

- сумма R и бакет P считаются из разбивки `--rank`, руками их не передать;
  разбивка проверяется по шкалам `RANKING.md` (серьёзность из {0,25,50,75},
  ценность 0-10, неопределённость 0-5, баг 0 или 5, рычаг 0-5);
- Backlog отсортирован по R убыванию, при равенстве по возрастанию ID; `add`
  сразу ставит строку в правильное место;
- `move` в blocked требует `--reason` и дописывает причину к заголовку;
- `close` дописывает строку в архив (без колонки R, с датой закрытия),
  переносит файл задачи в `tasks/archive/<год>/` (`git mv` с фолбэком на
  обычный rename) и ставит ссылку уже на новое место; хеши из `--commit`
  добавляются в ячейку ссылки;
- `lint` проверяет разбор всех строк, бакеты P, сортировку, дубли ID по доске
  и архиву, формат дат и что локальные markdown-ссылки ведут на существующие
  файлы.

Правило надёжности: нетронутые строки сохраняются байт в байт, файл вне таблиц
не переформатируется. Строка, которую не удалось разобрать, валит операцию с
ошибкой вида `файл:строка` до какой-либо записи на диск.
