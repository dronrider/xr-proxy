# Нагрузочный стенд (XR-087)

Воспроизведение и регресс-проверка зависаний прокси-тракта (XR-086). Разводит
роль прокси-сервера и цели нагрузки, чтобы одно не искажало другое:

```
ноут (LAN-клиент) -> роутер (TPROXY) -> mux к timeweb (xr-server) -> timeweb -> тест-сервер на aeza
```

## Компоненты

- `testserver.py` это TLS-сервер на **aeza**, нестандартный порт (по умолчанию 9443).
  Эндпоинты: `/download?size=N`, `/slow?size=N&rate=R`, `/small`, `/burst?size=N`.
- `loadgen.py` это нагрузчик с **ноутбука** (за роутером). Ключевой сценарий под
  XR-086: медленно читаемые крупные загрузки (`slow`-воркеры забивают серверный
  mux writer_tx) плюс шквал мелких открытий (`churn`-воркеры, нужен ConnectAck
  через переполненный канал).

## Развёртывание тест-сервера (aeza)

```
scp testserver.py root@85.192.38.29:/root/
ssh root@85.192.38.29 'cd /root && openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 30 -nodes -subj "/CN=loadtest.local"'
ssh root@85.192.38.29 'systemd-run --unit=xr-loadtest --collect --working-directory=/root python3 -u /root/testserver.py 9443'
# журнал: journalctl -u xr-loadtest -f ; стоп: systemctl stop xr-loadtest
```

## ВАЖНО: роутинг (грабли, обязательно к настройке)

Роутер **исключает IP серверов из редиректа** (чтобы mux к `server:8443` не
зациклился на `:1080`), поэтому трафик к `aeza:9443` по умолчанию идёт **Direct
мимо прокси**. Чтобы тест шёл через прокси, на роутере, за которым сидит ноут,
вставить точечный редирект ПЕРЕД возвратом по IP сервера:

```
# на роутере (например DE 37.208.69.187). handle 5 = "ip daddr <aeza> return"
nft insert rule ip xr_proxy prerouting position 5 iifname "br-lan" ip daddr 85.192.38.29 tcp dport 9443 redirect to :1080
# убрать после теста:
nft delete rule ip xr_proxy prerouting handle <handle правила 9443>
```

Проверка, что реально проксируется (источник на тест-сервере = timeweb, не роутер):

```
ssh root@85.192.38.29 'ss -tn | grep :9443 | grep ESTAB'
# 201.51.16.159:xxxx = через timeweb (ОК); 37.208.69.187 = Direct мимо прокси (НЕ ОК)
```

Дополнительные грабли:
- **Ноут кочует между роутерами** (gateway 192.168.1.1 = DE, 192.168.2.1 = RU).
  Правило редиректа ставить на тот роутер, за которым ноут сейчас.
- **Пер-девайс байпас** в цепочке (`ip saddr <device> tcp dport != {...} return`):
  убедиться, что IP ноута под него не попадает, иначе его 9443 не проксируется.

## Запуск нагрузки (ноут)

```
ulimit -n 12288
python3 loadgen.py 85.192.38.29 9443 <duration> <slow_workers> <churn_workers>
# напр.: python3 loadgen.py 85.192.38.29 9443 200 50 500
```

Наблюдение затыка (XR-086):
- роутер: `logread | grep -E "wedged|blocked >2s|open timed out"`, `hangwatch.log`;
- сервер: `journalctl -u xr-proxy-server | grep "blocked >2s"`.

## Состояние (2026-07-06)

Стенд собран, путь через прокси подтверждён (тест-сервер видит источник timeweb).
Первый прогон (slow=50 churn=500) затык НЕ вызвал: высокая параллельность с одного
ноута упирается в установку соединений (масса TLS-хендшейков таймаутит, до mux не
доходит, тест-сервер получил единицы соединений). Нужно тюнить генерацию:
несколько источников, либо нагрузка ближе к роутеру, либо иной микс (меньше
одновременных, дольше живущие slow-потоки для гарантированного забивания writer_tx).
